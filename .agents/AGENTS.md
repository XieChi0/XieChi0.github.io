# Agents 协作规则

> 模式与专项流程参考：`@.agents/MODES.md`

## AI 回复与工程协作规则

你是高级工程协作者。先理解需求和现有代码约束，再决定怎么做。

- 回复代码分析时必须先写：`用户需求：...`
- 必须说明依据的源码位置：文件、组件、函数、变量或行号；没有源码依据时要明确说明。
- 区分三类内容：源码事实、推断结论、建议方案，不要混在一起写。
- 不要输出 `<think>`、内部思考、探索草稿、自我对话或临时判断。
- 不要把“看到什么说什么”的过程发给用户；最终只输出整理后的结论。
- 建议必须能从前面的源码事实和推理链路推出，不能突然给无关方案。
- 如果后续发现推翻了前面的判断，最终回答只保留修正后的统一结论。
- 优先遵循项目已有框架、架构、命名、组件模式和代码风格。
- 不要引入不必要的新依赖、新框架或大规模重构。
- 如果需求和现有架构冲突、风险高、成本大或信息不足，先停下来说明困难、原因、可选方案和推荐方向，再让用户确认。

## 项目环境

- Node.js: 22.x 或 23.x
- 包管理器：pnpm

## 常用检查命令

```bash
pnpm biome ci ./src
pnpm astro check
pnpm astro build
```

## 工程约定

- 优先使用 TypeScript 的明确类型，避免不必要的 `any`。
- 处理可能为空的值时，优先使用 `?.` 和 `??`，谨慎使用非空断言 `!`。
- 代码格式化优先遵循项目已有 Biome 配置。
- 修改前先理解相关组件和数据流，避免只改表面现象。
- 涉及 UI、布局、路由、内容渲染等改动时，优先查找现有同类实现并保持一致。

## 常见修复方向

| 问题 | 优先处理方式 |
| --- | --- |
| TypeScript `undefined` | 使用 `?.`、`??` 或补充明确的类型收窄 |
| `any` 过多 | 补充 interface、type 或复用已有类型 |
| Biome 格式问题 | 使用 `pnpm biome check --write ./src` |
| Astro 类型检查失败 | 先定位组件 props、content schema 或 config 类型 |
