# 触发模式

> 当用户说触发模式，读这个文件。

## [讲解]

1. **先用大白话解释** - 先说明干什么的、有什么用，再说专业术语
2. **说明解决什么问题** - 为什么需要，它解决了什么麻烦
3. **讲整体用法** - 先给流程/思路，再进具体代码
4. **代码分步骤讲** - 拆成小段，每段说明为什么这么写、起什么作用
5. **顺带解释语法** - TypeScript、Vue、JavaScript 等不要默认用户已懂
6. **用简单例子说明设计思路** - 说明好处和适用场景
7. **控制复杂度** - 小例子 + 分步骤，避免长代码堆砌
8. **最后给记忆口诀** - 一句话总结核心

## [git commit]

当用户输入 `[git commit]-"提交信息"` 时，执行以下流程：

### 流程

1. **执行提交** - 用户已提前完成 `git add`，直接 `git commit -m "提交信息"`
2. **监测结果**
   - **成功** → 显示 commit hash，结束
   - **失败** → 进入修复流程
3. **失败处理** - 分析错误并修复代码，修复后重新 `git commit -m "提交信息"`（最多尝试 3 次）
4. **放弃条件** - 3 次仍未成功，汇总错误信息，提示用户手动处理

### 修复后说明

每次修复后说明改了什么。

## [git提交总结]

**重要**：这份总结必须和代码变更一起提交，而不是单独开一个 commit。

### 格式

```markdown
## [时间] 提交摘要
**Commit ID**: (留空，提交后补充)
**变更文件数**: N
**主要需求**: 简要描述

### 变更详情
| 文件 | 变更类型 | 说明 |
|-----|---------|------|
| src/xxx.ts | 修改 | xxx |

### 实现方式
- xxx
---
```

### 执行方式

1. 生成上述格式的摘要内容
2. **将摘要插入到 `commit-history.md` 最上方（紧跟标题 `# Commit History` 之后）**（无需写 Commit ID）
3. 如果有涉及知识库的，可以将当前的改动和知识库.md结合起来修改这个文档
4. 把 `commit-history.md` 一起 commit（和其他代码变更同时提交）
5. Commit ID 等提交成功后再补充到 commit-history.md 中

> **核心原则**：commit-history.md 的新记录必须写在最上面，并且和代码变更在同一个 commit 里，避免"上一次的需求到下一次才总结"的文不对题问题。

## [everydaylearn]

在 `src/content/posts/everyDayLearn/` 下新建当天日期的 MD 文件（如 `0522.md`）：
- 每个用户指定的主题作为一个一级标题
- tags 根据主题提取 1-2 个核心关键词
- 参考同目录下已有文件的格式
- 用户没指定内容 → 根据本次讨论填写

## [yaml]

为当前打开的 MD 文件添加 YAML frontmatter：

```yaml
---
title: (根据文章标题)
published: (最开始发布日期，如 2024-05-01)
updated: (当天日期)
description: '(简短描述)'
image: ''
tags: [(核心关键词)]
category: 'everyDayLearn'
draft: false
---
```

## [输出总结]

用户输入 `[输出总结]` 或要求"把内容总结成文章"时：

1. **默认新建文章** - 目录 `src/content/posts/ai`，文件名用英文短名+连字符
2. **确定总结范围** - 默认总结本次讨论；后续同一主题则更新已有文章
3. **文章格式** - `.md`，顶部 YAML，默认分类 `ai`，tags 提取具体关键词
4. **写作要求**
   - 先大白话讲概念，再说明解决什么问题
   - 给出整体流程，代码用小例子分步骤解释
   - 适合面试准备阅读，结尾给记忆口诀
   - 每次整理重新思考目录结构，不要机械追加
5. **完成后** - 告诉用户新建/更新了哪篇，简要说明内容

## [文章迁移]

当用户要求“把某篇文章移动到另一个分类 / 子分类目录”时，执行以下流程：

### 适用场景

- 用户只告诉目标目录，比如“把这篇文章从 `a` 挪到 `a/B`”。
- 文章内存在本地图片引用，需要一起调整。
- 目标目录下还没有 `assets` 目录，需要自动创建。

### 执行流程

1. **定位文章** - 先确认要移动的 Markdown 文件路径。
2. **确认目标目录** - 根据用户给的新分类目录，确定新的 Markdown 存放位置。
3. **扫描图片引用** - 检查文章里的 Markdown 图片语法和 HTML `img` 标签，找出当前文章依赖的本地图片。
4. **创建目标资源目录** - 如果目标目录下没有 `assets` 文件夹，自动新建。
5. **移动文章图片** - 把当前文章使用到的本地图片**移动**到目标目录的 `assets` 中，注意是移动不是复制。
6. **重写图片路径** - 把文章里的图片引用统一改成相对于新文章位置的路径，确保迁移后仍能正常显示。
7. **移动文章文件** - 把 Markdown 文件移动到目标分类目录。
8. **结果校验** - 检查迁移后文章引用是否都指向新位置，避免遗留失效路径。

### 重要规则

- **默认执行移动，不执行复制**。
- **只处理当前文章实际引用到的图片**，不要把整个旧 `assets` 目录全搬走。
- **优先保持相对路径规范**，迁移后图片引用统一落到新目录下的 `assets`。
- **如果目标目录不存在，可以一并创建目录结构**。
- **如果发现同名图片冲突**，先停下来告知用户，再决定是覆盖、重命名还是保留旧图。
- **如果文章引用了外链图片或公共资源**，这些资源不移动，只保留原引用。

### 完成后输出

完成后告诉用户：

- 文章从哪里移动到了哪里
- 一共移动了多少张图片
- 是否新建了 `assets` 目录
- 是否存在未移动的外链图片或冲突文件
