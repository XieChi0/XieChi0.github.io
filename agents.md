# 开发规范 (AI)

## 环境要求

- Node.js: **22.x 或 23.x**（必须与 CI 一致）
- pnpm: 最新版本

## 推送前检查

按顺序执行，全部通过后再推送：

```bash
pnpm biome ci ./src
pnpm astro check
pnpm astro build
```

## 常见问题

| 问题 | 解决方案 |
|-----|---------|
| TypeScript `undefined` 错误 | 用 `?.` 或 `??` 处理，或加空检查 |
| `any` 类型警告 | 定义具体 interface |
| Biome 格式错误 | `pnpm biome check --write ./src` |
| 导入排序错误 | 同上，自动修复 |

## 代码规范

- 优先使用 `?.` 和 `??`，避免非空断言 `!`
- 用模板字符串 `` `${var}` ``，不用拼接
- 导入按字母排序
- 避免 `@ts-ignore`

## 用户情况

当前用户在**面试准备阶段**，很多技术点没有实际项目经验。

### 对 Agent 的期望

1. 讲概念时，**顺便带代码示例**（最简单的实现方式）
2. 提到用到的库或框架，**给出名字**
3. 不要过度深入，知道**怎么用、解决什么问题**就够了
4. 如果有现成的工具或库，优先告诉用户，**不要让用户自己造轮子**

## 触发模式

### [讲解]

1. **先用大白话解释**
   - 先说明这个东西是干什么的、可以用来做什么。
   - 然后可以说说专业词、专业术语。

2. **说明它解决什么问题**
   - 讲清楚为什么需要它。
   - 它解决了开发中的什么麻烦。

3. **讲整体用法**
   - 先给整体流程或使用思路。
   - 再进入具体代码，不要直接丢复杂代码。

4. **代码要分步骤讲**
   - 代码尽量拆成小段。
   - 每段都说明：为什么这么写、表达什么意思、起什么作用。

5. **顺带解释相关语法**
   - 如果涉及 TypeScript、Vue、JavaScript、Nest、Express、Prisma 等语法，要顺手解释。
   - 不要默认用户已经懂。

6. **涉及设计思路时，用简单例子说明**
   - 说明它的好处和适用场景。
   - 避免只讲抽象概念。

7. **控制复杂度**
   - 不要一次性给特别长、特别难懂的代码。
   - 优先使用"小例子 + 分步骤解释"。

8. **最后给记忆口诀**
   - 用一句容易记的话总结核心理解。
   - 方便之后快速回忆。

### [git提交总结]

输入 `[提交总结]` 时，用 `git diff` 收集变更信息，追加到 `commit-history.md`：

```markdown
## [时间] 提交摘要
**Commit ID**: (提交后补充)
**变更文件数**: N
**主要需求**: 简要描述

### 变更详情
| 文件 | 变更类型 | 说明 |
|-----|---------|------|
| src/xxx.ts | 修改 | xxx |

### 实现方式
- xxx
---
```

注意：commit ID 留空，提交后再补充。

### [everydaylearn]

输入 `[everydaylearn]` 时，在 `src/content/posts/everyDayLearn/` 下新建当天日期的 MD 文件（如 `0522.md`）：
- 每个用户指定的主题作为一个一级标题
- tags 根据主题提取 1-2 个核心关键词
- 参考同目录下已有文件的格式
- 如若用户没有指定具体内容，则内容根据本次讨论填写

### [yaml]

输入 `[yaml]` 时，为当前打开的 MD 文件添加 YAML frontmatter：

```yaml
---
title: (根据文章标题)
published: (最开始发布日期，如 2024-05-01)
updated: (当天日期)
description: '(简短描述)'
image: ''
tags: [(核心关键词)]
category: 'everyDayLearn'
draft: false
---
```

### [输出总结]

当用户输入 `[输出总结]`，或明确要求“把当前内容总结成文章”时，执行以下操作：

1. **默认新建一篇文章**：
   - 默认目录：`src/content/posts/ai`
   - 默认行为：新建 Markdown 文件
   - 如果用户说明文章已经存在，则改为更新用户指定的已有文章

2. **确定总结内容**：
   - 如果用户没有指定总结范围，默认总结本次对话中正在讨论的内容
   - 如果用户指定了主题、资料、代码或某一段内容，则只总结用户指定的内容
   - 如果后续对话仍然围绕同一个主题展开，优先更新之前新建的那篇文章，让文章逐步完善
   - 如果后续内容明显变成新主题，再新建另一篇文章

3. **文章格式**：
   - 文件使用 `.md`
   - 顶部添加 YAML frontmatter
   - 默认分类为 `ai`
   - `published` 和 `updated` 使用当天日期
   - `tags` 根据文章主题提取具体关键词，避免只写过大的标签

4. **写作要求**：
   - 每次整理前，先重新思考现有文章的目录结构
   - 可以根据新增内容调整标题、顺序和分组，不要只机械追加到末尾
   - 先用大白话讲清楚概念
   - 再说明它解决什么问题
   - 给出整体流程
   - 涉及代码时使用小例子，并分步骤解释
   - 适合面试准备阶段阅读，不要过度深入
   - 结尾给一句记忆口诀

5. **命名规则**：
   - 默认使用能概括主题的英文短文件名
   - 文件名使用小写字母、数字和连字符
   - 例如：`agent-code-review.md`

6. **输出结果**：
   - 完成后告诉用户新建或更新了哪篇文章
   - 简要说明文章主要总结了哪些内容
