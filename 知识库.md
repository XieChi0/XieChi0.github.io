# 博客知识库与学习进度系统

## 当前目标

这个项目要在个人 Astro 博客里逐步建立一个“个人学习进度管理 + 博客知识索引系统”。第一阶段不是完整 AI 知识库，而是先验证一个最小可用闭环：

- 在博客里新增一个私人学习中心页面。
- 通过密码登录后读取云端学习数据。
- 从博客文章内容生成文章索引。
- 对文章手动记录学习进度，例如 0%、25%、50%、75%、100%。
- 进度保存在 Supabase，电脑和手机可以同步。
- 先预留后续目标管理、每日复盘、知识库问答所需的数据结构。

后续如果继续推进，系统会扩展为：

- 按主题和截止日期规划学习目标。
- 记录每天学习了哪些文章、进度变化多少。
- 对博客文章做切片、索引和向量化。
- 基于文章内容做个人知识库问答和学习规划。

## 当前阶段

目前处于“最小验证版”阶段。已经完成：

- 新增 `/studyProgress/` 页面。
- 在 About 页面添加“学习进度”入口。
- 使用 Supabase Auth 做密码登录。
- 使用 Supabase Postgres 保存学习进度。
- 从 `src/content/posts` 和 `src/content/external` 生成文章索引。
- 支持文章进度手动保存。
- 支持文章分类筛选和分页浏览。
- 支持创建学习目标、给目标关联多篇文章、计算目标完成率。
- 支持删除学习目标；删除目标不会删除博客文章，也不会删除文章当前学习进度。
- 已验证远端 `study_progress` 和 `study_events` 表可以写入数据。
- 已发现并修复一次前端进度展示问题：数据库中已经有进度，但 Svelte 组件因为响应式依赖隐藏在函数内部，导致卡片仍显示 0%。现在改为显式生成 `visibleRows` 后再渲染。

当前还没有实现：

- 目标编辑、目标内文章移除、目标排序。
- 艾宾浩斯复习计划/复习提醒。
- 自动滚动阅读百分比记录。
- 每日复盘页面。
- AI 问答。
- 文章切片和 embedding 写入。

## 当前技术栈

- 前端框架：Astro 5
- 交互组件：Svelte 5
- 部署方式：GitHub Pages 静态部署
- 云数据库：Supabase Postgres
- 登录方式：Supabase Auth email/password
- 前端公开 key：Supabase Publishable Key
- 权限控制：Supabase Row Level Security
- 包管理：pnpm

页面上只展示密码输入框，但内部使用固定 owner email 加用户输入的密码登录 Supabase。真实密码不写入项目。

## 数据流

当前学习进度流程：

1. `studyProgress.astro` 构建时调用 `getStudyProgressArticles()`。
2. `getStudyProgressArticles()` 从博客内容集合拿到文章 slug、标题、分类、标签和 URL。
3. 页面加载 `StudyProgressApp.svelte`。
4. 用户输入密码。
5. `supabase.auth.signInWithPassword()` 使用 `PUBLIC_STUDY_OWNER_EMAIL` 和用户输入的密码登录。
6. 登录成功后调用 `listProgress()` 读取 `study_progress`。
7. 同时调用 `listGoals()` 读取 `study_goals` 和 `study_goal_items`。
8. 页面把文章索引和进度记录合并成可见行，并支持按分类筛选和分页。
9. 用户点击 0/25/50/75/100。
10. 前端先做乐观更新，再写入 `study_progress`。
11. 保存成功后追加一条 `study_events` 日志。
12. 目标模式中可创建目标、给目标添加文章，并按关联文章进度计算目标完成率。

## 数据库模型

数据库真实表在 Supabase。项目里的模型说明位于：

```txt
docs/studyProgress/schema.prisma
```

这个文件是“Prisma 风格伪 schema”，只是为了让人和后续 AI 理解结构，不是真的使用 Prisma。

### StudyProgress -> public.study_progress

记录“某个用户对某篇文章当前看到哪里”。这是当前版本最核心、已经使用的表。

字段含义：

- `id` -> `id`：主键，UUID。
- `userId` -> `user_id`：Supabase `auth.users.id`，用于区分用户和 RLS 权限。
- `articleId` -> `article_id`：文章 slug，例如 `个人/护肤`。
- `articleSource` -> `article_source`：文章来源，`posts` 或 `external`。
- `articleTitle` -> `article_title`：保存进度时的文章标题快照。
- `articleUrl` -> `article_url`：文章访问 URL，例如 `/posts/个人/护肤/`。
- `progressPercent` -> `progress_percent`：当前进度，0 到 100。
- `status` -> `status`：状态，`not_started`、`reading`、`completed`。
- `lastReadAt` -> `last_read_at`：最后阅读/保存进度时间。
- `createdAt` -> `created_at`：创建时间。
- `updatedAt` -> `updated_at`：更新时间。

唯一约束：

- 同一个用户、同一个来源、同一篇文章只能有一条当前进度。

### StudyEvent -> public.study_events

记录“进度变化日志”。当前版本已经写入，用于后续每日复盘。

字段含义：

- `id` -> `id`：主键，UUID。
- `userId` -> `user_id`：Supabase 用户 ID。
- `articleId` -> `article_id`：文章 slug。
- `articleSource` -> `article_source`：文章来源。
- `articleTitle` -> `article_title`：文章标题快照。
- `previousPercent` -> `previous_percent`：变化前进度。
- `currentPercent` -> `current_percent`：变化后进度。
- `eventDate` -> `event_date`：日期桶，用于按天汇总。
- `createdAt` -> `created_at`：日志创建时间。

注意：

- 这个表是追加日志，不代表当前状态。
- 当前状态应该看 `study_progress`。

### StudyGoal -> public.study_goals

记录“某个主题在某个日期前学完”的学习目标。当前版本已经用于目标创建、展示、删除和完成率计算。

字段含义：

- `id` -> `id`：主键，UUID。
- `userId` -> `user_id`：Supabase 用户 ID。
- `title` -> `title`：目标标题。
- `topic` -> `topic`：主题名，后续用于规划和分组。
- `targetDate` -> `target_date`：目标截止日期。
- `status` -> `status`：目标状态，`active`、`paused`、`completed`、`archived`。
- `notes` -> `notes`：目标备注。
- `createdAt` -> `created_at`：创建时间。
- `updatedAt` -> `updated_at`：更新时间。

### StudyGoalItem -> public.study_goal_items

记录“目标关联了哪些文章”。当前版本已经用于把多篇文章加入同一个目标。

字段含义：

- `id` -> `id`：主键，UUID。
- `userId` -> `user_id`：Supabase 用户 ID，冗余保存是为了简化 RLS 和查询。
- `goalId` -> `goal_id`：所属目标 ID。
- `articleId` -> `article_id`：文章 slug。
- `articleSource` -> `article_source`：文章来源。
- `articleTitle` -> `article_title`：文章标题快照。
- `articleUrl` -> `article_url`：文章 URL。
- `targetPercent` -> `target_percent`：该文章在目标中要求达到的进度，通常是 100。
- `sortOrder` -> `sort_order`：目标内排序。
- `createdAt` -> `created_at`：创建时间。

唯一约束：

- 同一个目标中，同一篇文章不能重复添加。

### ArticleChunk -> public.article_chunks

预留给未来知识库问答。用于保存文章切片，后续可以接 embedding。

字段含义：

- `id` -> `id`：主键，UUID。
- `articleId` -> `article_id`：文章 slug。
- `articleSource` -> `article_source`：文章来源。
- `articleTitle` -> `article_title`：文章标题。
- `chunkIndex` -> `chunk_index`：文章内切片序号。
- `content` -> `content`：切片文本。
- `metadata` -> `metadata`：JSON 元数据，例如标题层级、标签、原文位置。
- `createdAt` -> `created_at`：创建时间。

唯一约束：

- 同一篇文章中的同一个 `chunkIndex` 只能出现一次。

## 已知问题和注意事项

- 当前页面仍是最小验证版，UI 文案和布局还没有做成最终形态。
- 临时 debug 面板已经移除；如果页面显示和数据库不一致，优先临时打开浏览器 console 或在数据适配层补充日志。
- 当前文章进度是手动按钮记录，还没有自动滚动记录。
- 当前 `study_progress` 表记录的是文章当前进度，读到 100% 后不会自动按记忆衰减回到待复习状态。
- 艾宾浩斯复习计划还没有落库；后续需要单独设计“复习计划/复习记录/下次复习时间”，不要直接把文章进度清零。
- AI 问答不能只靠 GitHub Pages 前端完成，后续需要 NAS API、Supabase Edge Function 或其他后端保护 AI key。
- Supabase 在国内网络的稳定性需要继续观察；若不稳定，数据访问层已经预留 NAS adapter。

## 后续建议

下一阶段建议按这个顺序推进：

1. 设计艾宾浩斯复习模型：复习计划、复习记录、下次复习时间、逾期状态。
2. 完善目标模式：编辑目标、从目标中移除文章、目标内排序。
3. 增加每日复盘视图，从 `study_events` 汇总。
4. 增加文章详情页自动记录滚动百分比。
5. 设计 `article_chunks` 的生成脚本。
6. 决定 AI 问答后端：NAS、Supabase Edge Function 或其他服务。
