# 校园食品安全监管平台前端项目讲解文档

## 项目介绍

这个项目是一个面向校园食品安全监管场景的多端系统，不是单一后台，也不是普通 H5 页面集合，而是一套围绕学校食堂日常管理、供应商协同和教育监管搭建的业务平台。

它要解决的核心问题，是把校园食品安全相关的关键流程放到线上统一管理。比如学校侧的采购计划、食谱公示、每日留样、出入库、库存、食材流转，供应商侧的接单、发货、配送，以及教育局和管理侧的权限配置、数据监管和统计报表。这个项目的价值，就是把这些流程串成一条完整链路，让业务执行、过程留痕和监管查看可以放在同一套系统里完成。

从系统形态上看，它分成两类端：

- 后台管理端：更偏管理、配置、监督和统计，主要给教育局、管理人员或校方管理角色使用。
- H5 移动端：更偏现场操作，主要给学校食堂人员和供应商使用。



这个项目还有一个特点，就是业务广度比较大。它不是只做“采购”或者“食谱”中的某一个点，而是覆盖了采购计划、采购审批、供应商协同、库存、留样、流转、公示、监管统计等多个环节。所以前端在这里面对的挑战，不只是页面数量多，而是角色多、状态多、重复业务结构多，既要能快速承接需求，也要能把重复场景抽象出来，不然项目会很快变重。

我在这个项目里的工作，主要集中在三块：

- 第一块是后台 CRUD 页面字段的配置化抽象，也就是用统一 Schema 派生查询、表格、编辑和详情结构。
- 第二块是学校端、供应商端多个 H5 模块开发，把采购、接单、留样等业务流程真正落到移动端。
- 第三块是业务组件封装，把高频重复的筛选区、列表项、空状态、详情块这类结构沉淀下来，提高后续复用率和维护效率。

所以如果面试里让我先整体介绍这个项目，我会这样说：

> 这是一个校园食品安全监管平台，服务学校、供应商和教育局三类角色，包含后台管理端和 H5 移动端。它覆盖采购计划、食谱、公示、留样、库存、配送和监管统计等核心环节。我在里面做的不是单点页面，而是同时参与业务落地和工程抽象，一方面负责学校端、供应商端多个 H5 业务模块，另一方面也推进了报表配置化和业务组件封装，去解决项目里重复页面多、角色差异大、后期维护成本高的问题。

下面这三部分，就是我简历里最值得展开的内容。

## 一、亮点一：Schema 驱动的 CRUD 页面配置能力

### 1. 先校准这条亮点：源码里真正存在的是什么

简历里原来的说法是：

> 搭建 JSON Schema 驱动的动态统计报表引擎，支持嵌套、条件显隐、自定义校验、字段级权限控制，减少 70% 业务报表代码量。

这句话方向是好的，但按照当前项目源码，表述有两个风险。

第一，项目使用的并不是 JSON Schema 官方规范，而是一套前端自定义的 `CrudSchema` 元数据协议。它描述的是字段如何进入查询表单、数据表格、编辑表单和详情展示，更准确的名称应该是“Schema 驱动的 CRUD 页面配置能力”。

第二，当前校园餐统计页面大部分仍然采用手写模板。比如供应商月入库金额统计、资金统计等页面，查询条件、表格列和图表配置都直接写在 `.vue` 文件中。因此不能把整个统计模块都说成已经由 Schema 引擎生成，也不能在没有改造前后数据对比的情况下直接声称减少了 `70%` 代码量。

真正可以从代码中确认的能力是：

- 用一份 `CrudSchema[]` 字段配置，派生 `searchSchema`、`tableColumns`、`formSchema`、`detailSchema` 四套结构。
- 支持字段在不同场景独立显示，例如只参与搜索、不进入表格，或者只进入详情、不进入编辑表单。
- 支持字典字段自动转换为下拉选项和表格标签，减少重复的数据映射。
- 支持远程加载下拉选项、组件参数、插槽、格式化函数和默认值。
- 表格渲染层支持递归 `children`，可以生成多级嵌套表头。
- 表单和表格暴露 `setSchema`、`setColumn` 等运行时修改能力，可以根据页面状态动态隐藏字段、禁用字段或调整组件参数。
- 校验最终复用 Element Plus 的规则体系，能够接入必填、格式校验和自定义 `validator`。
- 页面按钮权限已经通过 `v-hasPermi` 实现，但它和 Schema 还没有真正打通，不能直接说已实现 Schema 字段级权限。

所以，这一亮点最有价值的地方不是“写了一个 JSON 对象”，而是：

> 建立统一字段元数据，让同一个业务字段在查询、列表、编辑和详情四种页面形态中复用，减少字段定义散落和多处同步修改的问题。

### 2. 为什么要做：重复的不只是页面，而是字段定义

校园餐后台有大量相似页面，例如学校、供应商、采购计划、库存、台账和统计列表。它们通常都包含：

- 顶部查询条件。
- 中间数据表格。
- 新增或编辑弹窗。
- 详情抽屉或详情弹窗。
- 字典转换、日期格式化、远程下拉数据。
- 新增、导出、编辑、删除等权限按钮。

如果完全手写，一个“供应商”字段可能要重复定义四次：

1. 在搜索区定义一个供应商下拉框。
2. 在表格中定义供应商名称列。
3. 在编辑表单中再次定义供应商下拉框。
4. 在详情页中再次定义供应商展示项。

字段名称、组件类型、字典、格式化规则一旦变化，开发人员需要同时修改多个文件或多个代码块。真正的问题不是某一页代码长，而是同一个业务含义被分散维护。

`CrudSchema` 的设计就是把这些重复信息收回到字段维度。

### 3. 核心数据结构：一个字段描述多个使用场景

真实代码位于：

`20250218/src/hooks/web/useCrudSchemas.ts`

它定义的核心结构可以简化为：

```ts
export type CrudSchema = {
  field: string
  label?: string

  isSearch?: boolean
  search?: CrudSearchParams

  isTable?: boolean
  table?: CrudTableParams

  isForm?: boolean
  form?: CrudFormParams

  isDetail?: boolean
  detail?: CrudDescriptionsParams

  children?: CrudSchema[]
  dictType?: string
  dictClass?: 'string' | 'number' | 'boolean'
}
```

这段设计的关键是：Schema 的最小单位不是“页面”，而是“字段”。

一个字段可以同时声明：

- 在搜索区使用什么组件。
- 在列表中是否展示、列宽多少、如何格式化。
- 在编辑表单中使用什么组件、默认值是什么。
- 在详情中如何展示。
- 是否使用系统字典。
- 是否包含下级字段。

这样做比简单地分别维护 `form: []` 和 `table: []` 更有价值，因为它避免了字段元数据在多个数组中重复。

### 4. 一份 Schema 如何派生四套页面结构

`useCrudSchemas` 接收一份 `CrudSchema[]`，然后分别执行四次转换：

```ts
export const useCrudSchemas = (crudSchema: CrudSchema[]) => {
  const allSchemas = reactive({
    searchSchema: [],
    tableColumns: [],
    formSchema: [],
    detailSchema: []
  })

  allSchemas.searchSchema = filterSearchSchema(crudSchema, allSchemas)
  allSchemas.tableColumns = filterTableSchema(crudSchema)
  allSchemas.formSchema = filterFormSchema(crudSchema, allSchemas)
  allSchemas.detailSchema = filterDescriptionsSchema(crudSchema)

  return { allSchemas }
}
```

这里体现的是一个典型的“中间模型 + 多目标转换”思路：

```text
                 ┌─> Search 查询表单
CrudSchema[] ────┼─> Table 数据表格
                 ├─> Form 新增/编辑表单
                 └─> Descriptions 详情展示
```

页面层不再分别维护四套字段，而是消费转换后的结果：

```vue
<Search :schema="allSchemas.searchSchema" />
<Table :columns="allSchemas.tableColumns" />
<Form :schema="allSchemas.formSchema" />
<Descriptions :schema="allSchemas.detailSchema" />
```

面试时可以把这里概括为：

> 我没有把 Schema 只当作表单配置，而是把它设计成页面字段的统一元数据，再通过不同转换器生成查询、表格、编辑和详情结构。

现有文档原来的例子只有两个查询字段和两个表格列，无法体现配置化的价值。下面用“学校采购统计”构造一个更完整的例子。

注意：这是基于项目现有 `CrudSchema` 接口整理的讲解示例，用来展示引擎已经具备的组合能力，不代表该统计页面当前已经完成配置化改造。

### 6. 嵌套是怎么实现的：递归转换与递归渲染

“支持嵌套”不能只在配置里写一个 `children`，还必须保证转换层和渲染层都能递归处理。

在 `useCrudSchemas.ts` 中，表格结构通过 `treeMap` 递归转换：

```ts
const filterTableSchema = (crudSchema: CrudSchema[]) => {
  const tableColumns = treeMap<CrudSchema>(crudSchema, {
    conversion: (schema) => {
      if (schema.isTable !== false && schema.table?.show !== false) {
        return {
          ...schema.table,
          ...schema
        }
      }
    }
  })

  return filter(tableColumns, (data) => !!data.field)
}
```

在 `components/Table/src/Table.vue` 中，表格列又根据 `children` 递归生成 `ElTableColumn`：

```tsx
const renderTreeTableColumn = (columns: TableColumn[]) => {
  return columns.map((column) => (
    <ElTableColumn {...column} prop={column.field}>
      {{
        default: (data) =>
          column.children?.length
            ? renderTableColumn(column.children)
            : column.formatter?.(
                data.row,
                data.column,
                data.row[column.field],
                data.$index
              ) ?? data.row[column.field]
      }}
    </ElTableColumn>
  ))
}
```

因此这里的嵌套不是概念描述，而是：

1. Schema 数据结构允许 `children`。
2. 转换器递归保留子节点。
3. 表格渲染器递归生成多级列。
4. 叶子节点再执行插槽或 `formatter`。

这种结构适合校园餐统计中常见的“采购数据 -> 数量、金额、次数”或“月份 -> 入库量、出库量、库存量”等组合表头。

### 7. 条件显隐：当前实现是运行时修改，不是声明式规则引擎

表单 Schema 中已经存在 `hidden` 字段：

```ts
export type FormSchema = {
  field: string
  label?: string
  component?: ComponentName
  hidden?: boolean
}
```

表单渲染时会过滤隐藏字段：

```ts
schema
  .filter((item) => !item.hidden)
  .map((item) => renderFormItem(item))
```

同时，通用 Form 组件暴露了 `setSchema`：

```ts
const setSchema = (schemaProps: FormSetPropsType[]) => {
  for (const schemaItem of schema) {
    for (const item of schemaProps) {
      if (schemaItem.field === item.field) {
        set(schemaItem, item.path, item.value)
      }
    }
  }
}
```

因此可以根据业务状态动态修改字段：

```ts
watch(
  () => formModel.reportDimension,
  (dimension) => {
    setSchema([
      {
        field: 'supplierId',
        path: 'hidden',
        value: dimension !== 'supplier'
      },
      {
        field: 'schoolId',
        path: 'componentProps.disabled',
        value: dimension === 'currentSchool'
      }
    ])
  }
)
```

这里要准确表达：

- 项目具备动态显隐和动态禁用的底层能力。
- 当前方式是页面监听业务状态，再调用 `setSchema` 修改配置。
- 它还不是 `visibleWhen: { field, operator, value }` 这种完全声明式的条件规则引擎。

所以面试时可以说“支持运行时条件显隐”，不要直接夸大成“实现了复杂规则编排引擎”。

### 8. 自定义校验：Schema 负责承载，Element Plus 负责执行

通用 Form 在渲染字段时，会把 `formItemProps` 透传给 `ElFormItem`：

```tsx
<ElFormItem
  {...(item.formItemProps || {})}
  prop={item.field}
  label={item.label || ''}
>
  {renderComponent(item)}
</ElFormItem>
```

因此校验能力不需要在引擎里重新实现，而是复用 Element Plus 的成熟规则：

```ts
{
  field: 'totalAmount',
  label: '采购金额',
  form: {
    component: 'InputNumber',
    formItemProps: {
      rules: [
        { required: true, message: '请输入采购金额' },
        {
          validator: (_rule, value, callback) => {
            if (value <= 0) {
              callback(new Error('采购金额必须大于 0'))
              return
            }
            callback()
          },
          trigger: 'blur'
        }
      ]
    }
  }
}
```

这个设计的价值是职责清晰：

- Schema 描述“这个字段需要什么规则”。
- 通用 Form 负责把规则绑定到正确字段。
- Element Plus 负责触发时机、错误状态和错误文案展示。

### 9. 字段级权限：当前源码的真实边界

项目已经有按钮级权限能力，例如资金统计页面中的新增和导出按钮：

```vue
<el-button
  v-hasPermi="['business:stock-in-data:create']"
  @click="openForm('create')"
>
  新增
</el-button>

<el-button
  v-hasPermi="['business:stock-in-data:export']"
  @click="handleExport"
>
  导出
</el-button>
```

但 `CrudSchema` 类型中目前没有 `permissions`、`readRoles`、`writeRoles` 等字段，`useCrudSchemas` 也没有调用 `checkPermi` 过滤字段。

所以当前项目能够证明的是：

- 页面级和操作按钮级权限已经落地。
- Form 支持动态隐藏和禁用，因此具备扩展字段权限的技术基础。
- Schema 字段级权限尚未真正集成。

如果要让简历中的“字段级权限控制”完全成立，可以在现有结构上增加：

```ts
type FieldPermission = {
  read?: string[]
  write?: string[]
}

type CrudSchema = {
  field: string
  permissions?: FieldPermission
  // 其他配置
}
```

然后在 Schema 转换阶段统一处理：

```ts
function applyFieldPermission(
  schema: CrudSchema[],
  userPermissions: string[]
) {
  return schema
    .filter((item) =>
      item.permissions?.read
        ? item.permissions.read.some((code) =>
            userPermissions.includes(code)
          )
        : true
    )
    .map((item) => ({
      ...item,
      form: {
        ...item.form,
        componentProps: {
          ...item.form?.componentProps,
          disabled: item.permissions?.write
            ? !item.permissions.write.some((code) =>
                userPermissions.includes(code)
              )
            : false
        }
      }
    }))
}
```

这样才算真正把权限从页面模板收进 Schema：

- 无读取权限：字段不进入查询、表格、表单和详情结构。
- 有读取权限但无写权限：字段可见但禁用。
- 有写权限：字段正常编辑。

这一段可以作为“我会如何继续演进”的设计题来讲，但不能冒充当前源码已经完成的功能。

### 10. 这套能力真正减少了哪些代码

它减少的主要不是接口请求和业务计算，而是以下重复代码：

- 查询表单中的字段组件声明。
- 表格列声明。
- 新增、编辑表单中的字段声明。
- 详情项声明。
- 字典选项获取和状态格式化。
- 日期、金额、状态等通用展示逻辑。
- 字段显示范围和组件参数的重复维护。

它不能自动消除的代码包括：

- 特殊统计口径和数据聚合。
- ECharts 图表配置。
- 跨字段复杂联动。
- 特殊导出、打印和审批流程。
- 高度定制化的表格单元格。

这也是为什么统计页面不能简单地理解为“有了 Schema 就完全不用写代码”。更合理的架构是：

```text
Schema 层：字段、组件、列、字典、格式、基础规则
通用容器层：查询、重置、分页、加载、表格、详情
业务扩展层：统计口径、图表、汇总、导出、打印、权限
```

### 11. “减少 70% 代码量”应该怎么证明

如果简历保留 `70%`，面试官很可能追问计算方式。至少要准备一个可复现的对比：

```text
改造前：
查询表单 80 行
表格列 100 行
编辑表单 120 行
详情展示 60 行
合计 360 行

改造后：
字段 Schema 90 行
页面组合与业务逻辑 30 行
合计 120 行

重复结构代码减少：
(360 - 120) / 360 = 66.7%
```

只有在真实选取 2 至 3 个同类页面统计后，才能把结果写成 `约 60%`、`约 70%`。如果目前没有做过测量，建议先删除具体百分比，避免一个本来不错的工程亮点被数字拖累。

### 12. 推荐的简历表述

基于当前代码，最稳妥的版本是：

> 基于 Vue3 + TypeScript 搭建 Schema 驱动的 CRUD 页面配置能力，以统一字段元数据派生查询、表格、编辑和详情结构，支持多级表头、远程选项、字典映射、自定义格式化与运行时字段显隐，降低同类管理页面的重复开发和多处维护成本。

如果后续把字段权限真正接入 Schema，并完成代码量统计，可以升级为：

> 搭建 Schema 驱动的动态报表与 CRUD 页面引擎，以统一字段元数据生成查询、表格、编辑和详情结构，支持嵌套表头、条件显隐、自定义校验及字段读写权限；在同类报表改造中减少约 70% 的重复结构代码。

### 13. 面试时可以直接这样讲

> 项目后台有很多查询、表格、编辑和详情结构相似的页面。最初每个字段会在四个位置重复定义，字段名称、字典或格式变化时需要多处同步修改。
>
> 我把字段抽成统一的 `CrudSchema` 元数据，通过 `useCrudSchemas` 分别转换成查询 Schema、表格列、编辑表单 Schema 和详情 Schema。表格侧通过 `children` 和递归渲染支持多级表头；表单侧通过 `hidden`、`setSchema` 支持运行时显隐和禁用；校验规则通过 `formItemProps` 透传给 Element Plus；字典和远程下拉选项也统一在字段配置中处理。
>
> 这套能力最主要的收益是避免同一个字段在多个页面结构中重复声明，让新增同类页面更多变成配置字段和补充业务逻辑。当前项目的按钮权限已经落地，但字段权限还没有完全并入 Schema，所以我会把它作为后续演进方向，而不会说成已经全部完成。

## 二、亮点二：多模块 H5 移动端布局与交互适配

### 1. 先校准简历中的说法

简历原文是：

> 开发学校 / 供应商端 15+ H5 模块，适配 10+ 移动端机型。

“开发 15+ H5 模块”可以从项目目录和提交记录中得到支持。学校端、供应商端确实覆盖了首页、我的、待接单、配送单、发货单、退货单、资质、新品申购等多个模块。

但“适配 10+ 移动端机型”目前缺少明确证据。代码仓库中没有设备测试清单、兼容性报告或自动化截图记录，因此面试官如果继续追问“是哪十台设备、什么系统版本、发现过什么兼容问题”，很容易答不上来。

这条亮点更应该回答的是：

> 面对不同手机宽度、长内容、固定操作区、图片比例和触摸操作，我具体采用了哪些布局与交互策略。

从当前代码能够确认的适配能力主要有：

- 使用移动端 viewport 和 Vant 组件体系建立统一的移动页面基础。
- 使用百分比宽度、Flex、Grid 和 `flex: 1` 实现流式布局，避免按某一台手机写死整页宽度。
- 将后台表格信息改造成卡片式纵向布局，降低小屏横向空间压力。
- 对长标题、供应商名称等内容使用省略和弹性收缩，防止撑破布局。
- 对商品、检验报告等图片统一容器尺寸和裁剪方式，避免原图比例不同造成页面跳动。
- 对长表单和长详情页采用“内容区滚动 + 底部操作栏固定”的结构，并为正文预留底部空间。
- 使用移动端弹层、日期选择器、上传器、Toast 等触摸交互组件，减少原生输入控件在不同浏览器中的差异。

### 2. 项目采用的适配方案，不是简单把 PC 页面缩小

这个 H5 项目没有引入 `postcss-pxtorem` 或 `postcss-px-to-viewport`，也没有把所有尺寸自动换算为 `rem` 或 `vw`。

它采用的是一套更直接的移动端实现方式：

```text
viewport 控制页面视口
        +
Vant 提供移动端组件尺寸和交互
        +
Flex / Grid / 百分比宽度负责流式布局
        +
卡片化信息结构减少横向内容
        +
滚动容器和固定操作栏处理长页面
```

所以面试时不要说“我基于 rem 做了全端等比缩放”，因为项目中没有这套实现。更准确的说法是：

> 我采用 viewport 配合流式布局，不追求所有元素按设计稿等比例缩放，而是优先保证内容不溢出、操作区始终可达、不同宽度下信息层级稳定。

### 3. 视口适配：让页面宽度跟随设备

项目入口设置了 viewport：

```html
<meta
  name="viewport"
  content="width=device-width, initial-scale=1.0,
           maximum-scale=1.0, user-scalable=no"
/>
```

它解决两个基础问题：

- `width=device-width` 让 CSS 视口宽度与手机设备宽度一致，避免浏览器按桌面页面宽度缩放。
- 限制页面缩放，减少现场操作时双击或手势误缩放导致按钮和表单位置变化。

不过要注意，禁止缩放会影响部分用户的可访问性。面试中可以补一句：

> 当时业务更关注现场录入稳定性，所以限制了页面缩放；如果面向公众长期使用，我会重新评估无障碍需求，优先通过合理字号和布局解决误触，而不是完全禁止缩放。

### 4. 小屏布局：把横向表格改成纵向卡片

后台管理端可以横向展示很多列，但手机宽度有限。如果把订单号、供应商、数量、金额、状态和操作按钮都放进表格，通常会出现横向滚动或文字被压得很窄。

供应商端页面采用卡片式结构：

```vue
<div class="order-card">
  <div class="card-header">
    <span class="order-no">{{ order.orderNo }}</span>
    <span class="status">{{ order.statusName }}</span>
  </div>

  <div class="card-body">
    <div class="info-row">
      <span class="label">学校</span>
      <span class="value">{{ order.schoolName }}</span>
    </div>
    <div class="info-row">
      <span class="label">配送日期</span>
      <span class="value">{{ order.deliveryDate }}</span>
    </div>
  </div>
</div>
```

这里的适配思路是：

- 一条业务记录占一张卡片。
- 卡片内部按照“标题和状态 -> 关键字段 -> 操作”纵向排列。
- 重要字段固定在前面，次要信息放到详情页。
- 页面主要沿纵向滚动，避免用户同时处理横向和纵向两个滚动方向。

这是真正的移动端信息架构适配，而不只是修改字体大小。

### 5. 流式布局：固定关键尺寸，让剩余空间自动分配

项目中的列表项和详情项大量使用 Flex：

```css
.info-row {
  display: flex;
  align-items: center;
}

.label {
  flex-shrink: 0;
  margin-right: 10px;
}

.value {
  flex: 1;
  min-width: 0;
}
```

这几个属性各自解决不同问题：

- `flex-shrink: 0`：标签、图标和按钮不被压缩到不可识别。
- `flex: 1`：内容区占据剩余宽度，而不是写死具体像素。
- `min-width: 0`：允许 Flex 子项真正收缩，否则长文本仍可能撑破父容器。

对于首页功能入口，项目使用等分 Grid：

```css
.function-grid {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  gap: 16px;
}
```

`1fr` 表示四列平分可用宽度。屏幕变宽时每列自动变宽，屏幕变窄时每列自动收缩，不需要分别计算不同手机的卡片宽度。

### 6. 长文本适配：允许内容收缩，并控制展示边界

学校名称、供应商名称、订单号和备注都可能超出预估长度。项目中使用了典型的单行省略方案：

```css
.title {
  min-width: 0;
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

这一组属性要一起使用：

- `white-space: nowrap` 不换行。
- `overflow: hidden` 隐藏超出部分。
- `text-overflow: ellipsis` 显示省略号。
- 在 Flex 容器中补上 `min-width: 0`，否则省略可能不生效。

标题栏也采用相同思路，将中间标题限制在可用宽度内：

```css
.header-center {
  max-width: 60%;
}

.header-title {
  white-space: nowrap;
  overflow: hidden;
  text-overflow: ellipsis;
}
```

这能避免标题过长时覆盖左右两侧的返回按钮和操作按钮。

### 7. 图片适配：统一展示尺寸，不让原图比例破坏卡片

食材图片和检验报告的原始尺寸并不统一。项目中通过固定展示容器和 `object-fit: cover` 控制图片：

```css
.image-container {
  width: 60px;
  height: 60px;
  overflow: hidden;
  border-radius: 8px;
  flex-shrink: 0;
}

.image-container img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

这里解决的是：

- 不同图片比例不会改变卡片高度。
- 图片始终占据稳定区域，右侧文字获得可预测的剩余宽度。
- `cover` 保持图片比例，通过裁剪填满容器，避免图片被压扁。
- `flex-shrink: 0` 防止小屏下图片被压缩。

项目还使用 `van-image`、`van-uploader` 等移动组件处理图片展示和上传，统一预览、删除、数量限制等交互。

### 8. 长页面适配：固定操作栏，同时避免遮挡正文

供应商配送单详情属于典型的长表单：一张订单可能包含多种食材、数量、价格、生产日期和检验报告。如果把保存按钮放在页面末尾，用户需要滚动到底部才能提交。

项目将总计和操作按钮固定在底部：

```css
.bottom-action {
  position: fixed;
  left: 0;
  right: 0;
  bottom: 0;
  z-index: 1000;
  display: flex;
  align-items: center;
  justify-content: space-between;
  padding: 10px 16px;
  background: #fff;
  box-shadow: 0 -2px 8px rgba(0, 0, 0, 0.1);
}
```

同时必须给正文预留空间：

```css
.detail-content {
  padding: 16px;
  padding-bottom: 100px;
}
```

这是一个很适合面试追问的点。只写 `position: fixed` 会导致最后几行内容被操作栏盖住，所以正文的 `padding-bottom` 至少要大于操作栏高度。

如果继续完善，还应该加上 iPhone 安全区：

```css
.bottom-action {
  padding-bottom: calc(10px + env(safe-area-inset-bottom));
}

.detail-content {
  padding-bottom: calc(100px + env(safe-area-inset-bottom));
}
```

当前源码没有完整处理 `safe-area-inset-bottom`，所以这属于可以继续优化的地方，不能说成已经完成。

### 9. 滚动适配：让页面壳稳定，只滚动内容区

在食材选择等页面中，项目使用：

```css
.page {
  height: 100vh;
  overflow: hidden;
}

.list-content {
  overflow-y: auto;
}
```

这种结构的作用是：

- 顶部筛选区和底部操作区位置稳定。
- 中间列表独立滚动。
- 避免整个页面滚动时操作区域被带走。

项目中还存在一个通用 `useLoadMore` Hook，通过计算滚动容器距离底部的距离触发下一页：

```ts
const distance =
  el.scrollHeight - el.scrollTop - el.clientHeight

if (distance < threshold && canLoadMore() && !loading.value) {
  loading.value = true
  await loadDataFn()
  loading.value = false
}
```

它解决了两个移动端长列表问题：

- 距离底部一定阈值时提前加载，减少用户看到空白等待。
- 使用 `loading` 锁防止一次滚动连续发出多个请求。

不过根据 Git 记录，这个 Hook 由其他成员提交。可以把它作为“项目采用的长列表方案”理解，但不要在面试中说成是自己独立封装的成果。

### 10. 触摸交互适配：优先使用移动端组件

项目使用 Vant 的 `Button`、`Popup`、`DatePicker`、`Uploader`、`Tabs`、`Toast` 等组件，而不是直接套用 PC 端交互。

它们带来的价值包括：

- 按钮和表单项具有更适合触摸的默认高度。
- 日期选择采用底部弹层和滚轮，避免手机原生日期控件表现不一致。
- 上传组件支持拍照或相册选择。
- Toast 不依赖鼠标悬停，能提供短时操作反馈。
- Tabs、弹窗和遮罩具备移动端手势和滚动锁定处理。

页面中的按钮通常保持约 `40px` 高度，图标入口也采用较大的点击区域，而不是只让文字本身可点击。这类“命中区域适配”比单纯调整视觉尺寸更重要。

### 11. “适配 10+ 机型”应该怎样成立

设备适配是测试结论，不是只看 CSS 就能得出的结论。如果简历保留“10+ 机型”，至少应准备一份测试矩阵：

| 维度 | 示例 |
| --- | --- |
| iOS | iPhone SE、iPhone 12/13、iPhone 14 Pro Max |
| Android | 小米、华为、荣耀、OPPO、vivo 不同尺寸机型 |
| 屏幕宽度 | 320、360、375、390、414、430 CSS px |
| 浏览器环境 | Safari、Chrome、微信内置浏览器 |
| 检查项 | 页面溢出、底栏遮挡、弹层滚动、上传、键盘顶起、长文本、图片比例 |

测试后还要能回答：

- 最小支持宽度是多少。
- 是否测试过 iOS Safari。
- 是否测试过微信内置浏览器。

当前仓库没有这份记录。因此现阶段建议将“适配 10+ 机型”改成“适配主流移动端屏幕尺寸”，或者直接写清采用的技术措施。

### 12. 推荐的简历表述

基于当前代码，推荐写成：

## 三、亮点三：业务组件封装与复用能力建设

 \-  ***\*组件封装：\****封装高可复用业务组件***\*25+\****，组件复用率提升***\*50%\****；

### 1. 封装了什么

- 筛选区
- 列表卡片
- 空状态
- 底部操作栏
- 详情模块

### 2. 真实代码一：采购计划页已经开始按业务组件拆分

看采购计划页的引入：`h5/CampusFoodSafety-FrontendHub-H5/food-ui-h5/src/view/School/purchasing/index.vue:124`

```vue
import SearchForm from './components/SearchForm/index.vue';
import PurchasingItem from './components/purchasingItem/index.vue';
import Empty from './components/empty/index.vue';
```

- `SearchForm`：负责筛选逻辑和筛选交互。
- `PurchasingItem`：负责采购计划列表项展示。
- `Empty`：负责统一空状态。

也就是说，页面本身只负责组合，不负责所有细节。

### 3. 如果我来讲自己的封装思路，我会怎么说

我会把组件分成三层。

第一层是基础展示层。

> 比如空状态、通用卡片壳子、标签、状态块。

第二层是交互层。

> 比如筛选区、日期筛选、底部操作栏、弹窗编辑区。

第三层是业务语义层。

> 比如采购计划项、订单卡片、留样列表项、详情信息块。

我封装组件不是为了“看起来有组件化”，而是为了让相似业务场景共享结构、交互和样式，从而降低新增需求成本。

对于基础展示层可以先用官方的UI组件，但是如果觉得在这之上还是有需求的话，可以新建一个组件页面，把原来的那个组件引用过来，在这之上进行修改，如果是样式有修改的话可以用深度选择器进行修改

| 文件夹路径                | 存放内容                                                     | 核心特点                                        |
| ------------------------- | ------------------------------------------------------------ | ----------------------------------------------- |
| `src/components`          | 项目所有可复用 Vue 组件总目录，按类型 / 模块拆分子文件夹     | 包含 DOM 结构、样式、页面模板，具备 UI 视图能力 |
| `src/components/common`   | 通用基础组件（弹窗、加载动画、二次封装 UI 框架组件、通用按钮等） | 全项目所有页面均可复用，和具体业务无关          |
| `src/components/business` | 通用业务组件（订单列表、用户信息卡片、权限选择器等）         | 依附业务场景，多个业务页面会重复使用            |
| `src/components/user`     | 用户模块专属组件（用户头像、用户详情、角色选择组件等）       | 仅**用户相关页面**使用，按业务模块划分          |
| `src/components/order`    | 订单模块专属组件（订单状态标签、订单明细卡片等）             | 仅**订单相关页面**使用，按业务模块划分          |
| `src/utils`               | 纯工具函数（接口请求封装、权限处理、日期格式化、表单校验、本地存储、防抖节流等） | 无 UI、无模板，纯逻辑代码，全项目全局通用       |

### 组件是怎么参数化的

为了让这条亮点更像你真的做过，我建议这里也补一段示意代码，说明你封装的不是死组件，而是可复用业务组件。

```vue
<template>
  <StatusCard
    title="配送单 #20240601"
    status="pending"
    statusText="待处理"
    tagType="warning"
    description="当前订单待学校确认收货"
    :actions="cardActions"
    @action="handleCardAction"
  />
</template>
```

这段示意代码要讲的是：

- 我封装的不是某一页专用组件。
- 同一套卡片结构，可以服务待接单、采购计划、留样记录等不同页面。
- 页面差异主要体现在传参，而不是重新写结构。

### 4. 这条亮点最后怎么总结

> 我在项目里持续把高频重复结构往业务组件方向收，比如筛选区、列表项、空状态、详情块。这样页面本身更薄，新增需求更多是在复用组件和调整参数，而不是整页重写。
