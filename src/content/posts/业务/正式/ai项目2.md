## 大体说说

根据文档，建议按这个顺序学习：

```
前端入口：useAiChat.ts（发送消息 + SSE 解析）
前端页面：index.vue（路由同步 + 会话 UI）
后端会话：AiChatServiceImpl.java（会话创建 / 续接）
后端 Agent 循环：AgentLoop.java（核心编排逻辑）
后端 SSE 推送：ToolStreamCallback.java
LLM 调用：QsmLlmProvider.java（模型协议适配）
```

第二个agent的入口是：

```
前端：useAiChat.ts 里的 sendMessage()
后端：AgentLoop.java 里的 run()
```

这些代码文件分为**前端**（交互展示 + 流式处理）和**后端**（会话管理 + Agent 核心逻辑 + SSE 推送）两大模块，核心是实现 “用户发消息→后端 Agent 推理 / 调用工具→前端流式展示结果” 的 AI 对话流程。

------

## 一、前端文件（负责交互与流式接收）

前端核心是「页面展示（index.vue）+ 逻辑处理（useAiChat.ts）」的分离架构，index.vue 管 “看”，useAiChat.ts 管 “做”。

1. **index.vue（页面展示 + 路由同步）**

核心定位

Vue 页面组件，是用户视觉交互的入口，负责 UI 渲染和「URL 地址栏 ↔ 页面状态」的同步。

核心功能

- 布局渲染：

  - 左侧：会话列表（ChatSidebar），展示 “今天 / 昨天 / 7 天内” 的会话；
  - 右侧：主区域（无消息时显示欢迎页 ChatWelcome，有消息时显示聊天页 ChatConversation）。

- 路由同步

  （核心设计）：

  保证 URL 中的 

  ```
  conversationUid
  ```

  （会话 ID）与页面内 

  ```
  activeConversationUid
  ```

  （当前激活会话 ID）完全一致：

  - URL 变化（如浏览器后退 / 手动改地址）：通过 Vue `watch` 监听路由参数，触发会话切换；
  - 页面状态变化（如点击切换 / 删除会话、新建会话）：手动调用 `syncRoute` 同步 URL（避免异步调度打断 SSE 流）。

- **交互透传**：将用户操作（发送消息、切换 / 删除会话）转发给 `useAiChat.ts` 处理。

**2. useAiChat.ts（核心逻辑层，类 React Hook）**

核心定位

前端 “大脑”，管理会话相关状态 + 封装核心业务逻辑（SSE 流式处理、消息 / 会话管理）。

核心功能

##### （1）状态管理（核心数据）

| 状态名                  | 作用                  |
| :---------------------- | :-------------------- |
| `sessions`              | 所有会话列表          |
| `activeConversationUid` | 当前激活的会话 ID     |
| `messages`              | 当前会话的消息列表    |
| `loading`               | AI 是否正在回复       |
| `toolStatus`            | AI 当前调用的工具状态 |
| `taskRunning`           | 后台是否有任务在运行  |

##### （2）核心函数（核心动作）

| 函数名            | 核心作用                                                     |
| :---------------- | :----------------------------------------------------------- |
| `sendMessage()`   | 用户发消息的入口，区分「新 / 老会话」逻辑：✅ 老会话：先加用户消息 + AI 占位消息（乐观更新），再发 SSE 请求；✅ 新会话：先发 SSE 请求，等后端返回`meta`事件（含新会话 UID）后，再创建会话 + 加消息 |
| `streamSSE()`     | SSE 流式接收核心：打开连接→读取数据流→缓冲拼接（处理网络分片）→识别事件→分发回调（如 onText 更新文字、onToolStart 更新工具状态） |
| `selectSession()` | 切换会话                                                     |
| `loadSessions()`  | 加载所有会话列表                                             |
| `loadMessages()`  | 加载某个会话的历史消息                                       |
| `deleteSession()` | 删除会话并同步路由                                           |

##### （3）SSE 处理核心逻辑

`streamSSE()` 是前端流式展示的关键，核心是 “按 `\n\n` 分割完整 SSE 事件”，解决网络分片问题：

- 用 `buffer` 缓存数据流，确保只处理完整的 SSE 事件（避免 “半个事件” 导致的 bug）；
- 解析 `event:`（text/tool_start/meta/error 等）和 `data:` 字段，分发给对应回调更新 UI；
- 支持 `AbortController` 取消连接（用户快速发新消息时，终止旧 SSE 流，避免消息混叠）。

------

**二、后端文件（负责会话管理 + Agent 推理 + SSE 推送）**

后端核心是「总调度（AiChatServiceImpl）+ 大脑（AgentLoop）+ 回调桥梁（ToolStreamCallback）」的三层架构。

**1. AiChatServiceImpl.java（总调度：会话管理 + SSE 入口）**

核心定位

后端 API 入口，兼具 “前台 + 服务员” 角色：前台管会话生命周期，服务员管 SSE 推送。

核心功能

- 会话管理：校验用户权限、创建 / 续接会话、存储用户消息、生成 `conversationUid`；
- SSE 入口：接收前端请求后，启动工作线程执行 Agent 逻辑，同时搭建 SSE 连接，将后端事件（文字 / 工具调用 / 错误）实时推送给前端；
- 事件转发：接收 `ToolStreamCallback` 的广播事件，封装为 SSE 格式（meta/text/tool_start 等）推送给前端。

**2. AgentLoop.java/AgentLoopImpl.java（Agent 核心大脑：ReAct 循环）**

核心定位

实现 ReAct（Reasoning + Acting，推理 + 行动）逻辑，是 AI 处理用户请求的核心。

核心功能

- 意图识别：匹配用户请求对应的 Skill（技能）；
- 循环推理（单次 Loop 流程）：
  1. 调用 LLM（通过 QsmLlmProvider 适配模型）；
  2. 解析 LLM 返回：提取文字内容 + 工具调用列表；
  3. 串行执行工具调用：逐个执行工具并获取结果；
  4. 结果回填：将工具结果返回给 LLM，进入下一轮 Loop；
- 终止条件：直到 LLM 仅返回纯文本（无工具调用），结束循环。

**3. ToolStreamCallback.java（回调桥梁：Agent ↔ SSE）**

核心定位

“广播员” 角色，解耦 AgentLoop 和 AiChatServiceImpl，是两者的事件传递桥梁。

核心功能

- 定义回调接口：包含文字片段、工具开始 / 结束、错误、完成等事件的回调方法；
- 事件转发：AgentLoop 每执行一步（如生成文字、调用工具），通过该接口触发回调，将事件传递给 AiChatServiceImpl，最终推送给前端。

**4. QsmLlmProvider.java（LLM 模型适配层）**

核心定位

屏蔽不同大模型的协议差异，统一 LLM 调用接口。

核心功能

- 协议适配：封装 GPT / 文心一言等不同模型的调用协议；
- 格式转换：将 AgentLoop 的请求转换为模型能识别的格式，再将模型返回转换为 AgentLoop 可解析的格式（如提取文字、工具调用列表）。

------

**三、核心流程串联（用户发消息为例）**

1. 前端：用户在 `index.vue` 点击发送 → 调用 `useAiChat.ts` 的 `sendMessage()` → `streamSSE()` 打开 SSE 连接；
2. 后端：`AiChatServiceImpl` 接收请求 → 启动 `AgentLoop` 执行 ReAct 循环（调用 LLM → 执行工具 → 回填结果）；
3. 回调：`AgentLoop` 每一步通过 `ToolStreamCallback` 广播事件 → `AiChatServiceImpl` 接收后封装为 SSE 推送给前端；
4. 前端：`streamSSE()` 解析 SSE 事件 → 回调更新 `messages` → `index.vue` 渲染 “打字机效果” 的文字 / 工具状态。

- 我们项目：先发请求，等 meta 后异步加入
- ChatGPT 等：先乐观更新，再发请求替换

但结果一样——用户发完消息，立刻看到"AI正在思考..."。

#### streamSSE()

你可以把它理解成一个"听收音机"的人：

```
后端        →→→→→→→  源源不断的数据流  →→→→→→→→  前端

                     streamSSE()
                     就是这个"人"
                     负责：
                     1. 把流打开
                     2. 一段一段接收数据
                     3. 识别这是什么类型（text/tool/error）
                     4. 告诉相应的处理函数
```

下面我们来详细理解：

**SSE**-Server-Sent Events（服务器发送事件）

SSE是指服务器单向、持续地向浏览器推数据。

> 后端不是全部想完把消息发给前端，而是每想一个字就发一个片段过来，对于用户来说，屏幕上的字是一个一个蹦出来。

为什么用 SSE 而不是其他方式？

| 方式           | 能不能用 | 原因                                 |
| :------------- | :------- | :----------------------------------- |
| 普通 HTTP 请求 | ❌        | 等全部返回才显示，用户要等很久       |
| WebSocket      | ✅ 可以   | 但太重了，需要双向通信、握手、心跳   |
| SSE            | ✅ 最适合 | 轻量、单向、浏览器原生支持、自动重连 |

---

**下面开始完整拆解streamSSE()**

```
┌─────────────────────────────────────────────────────────┐
│                    streamSSE() 五大步骤                    │
│                                                         │
│  1. 打开连接      →  fetch() 发起请求                     │
│  2. 读取数据流    →  getReader() 获取读取器               │
│  3. 缓冲拼接      →  buffer 攒数据，按 \n\n 切开           │
│  4. 识别事件      →  每块里找 event: 和 data:             │
│  5. 分发处理      →  调用对应的回调函数                    │
└─────────────────────────────────────────────────────────┘
```

##### 1 打开连接（发起请求）

```javascript
const res = await fetch(url, {
  method: "POST",
  headers: { "Content-Type": "application/json", ... },
  body: JSON.stringify(body),
  signal,  // 中断信号
});
```

大白话：这就是一个普通的 HTTP POST 请求，但是响应类型是 `text/event-stream`（SSE 格式）。

注意这里传了 `signal`，它是 `AbortController` 给的。**作用是：如果用户发新消息，可以取消当前的 SSE 连接。**

---

##### 2 获取数据流读取器

```javascript
const reader = res.body.getReader();
const decoder = new TextDecoder("utf-8");
let buffer = "";
```



```
浏览器收到的是一个"流"（不是一次性拿到全部内容）

getReader() 就像给你一个"水龙头"
每次拧一下，流出一小段数据

TextDecoder 把浏览器收到的"字节"转成"文字"

buffer 是一个"临时攒数据的杯子"
```

---

##### 3 最关键的 buffer 拼接逻辑

这是整个函数**最难、也最重要**的部分：

```javascript
while (true) {
  const { value, done } = await reader.read();  // 读一小段
  if (done) break;
  
  buffer += decoder.decode(value, { stream: true });  // 加入杯子
  buffer = buffer.replace(/\r\n/g, "\n");  // 统一换行符
  
  const parts = buffer.split("\n\n");  // 按空行切开
  buffer = parts.pop() || "";  // 最后一段不完整的留在杯子里
  
  for (const part of parts) {
    // 处理每一个完整的事件块
  }
}
```

为什么要用 buffer？

**因为网络是"一段一段"来的，不是一个完整消息一次性到。**

想象你在收快递：

```
buffer = ""
       │
       ▼
收到："event:te"     → buffer = "event:te"        （不完整，不处理）
收到："xt\ndata:你"  → buffer = "event:text\ndata:你"  （不完整，不处理）
收到："好！\n\n"     → buffer = "event:text\ndata:你好！\n\n"  （完整了！）
       │
       ▼
split("\n\n") 切开
       │
       ▼
["event:text", "data:你好！"]
       │
       ▼
处理这个完整事件
       │
       ▼
buffer 清空，等下一个事件
```

`split("\n\n")` 和 `pop()` 的作用

```
假设 buffer 里有： "event:text\ndata:你好\ndata:世界\n\n剩"
       │
       ▼
split("\n\n") → ["event:text\ndata:你好\ndata:世界", "剩"]
       │
       ▼
pop() 把最后那个不完整的 "剩" 拿出来留到下次
       │
       ▼
处理前面完整的 ["event:text\ndata:你好\ndata:世界"]
```

**这就是为什么流断了之后还要再处理一次 buffer**（代码第 232-244 行）——因为最后一个事件可能没有 `\n\n` 结尾。

---

##### 4 识别事件类型

```javascript
for (const line of part.split("\n")) {
  if (line.startsWith("event:")) {
    eventName = line.slice(6).trim();  // 取出事件名
  } else if (line.startsWith("data:")) {
    dataLines.push(line.slice(5).replace(/^ /, ""));  // 取出数据
  }
}
```

**SSE 协议格式长这样：**

```
event:text          ← 事件名
data:你好！           ← 数据内容
                     ← 空行表示一个事件结束

event:tool_start    ← 下一个事件
data:{"functionName":"search_destination"}
                     ← 空行
```

代码就是在做两件事：
1. 找 `event:` 后面的名字（text / tool_start / error 等）
2. 找 `data:` 后面的内容（真正的数据）

---

##### 5 分发给回调函数

```javascript
switch (name) {
  case "meta":      callbacks.onMeta?.(data);      break;
  case "text":      callbacks.onText(data);         break;
  case "error":     callbacks.onError(data);        break;
  case "tool_start": callbacks.onToolStart?.(data); break;
  case "tool_end":  callbacks.onToolEnd?.(data);    break;
}
```

**大白话**：streamSSE 只负责"收快递、拆快递、看快递单"，至于快递里的东西怎么处理，它不管，全交给外面传进来的回调函数。

```
streamSSE() 像一个前台接待
                    │
                    │ 收到不同类型的事件
                    ▼
        ┌──────┬──────┬──────┬──────┐
        │ text │tool_ │ error│ meta │
        │      │start │      │      │
        └──┬───┴──┬───┴──┬───┴──┬───┘
           │      │      │      │
           ▼      ▼      ▼      ▼
      onText()  onTool  onError  onMeta
                Start()
```

##### 完整流程走一遍

假设用户发了一条消息："帮我规划北京三日游"

```
streamSSE() 内部发生了什么？
════════════════════════════════════════════════════

时间轴：

t=0ms  用户发消息
t=10ms  fetch() 请求发出，等待后端响应
t=50ms 后端开始处理，SSE 连接建立
t=100ms 收到第一个 chunk（可能是工具调用）

  buffer 逐步积累：
  "event:tool_start\ndata:search_destination\n\n"
        │
        ▼ split("\n\n")
  处理：event="tool_start", data="search_destination"
        │
        ▼ 调用 onToolStart()
  toolStatus = "搜索目的地"  ← 页面上出现"正在搜索目的地..."

t=200ms 收到文字片段

  buffer = "event:text\ndata:你\n\n"
        │
        ▼ 处理
  onText("你")  → assistantReactive.content = "你"

t=220ms 收到下一段

  buffer = "event:text\ndata:好，\n\n"
        │
        ▼ 处理
  onText("好，")  → assistantReactive.content = "你好，"

t=240ms 又收到一段

  onText("我来帮你...")  → content = "你好，我来帮你..."

t=1000ms 流结束

  onDone()  → pending = false  → AI 回复完成
```

##### 函数优点讲解

**1：回调模式让逻辑分离**

```javascript
// streamSSE 只负责收数据
await streamSSE(url, body, {
  onText(chunk) { /* 前端决定怎么显示文字 */ },
  onToolStart(data) { /* 前端决定怎么展示工具状态 */ },
  onError(msg) { /* 前端决定怎么显示错误 */ },
});
```

**好处**：streamSSE 是一个纯工具函数，不关心业务逻辑，可以被复用。

**2：buffer 处理网络碎片**

网络传输会把数据切碎，代码用 buffer + `\n\n` 分割确保**只处理完整的事件**，不会出现"半个事件被当成完整事件处理"的 bug。

**3：AbortController 支持取消**

```javascript
if (abortController) {
  abortController.abort();  // 取消上一次的 SSE 连接
}
```

**好处**：用户快速发消息时，旧的 SSE 会被取消，不会出现"两条消息的文字混在一起"的 bug。

---

##### 总结

> "我们用 SSE 做流式输出。前端用 fetch + getReader 读取流，用一个 buffer 按 `\n\n` 分割完整事件，然后识别 event 和 data 字段，分发给不同的回调处理。文字通过 onText 回调一段一段拼到 assistant 消息的 content 上，实现打字机效果。"

#### **路由同步**

这个文件它还负责了路由同步,路由同步是指 `地址栏里的URL` 和 `页面内部的状态` 要保持一致。

URL 和当前会话要保持一致

```
┌─────────────────────────┐         ┌─────────────────────────┐
│     地址栏里的 URL        │  ←→     │   页面内部的状态          │
│                         │         │                         │
│  /ai-assistant/abc123   │         │  activeConversationUid  │
│                         │         │  = "abc123"             │
│  URL 里的 conversationUid│         │  当前正在看哪个会话        │
└─────────────────────────┘         └─────────────────────────┘
         这俩必须永远相等
```

**URL 和页面状态两个地方，谁变了都通知对方。URL 变用 watch 监听，页面变手动调用 syncRoute，核心是为了避免异步调度打断 SSE 流。**



场景 A：用户点浏览器"后退/前进"或手动改 URL

```
用户把地址从 /ai-assistant/aaa 改成 /ai-assistant/bbb

URL 变了，但页面内部 activeConversationUid 还是 "aaa"

→ 不一样了！需要把页面状态改成 "bbb"
```

场景 B：用户在页面里点击了某个会话

```
用户在左侧点击了会话 bbb

页面内部 activeConversationUid 变成了 "bbb"

但 URL 还是 /ai-assistant/aaa

→ 不一样了！需要把 URL 改成 /ai-assistant/bbb
```



**代码怎么解决的？**

**方向 1：URL 变了 → 通知页面**

用 Vue 的 `watch` 监听路由参数：

```javascript
watch(
  () => route.params.conversationUid,   // 监听 URL 里的 conversationUid
  (uid) => {
    if (uid !== activeConversationUid.value) {
      selectSession(uid);               // 让页面切到对应的会话
    }
  }
);
```



**方向 2：页面状态变了 → 通知 URL**

通过 `syncRoute` 函数手动改 URL：

```javascript
function syncRoute(uid) {
  router.replace({ name: "AiAssistant", params: { conversationUid: uid } });
}
```

这个函数在三个时机被调用：

| 时机                 | 在哪里                         | 说明                     |
| -------------------- | ------------------------------ | ------------------------ |
| 用户点了左侧某个会话 | `selectSession()` 里           | 切到会话 bbb，同步改 URL |
| 用户删除了当前会话   | `deleteSession()` 里           | 切到下一个，同步改 URL   |
| 后端返回新会话 UID   | `sendMessage()` 的 `onMeta` 里 | 新会话创建了，同步写 URL |

**大白话**：页面切会话时，主动把地址栏也改了。

---

> 注意：为什么不用 `watch(activeConversationUid)` 自动改 URL？
>
> 这是这个代码里**最容易被忽略的一个设计**。
>
> 他们本来可以这样写：
>
> ```javascript
> // ← 他们没这么写
> watch(activeConversationUid, (uid) => {
>   syncRoute(uid);
> });
> ```
>
> 但他们**没有**，而是选择了在 `selectSession`、`deleteSession` 等地方**手动调用** `syncRoute`。
>
> 原因是：
>
> `watch + router.replace` 的异步调度会打断 SSE 流的响应式链路
>
> 因为使用函数去进行路由变更，是很可控的，你想什么时候变更就什么时候变更，变更完去接收sse的消息，perfect，
>
> 但是如果用watch，watch里是不会立马变更的，而是等整个函数跑完才执行，相当于你接收完了全部的SSE才开始跳路由，那SSE和要跳的路由不在一个状态，这样就很不好。
>
> 所以：
>
> ```
> 在关键动作（选会话、删会话、创建会话）之后
> 立刻同步调用 syncRoute()
> 不走 watch，不走异步
> ```
>

### safeSend() 是什么

这是一个**细节但很重要**的函数：

```java
private void safeSend(SseEmitter emitter, String event, String data) {
    try {
        // 把 \n 替换成零宽空格，前端收到后再还原（把用户消息里的\n换成等宽空格）
        if (data != null) {
            data = data.replace("\n", "");
        }
        emitter.send(SseEmitter.event().name(event).data(data));
    } catch (Exception ignored) {
        // 客户端断开，忽略
    }
}
```

用户发送的消息会通过 saveSend 存库时替换`\n`，AI 生成的回复文本在通过 SSE 推送前，也会在 AiChatServiceImpl 里用类似逻辑替换`\n`，这样不管是用户输入的换行还是 AI 输出的换行，都不会干扰 SSE 协议的事件解析。

## 二、AgentLoopImpl.java — 大脑

它是**真正的 Agent 核心**，负责：

```
1. 意图识别   → 用户想干什么？匹配哪个 Skill？
2. LLM 调用   → 发给 AI，返回结果
3. 工具循环   → AI 调用工具 → 执行 → 返回结果 → 再调用
4. 流式输出   → 边思考边返回文字
```

## 三种输入模式的区别

| 模式             | 条件                           | 说明                                               |
| ---------------- | ------------------------------ | -------------------------------------------------- |
| **直达模式**     | `skillCode != null`            | 前端点了技能面板，直接跳过意图识别，用指定的 Skill |
| **意图识别模式** | `needsSkillRouting() == true`  | 第一轮对话，AI 判断用户要什么技能                  |
| **后续轮次**     | `needsSkillRouting() == false` | 对话已在进行中，复用之前的 Skill 上下文            |

## 三、ToolStreamCallback.java — 回调接口

## 这个文件干了什么？

它就是一个**接口**（interface），定义了回调方法：

```java
public interface AgentLoopStreamCallback {
    void onTextChunk(String chunk);        // 收到文字片段
    void onReasoningChunk(String chunk);   // 收到推理内容
    void onToolStart(String functionName, String arguments);  // 工具开始
    void onToolEnd(String functionName, String status, String resultSummary); // 工具结束
    void onInteraction(Interaction interaction);  // 需要用户交互
    void onComplete(AgentLoopResult result);    // 完成
    void onError(Throwable e);                  // 出错
}
```

---

## 它和前端 SSE 的对应关系

```
后端 AgentLoop                    前端 streamSSE
────────────────────────────────────────────────
callback.onTextChunk()    ←──→    onText()
callback.onToolStart()    ←──→    onToolStart()
callback.onToolEnd()      ←──→    onToolEnd()
callback.onComplete()     ←──→    onDone()
callback.onError()        ←──→    onError()
```

# 六、面试时可以这样回答

> "后端分三层。第一层是 AiChatServiceImpl，负责会话管理和 SSE 入口，收到前端消息后启动工作线程，实时推送 SSE 事件。第二层是 AgentLoopImpl，是真正的 Agent 大脑，它会先做意图识别匹配 Skill，然后进入循环：调用 LLM → 收到工具调用 → 执行工具 → 结果回填 → 继续调用，直到 LLM 返回纯文本。第三层是 ToolStreamCallback，作为回调接口把 Agent 的每一步（文字、工具开始/结束、完成）通知给 SSE 层推送出去。"
