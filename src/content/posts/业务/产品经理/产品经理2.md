## 介绍

我之前主要做前端开发，但是因为公司项目比较多，所以有不少项目，我都是去兼任项目经理，产品经理的角色，

比如说根据对方的需求我会直接出原型，然后对于交互什么的可以直接根据原型和对方去进行沟通，自己去设计交互，

然后到具体实施开发的时候，我会和对方一直沟通相应的时间，定期去开会议看当前的状况，如果遇到了一些困难会进行协调，比如说有两个公司，一个公司他数据库在他那边儿，另一个公司要根据那个数据库去出接口，这种情况下应该去督促有数据库的公司去出部分接口和数据库模型表，然后另一个公司针对它的接口进行加工。

对于流程上熟悉从需求、交互、页面实现到上线联调的流程。



产品经理和ai产品经理的区别，看业务情况，首先我觉得传统的开发技术选型可能架构师决定的比较多，但是ai的话由于很多产品比较新可能产品经理也要参与选型，毕竟ai产品经理本身就对ai有一定程度的了解，

再厉害一点的ai产品经理，会对模型不确定性，比如回答准确率、幻觉、数据权限、人工兜底、反馈闭环和效果评估。



产品经理的能力在我看来，在需求分析，在沟通能力，对于边界的把控，以及排期的能力。

### 【PRD】愿望单降价 App 推送功能 V1.0

> 我们假设要为一个**数字游戏分发平台**（类似 Steam 或 Epic）做一个“愿望单游戏降价 App 推送”功能。以下是这份 PRD 的核心部分摘录：

**1. 基础信息**

- **文档状态：** 评审中
- **产品经理：** [你的名字]
- **目标上线时间：** 2026-08-15

**2. 项目背景与目标**

- **背景：** 当前大量用户会将游戏加入“愿望单”，但在大促或史低（历史最低价）时，由于用户未主动打开 App，导致错失购买时机。
- **目标：**
  - 核心指标：通过及时的 App Push（推送通知），将“愿望单”转化为实际购买的**转化率提升 15%**。
  - 辅助指标：提升 App 日均打开率。

**3. 目标用户与使用场景**

- **用户故事：** 作为一名预算有限的玩家，我希望在我关注的游戏打折时立刻收到通知，这样我就能以最划算的价格买到它，而不需要每天手动去商城刷价格。

**4. 功能需求详情**

- **4.1 触发规则（后端逻辑）**
  - 当一款游戏的价格下降幅度 **≥ 10%** 时，触发降价通知逻辑。
  - 限制频率：针对同一款游戏，在同一个打折周期内，只向用户推送**一次**通知，避免过度打扰。
- **4.2 推送文案策略**

| **降价幅度**      | **推送标题**     | **推送正文 (变量替换)**                                      |
| ----------------- | ---------------- | ------------------------------------------------------------ |
| **10% - 49%**     | 愿望单降价提醒 📉 | 你关注的《{游戏名称}》正在打折，当前价格 {现价} (原价 {原价})，快来看看吧！ |
| **≥ 50% 或 史低** | 🚨 史低折扣预警！ | 骨折价！《{游戏名称}》迎来了历史最低价 {现价}，直降 {折扣比例}，立刻入手！ |

- **4.3 交互与异常处理（前端逻辑）**
  - **正常流转：** 用户点击推送通知 -> 唤起 App -> 直接跳转至该游戏的商品详情页。
  - **异常处理：** 如果用户点击推送时，游戏打折已经结束（价格已恢复正常），则在商品详情页顶部弹出 Toast 提示：“该商品的促销活动已结束”。

**5. 非功能需求**

- **推送性能：** 大促开启（如夏季特卖）的前 30 分钟内，服务器需支持向百万级用户并发发送推送，延迟不得超过 5 分钟。

**6. 数据埋点需求**

- 记录每条降价 Push 的**送达率**、**点击率**。
- 记录从 Push 唤醒 App 到最终完成支付的**订单转化率**。

> **核心洞察：** 写 PRD 的精髓在于**“穷举边界”**。初级产品经理通常只写“用户点了这里会发生什么（Happy Path）”，而优秀的产品经理会把大量精力花在定义异常情况上——“如果没网呢？”“如果用户狂点两次呢？”“如果这是历史最低价呢？”。



## 针对ai的不可预测性

可以通过 “约束框架” 来把控，比如在 prompt 里明确 “禁止调用付费工具”“优先选择官方数据源”，给 Agent 划清行动边界。

也可以用 “人类审核节点”（HIL），比如关键步骤让用户确认后再执行，像大额转账前弹窗提示。（Human in loop）

另外，通过 “历史案例库” 让 Agent 学习 “正确路径”，比如之前成功完成的任务流程，减少离谱选择。



## Agent run 的流程

Agent run：规划西藏旅行

### Loop 1：理解需求

Agent 先判断用户意图：

```
用户想去西藏旅行
需要规划路线、时间、预算、交通、注意事项
```

它可能发现缺少信息：

```
不知道用户从哪里出发
不知道旅行天数
不知道预算
不知道是否第一次去高海拔地区
```

这时候 Agent 可以选择：

```
直接先给通用方案
```

或者：

```
先问用户几个关键问题
```

------

### Loop 2：检索目的地信息

如果系统有搜索工具，Agent 可能调用工具查：

```
西藏当前旅游政策
景点开放情况
天气
交通情况
机票或火车信息
```

这个 loop 是：

```
思考：我需要最新信息
行动：调用搜索工具
观察：拿到搜索结果
更新上下文
```

------

### Loop 3：规划路线

有了基础信息后，Agent 开始生成路线，比如：

```
拉萨 2 天
林芝 2 天
羊湖 1 天
纳木错 1 天
```

但它可能又发现：

```
海拔适应需要安排缓冲
第一天不能安排太累
路线不能太赶
```

于是它调整计划。

------

### Loop 4：检查约束

Agent 可能继续判断：

```
这个路线是否适合第一次去西藏的人？
预算是否合理？
交通衔接是否现实？
有没有高反风险？
```

如果发现问题，就继续修改。

------

### Loop 5：最终输出

最后 Agent 觉得信息足够了，生成最终回答：

```
7 天西藏旅行方案
每日行程
交通建议
预算范围
注意事项
高反建议
```

这时 run 结束。

------

### 用图表示

```
用户：我想去西藏旅行
        ↓
Agent Run 开始
        ↓
Loop 1：理解用户意图
        ↓
Loop 2：判断缺少哪些信息
        ↓
Loop 3：调用工具查目的地/交通/天气/政策
        ↓
Loop 4：根据结果规划路线
        ↓
Loop 5：检查路线是否合理
        ↓
Loop 6：生成最终旅行方案
        ↓
Agent Run 结束
```

所以：

> **run 是外层任务。loop 是内部循环。**

## 多agent处理

**第一，**先判断任务是否真的需要 multi-agent。如果只是简单问答，用单 Agent 加 一些内容输入就够了；

只有当任务涉及多个专业步骤、多轮工具调用、交叉检查时，才考虑 multi-agent。

**第二，**定义agent分工。每个Agent要有清晰的职责边界，比如检索、规划、审核、执行...

**第三，**设计协作方式。通常会有一个主Agent或 Orchestrator，负责拆解任务、调用子 Agent、汇总结果，并判断是否需要重新执行。

**第四，**设计质量控制。比如用结构化输出、置信度判断、结果冲突处理、人工确认节点、失败兜底，以及成本和延迟控制。

## LangChain

 普通的 API 调用是无状态的、单次的。但在实际业务中，我们需要大模型具备**记忆（Memory）、外接知识（RAG）、以及执行动作（Tools）**的能力。LangChain 的核心价值在于它提供了一套**标准化的组件接口（如 LCEL - LangChain 表达式语言）**。它将应用逻辑与底层的模型提供商解耦，无论底层是 GPT-4 还是开源的 Llama，上层应用层的 Prompt、工具链和记忆模块都不需要大改。

> 无状态：每次 API 调用都是独立的，服务器不会记住你上一次请求了什么。比如你调用一次获取用户信息，再调用一次修改密码，服务器不会因为你之前获取过信息就 “认识” 你，每次都需要重新验证身份、传递必要参数。

LangGraph 是 LangChain 生态里更底层的 Agent **编排**框架，用**有向图**结构控制 Agent 的执行流程。

普通 API 调用主要是一次请求一次响应。LangChain 更偏应用层封装，它能把模型、工具、Prompt、RAG、memory、streaming、structured output 等组合成一个完整应用流程。

```
Input
  ↓
Messages / Prompt
  ↓
Model
  ↓
Tool / Retriever / Memory / Structured Output
  ↓
Agent Loop / LangGraph
  ↓
Output / Streaming / Trace
```



### **1. Input -> Messages / Prompt**

- **深度拓展：** 别只提 Prompt，提一提 **动态上下文管理**。
- **话术：** “输入层不仅仅是用户的 Query，LangChain 的 PromptTemplate 允许我们动态注入变量（比如 RAG 检索回来的文档、用户的历史对话等），将其组合成结构化的 Messages 发给模型。”

### **2. Model -> Tool / Retriever / Memory / Structured Output**

- **深度拓展：** 这里是展现“大模型作为大脑”的核心。
- **话术：** * **Memory：** “由于 API 是无状态的，LangChain 提供了机制（如 Window Memory 或 基于数据库的记忆池）在每次请求前自动拼接历史记录。”
  - **RAG (Retriever)：** “解决模型幻觉和知识库滞后问题。本质是`查询 -> 向量数据库检索 -> 组装上下文 -> 模型生成`的流水线。”
  - **Structured Output：** “在业务落地中，我们往往需要 JSON 格式的输出以便下游系统解析。LangChain 封装了输出解析器（Output Parsers），甚至会在模型输出格式错误时自动触发重试（Retry）机制。”
  - **Tool (工具)：** “赋予模型‘手和脚’。比如让模型能搜索网页、查 SQL 数据库。模型通过判断是否需要调用工具，返回特定的指令，这就是 Agent 的雏形。”

### **3. Agent Loop / LangGraph (🌟 面试的绝对亮点)**

这是你拉开与其他候选人差距的关键部分。你需要讲清楚 **LangChain Agent** 和 **LangGraph** 的区别。

- **深度拓展：** 为什么有了 LangChain 还要出 LangGraph？
- **话术：** “早期的 LangChain Agent（比如 AgentExecutor）本质上是一个黑盒的 `While` 循环（基于 ReAct 框架：推理 -> 行动 -> 观察）。它会一直跑，直到得出结果。但在生产环境中，**这种黑盒循环极度不可控，很难调试，也容易死循环。**”
- “这就是 **LangGraph** 诞生的原因。LangGraph 把 Agent 的执行流程变成了**有向图（Directed Graph）**。
  - 它引入了 **State（状态）**：图的每个节点都会更新这个全局状态，方便我们在任何步骤追踪数据。
  - 它支持 **Cycles（循环）** 和 **条件边（Conditional Edges）**：我们可以精细定义模型什么时候该调用工具，什么时候该返回给用户。
  - 它支持 **Human-in-the-loop（人类在环）**：比如在执行敏感操作（如发邮件、扣款）前，图可以暂停，等人类确认后再继续。这让多 Agent 协同和复杂业务流变得可控和工程化。”

### **4. Output / Streaming / Trace**

- **深度拓展：** 展现你对用户体验和运维的关注。
- **话术：** * **Streaming（流式）：** “为了减少用户等待的首字响应时间（TTFB），应用层必须支持流式输出。LangChain 对流式做了很好的封装。”
  - **Trace（可观测性）：** “提到 Trace 就不得不提生态里的 **LangSmith**。在生产上线后，我们需要监控每一次 Agent 执行的延迟、Token 消耗、以及每一步 Tool 调用的输入输出。可观测性是保障 AI 应用稳定性的基础设施。”



```
Graph：整个流程图
  ↓
Node：图里的一个步骤
  ↓
State：节点之间共享的数据
  ↓
Loop：节点之间反复执行形成的循环
  ↓
Checkpoint：某一步执行后的 state 存档
```
