# SSO 单点登录：从零理解多系统统一登录

这篇文章只解决一个问题：

> **多个系统怎么共用一个登录入口，让用户登录一次就能访问多个系统？**

所以这里重点讲 SSO 的角色、流程和常见协议。JWT 只是 SSO 可以使用的一种凭证格式，具体 JWT 结构看《Cookie、Session、Token、JWT 核心知识总结》，登录态安全配置看《登录态安全与接口防护》。

## 先说大白话：SSO 是什么？

**SSO = Single Sign-On = 单点登录**

它的核心作用就一句话：

> 用户只登录一次，就可以访问多个相关系统，不用每个系统都重新登录。

举个例子：

你们公司有这些系统：
- OA 办公系统
- 财务系统
- 人事管理系统
- 项目管理系统

如果没有 SSO，你每天上班要登录 4 次。

有了 SSO，你只需要登录 1 次，后面的系统都自动"认识"你了。

---


## SSO 和 JWT 是什么关系？

回头看你的 JWT 笔记，SSO 和 JWT 的关系就很清楚了：

1. **SSO 是一种架构设计**：解决多个系统怎么共享登录状态
2. **JWT 是 SSO 中的凭证格式之一**：登录中心签发 JWT，业务系统验证 JWT

```
SSO 架构
    ↓
用什么做凭证？
    → 可以用 JWT
    → 可以用 Session ID
    → 可以用 Ticket
    → 可以用 OIDC 的 id_token
```

如果你用 JWT 作为 SSO 的凭证，那么：

- 登录中心负责签发 JWT（用登录中心的密钥签名）
- 业务系统负责验证 JWT（用登录中心的公钥验签）
- JWT 的 Payload 里包含用户身份信息

这样就实现了一个基于 JWT 的简单 SSO 系统。

---

## 普通登录 vs SSO 登录

### 普通登录（每个系统独立登录）

```
用户访问系统 A
    ↓
系统 A 让用户输入账号密码
    ↓
系统 A 验证账号密码
    ↓
系统 A 给用户发一个凭证（Session ID 或 JWT）
    ↓
用户后续访问系统 A，都带着这个凭证
```

这个凭证只在**系统 A**里有效。

如果用户再访问系统 B，系统 B 不认识系统 A 的登录状态，所以又要重新登录。

### SSO 登录（统一登录中心）

```
用户访问系统 A
    ↓
系统 A 发现用户没登录
    ↓
系统 A 把用户跳转到"统一登录中心"
    ↓
用户在登录中心输入账号密码
    ↓
登录中心验证成功，返回一个凭证给系统 A
    ↓
系统 A 校验凭证，用户进入系统 A
```

之后用户再访问系统 B：

```
用户访问系统 B
    ↓
系统 B 发现用户没登录
    ↓
系统 B 把用户跳转到"统一登录中心"
    ↓
登录中心发现：这个人刚才已经登录过了
    ↓
不需要再输入密码，直接返回凭证给系统 B
    ↓
系统 B 校验凭证，用户进入系统 B
```

用户感觉就是：只登录了一次，其他系统都自动进去了。

---

## SSO 的三个核心角色

### 1. 用户

使用系统的人

### 2. 业务系统

比如 OA、财务、人事这些具体的业务系统。

它们**不直接处理登录**，而是把登录这件事委托给登录中心。

### 3. 认证中心（SSO Server）

也叫：
- 统一认证中心
- 登录中心
- SSO Server
- IAM（Identity and Access Management）

它的职责：
- 处理用户登录
- 验证账号密码
- 发放身份凭证（Token/Ticket）
- 校验凭证有效性
- 管理用户身份和权限

---

## SSO 的核心流程（结合 JWT）

假设用 JWT 作为 SSO 的凭证格式，整个流程是这样的：

### 第一次登录（访问系统 A）

```
1. 用户访问系统 A

2. 系统 A 检查：有没有有效的登录凭证？
   → 没有（用户第一次访问）

3. 系统 A 跳转到统一登录中心，并告诉登录中心：
   "用户从系统 A 来的，登录成功后把用户带回来"

4. 统一登录中心显示登录页面

5. 用户输入账号密码

6. 登录中心验证成功，生成 JWT：
   - Payload 里包含：用户 ID、用户名、角色等
   - 用登录中心的密钥签名

7. 登录中心把 JWT 附加在 URL 参数或 Cookie 里，
   把用户带回系统 A

8. 系统 A 拿到 JWT，用登录中心的公钥验证签名：
   - 验证签名是否正确
   - 验证是否过期
   - 读取 Payload 里的用户信息

9. 验证通过，系统 A 建立自己的会话（或者也用这个 JWT）
   → 用户进入系统 A
```

### 第二次登录（访问系统 B）

```
1. 用户访问系统 B

2. 系统 B 检查：有没有有效的登录凭证？
   → 没有（系统 B 之前没见过这个用户）

3. 系统 B 跳转到统一登录中心

4. 登录中心检查：用户之前登录过吗？
   → 登录过（登录中心还保留着这个用户的登录状态）

5. 登录中心不需要显示登录页面，直接生成 JWT

6. 登录中心把 JWT 带回系统 B

7. 系统 B 验证 JWT，进入系统 B
   → 用户感觉：没有再登录，但进来了
```

### 关键点

为什么系统 B 能"认出"用户？

因为：
- 登录中心在第一次登录时，可能在用户的浏览器里种了一个 Cookie
- 或者在 URL 参数里传递了一个临时凭证
- 系统 B 把用户带到登录中心时，登录中心看到这个 Cookie/凭证，就知道"这个用户之前登录过"

## SSO 的常见实现方案

你大概知道这些名字就够了。

### 方案一：Session + Cookie（同域 SSO）

**适用场景**：多个系统在同一域名下，或者子域名

```
a.company.com      → 系统 A
b.company.com      → 系统 B
login.company.com  → 登录中心
```

登录中心登录后，在顶级域名 `company.com` 下写 Cookie。

因为 Cookie 的 `Domain` 可以设置成 `.company.com`，浏览器访问 `a.company.com` 和 `b.company.com` 时，都可能自动带上这个 Cookie。

**优点**：简单，浏览器原生支持

**缺点**：跨域困难，移动端、第三方系统接入麻烦

---

### 方案二：CAS（Central Authentication Service）

CAS 是一个**专门为 SSO 设计的协议**， Yale 大学开发的，开源项目。

**历史背景**：出现比较早（2001年），所以很多老系统（学校、政府、医院内部系统）还在用。

**专业术语**：

| 术语 | 含义 |
|-----|-----|
| TGT（Ticket Granting Ticket） | 用户的"总通行证"，登录成功后颁发，存 Cookie |
| ST（Service Ticket） | 访问具体业务系统时用的"一次性门票" |
| CAS Server | 登录中心服务器 |
| CAS Client | 接入 CAS 的业务系统 |

**为什么 Ticket 要设计成一次性的？**

```
1. 用户登录成功 → CAS 给浏览器发一个 TGT（存 Cookie）
        ↓
2. 用户访问系统 A → 浏览器带着 TGT 去 CAS
        ↓
3. CAS 生成一个 ST 返回给浏览器
        ↓
4. 浏览器把 ST 发给系统 A
        ↓
5. 系统 A 拿 ST 去 CAS 验证
        ↓
6. 验证成功，系统 A 让用户进入
        ↓
7. 这个 ST 立刻作废
        ↓
8. 用户再访问系统 A？要重新申请 ST
```

> 既然有了总通行证为什么还要一次性门票？
>
> - TGT 存在 Cookie 里，浏览器会自动携带，会被 CSRF 攻击利用。
> - 职责分离：TGT 的作用是"证明你已经登录"，ST 的作用是"证明你想访问这个特定的系统"。

**优缺点**：

| | 说明 |
|--|-----|
| 优点 | 协议成熟、安全性好、单点登出支持好 |
| 缺点 | 比较老，学习成本高，第三方接入麻烦 |

**适合场景**：内部系统、师生系统、医院系统等老牌项目

---

### 方案三：OAuth2 + OpenID Connect（OIDC）

这是**现在最流行**的方案。

---

#### OAuth2 和 OIDC 关系

| | OAuth2 | OIDC |
|--|--------|------|
| 全称 | Open Authorization 2.0 | OpenID Connect |
| 主要解决 | **授权**：我能用你的哪些数据？ | **认证**：你是谁？ |
| 问的问题 | "允许第三方读取我的哪些信息？" | "这个人是不是 xxx？" |
| 典型场景 | APP 授权获取微信头像 | 用微信登录网站 |

**大白话理解**：

```
OAuth2：你授权微信，告诉第三方"允许他们用我的头像和昵称"
         → 关注的是"允许访问哪些数据"

OIDC：你用微信登录网站
         → 关注的是"证明你是微信用户 xxx"
```



#### **授权码模式（最安全，适合后端）**

OAuth2 是一套标准，规定了"怎么获取授权"，有 4 种方式，这里只介绍最常用的一种方式。

**适用场景**：有后端服务器的应用（Web 网站）

```
1. 用户点击"用微信登录"
        ↓
2. 网站跳转到微信授权页面
        ↓
3. 用户同意授权
        ↓
4. 微信跳转回网站，带一个"授权码"（code）
        ↓
5. 网站后端拿 code，换取 token（这是两个服务端之间的请求，中间会带客户端密钥）
        ↓
6. 如果返回的是 id_token（通常是 JWT），网站后端会用公钥验签
        ↓
7. 验签通过，解析用户身份信息
        ↓
8. 允许登录
```

**为什么安全？**：code 只能用在后端，前端拿不到，access_token 也不会经过浏览器

---

#### OIDC 的核心概念

OIDC 在 OAuth2 基础上加了"认证"，会返回一个 **id_token**。

**OIDC 返回的三个 Token**

| Token | 作用 |
|-------|------|
| access_token | 访问资源（比如获取用户头像） |
| id_token | 证明用户身份（通常是 JWT） |
| refresh_token | 刷新 access_token |

**其中 id_token 就是 JWT**，里面包含用户身份信息。

#### OIDC 的登录流程

```
1. 用户点击"用 GitHub 登录"
        ↓
2. 网站跳转到 GitHub 授权页面
        ↓
3. 用户同意授权
        ↓
4. GitHub 跳转回网站，带 code
        ↓
5. 网站后端拿 code 换 token
        ↓
6. GitHub 返回 id_token（JWT）
        ↓
7. 网站验证 JWT，读取用户信息，允许登录
```

---

#### 适合场景

| 场景 | 推荐 |
|-----|------|
| 第三方登录（微信/GitHub/Google） | OAuth2 + OIDC |
| 企业内部多系统 SSO | OIDC（企业微信/飞书/Okta） |
| 微服务间认证 | OAuth2（客户端凭证模式） |

---

#### 一句话总结

```
OAuth2：解决"我能访问哪些数据"（授权）
OIDC：解决"这个用户是谁"（认证）
OIDC = OAuth2 + 身份认证
```

**id_token = JWT = 证明用户身份的令牌**

## SSO 和单点登出

有单点登录，也要有单点登出。

**单点登出**：用户在一个地方退出，所有接入 SSO 的系统都应该退出。

---

### 方案一：会话列表方案（CAS 的做法）

登录时，登录中心记录用户访问过哪些系统。退出时，一个个通知清除。

```
用户从系统 A 点退出
        ↓
系统 A 通知登录中心退出
        ↓
登录中心查表，找到该用户访问过的系统：系统 B、系统 C
        ↓
登录中心依次通知系统 B、系统 C："该用户退出登录了"
        ↓
系统 B、系统 C 清除本地会话
        ↓
用户再访问任何一个系统，都要重新登录
```

**优点**：能精准清除所有会话
**缺点**：实现复杂，每个系统都要接监听器

---

### 方案二：前端清除 + 后端过期（更常见）

**前端**：退出时清除所有系统的 Cookie 或 localStorage

**后端**：Token 设置短过期时间，用户不活跃就自动失效

```
用户点击退出
        ↓
前端清除：Cookie、localStorage、所有 Token
        ↓
后端：Token 本身还没过期，但用户已经拿不出有效的 Token 了
        ↓
再访问任何系统，都需要重新登录
```

**适用场景**：JWT 场景下用得最多

---

### 方案三：Token 黑名单机制

把失效的 Token 加入黑名单，后端检查 Token 时先查黑名单。

```
用户退出
        ↓
登录中心把该 Token 加入黑名单
        ↓
用户带着 Token 访问系统 A
        ↓
系统 A 问登录中心："这个 Token 还能用吗？"
        ↓
登录中心查黑名单，发现已失效
        ↓
拒绝访问，要求重新登录
```

**优点**：实时生效
**缺点**：每次请求都要查黑名单，有性能开销



## 面试时可以这样回答

**完整版**：

> 单点登录就是用户只需要在统一认证中心登录一次，就可以访问多个接入该认证中心的业务系统。业务系统本身不直接保存用户密码，而是把登录认证交给统一登录中心。业务系统间通过 Ticket 或 Token 实现身份互认，常见实现方式包括 Session + Cookie 同域方案、CAS 协议、以及现在更流行的 OAuth2/OIDC 方案。

**简短版**：

> SSO 解决的是多系统统一登录问题，认证中心统一验证用户身份，验证通过后签发凭证（如 JWT），各个业务系统校验这个凭证即可建立登录状态。
