# 移动端 H5 与性能优化面试复习

这篇主要面向移动端 Web / H5 / 小程序岗位面试。重点不是背 API，而是能讲清楚：**我如何把一个移动端页面从需求、开发、联调、适配、优化做到上线维护**。

## 一、岗位真正想考什么

移动端前端岗位通常会围绕下面几类能力提问：

| 能力 | 面试官想确认什么 |
| --- | --- |
| 页面还原 | 是否能根据 UI 稿稳定还原页面，处理不同屏幕、不同 DPR、不同浏览器差异 |
| 移动端适配 | 是否熟悉 rem、vw、Flex、Grid、响应式、安全区、1px 边框等方案 |
| 框架开发 | 是否能用 Vue / React 完成路由、状态、组件、接口、表单、列表等业务开发 |
| 性能优化 | 是否知道首屏加载、资源压缩、图片优化、缓存、懒加载、长列表优化 |
| 兼容问题 | 是否踩过 iOS、Android、微信内置浏览器、软键盘、滚动、fixed 等坑 |
| H5 与原生交互 | 是否理解 JSBridge、环境判断、协议约定、回调和异常兜底 |
| 工程化 | 是否会 Vite / Webpack、Git、组件化、环境配置、打包部署 |
| 协作交付 | 是否能和产品、UI、后端拆需求、对接口、联调、排期、上线 |

一句话概括：

> 这个岗位要的是能独立负责移动端核心模块的人，不只是会写页面，还要能处理适配、兼容、性能和联调问题。

## 二、移动端适配

### 1. 常见适配方案

#### rem 适配

rem 相对于根元素 `html` 的 `font-size`。常见做法是根据设计稿宽度动态设置根字号。

```js
function setRootFontSize() {
  const designWidth = 375
  const baseSize = 16
  const width = document.documentElement.clientWidth
  document.documentElement.style.fontSize = `${(width / designWidth) * baseSize}px`
}

setRootFontSize()
window.addEventListener('resize', setRootFontSize)
```

优点：

- 适合按比例还原移动端设计稿。
- 和 `postcss-pxtorem` 配合后，开发时仍然可以写 px。

缺点：

- 根字号变化会影响所有 rem 单位。
- 对平板、大屏需要额外限制最大宽度。

#### vw / vh 适配

`vw` 表示视口宽度的 1%。例如 375px 设计稿中，`75px` 可以换算为 `20vw`。

优点：

- 不依赖 JS 动态设置根字号。
- 和 `postcss-px-to-viewport` 配合使用比较方便。

缺点：

- 纯等比缩放在大屏和平板上可能显得过大。
- `100vh` 在移动端浏览器里可能受到地址栏、底部工具栏影响。

#### Flex / Grid 布局

移动端最常用的是 Flex：

```css
.toolbar {
  display: flex;
  align-items: center;
  justify-content: space-between;
}
```

Grid 适合九宫格、商品卡片、宫格菜单：

```css
.menu {
  display: grid;
  grid-template-columns: repeat(4, 1fr);
  gap: 12px;
}
```



### 2. 1px 边框问题（DPR）

高 DPR 屏幕下，CSS 的 `1px` 可能看起来偏粗。常见方案是用伪元素加缩放：

```css
.hairline {
  position: relative; /* 作为伪元素的定位父容器 */
}

.hairline::after {
  content: '';       /* 生成一个看不见的虚拟元素 */
  position: absolute;
  left: 0;
  right: 0;
  bottom: 0;         /* 贴在父容器底部，充当底部分割线 */
  height: 1px;       /* CSS像素高度1px */
  background: #ddd;

  transform: scaleY(0.5);       /* 纵向压缩到原来的一半 */
  transform-origin: bottom;     /* 缩放基准点：底边不动，向上压缩 */
}
```

> DPR（Device Pixel Ratio，设备像素比）
>
> DPR = 物理像素 ÷ CSS（逻辑）像素
>
> 高 DPR = 高清屏

DPR=2：系统用2 行物理像素渲染，DPR=3：系统用3 行物理像素渲染

### 3. iOS 安全区

全面屏 iPhone 底部有安全区，底部按钮、TabBar 要避免贴边。（边距多做一点）

```css
.bottom-bar {
  padding-bottom: calc(12px + env(safe-area-inset-bottom));
}
```

## 三、移动端兼容性问题

### 1. iOS fixed 定位异常

常见场景：

- 点击输入框唤起软键盘 → fixed 底部按钮被键盘顶上去、飘起来、布局乱掉
- 关闭键盘（blur 失焦）→ 页面滚动条停留在上方，不会自动回弹，页面留白 / 错位

根本原因：

移动端浏览器软键盘弹出时视口、滚动高度计算混乱，`fixed` 定位在键盘弹出时行为异常，浏览器自带的滚动恢复机制经常失效。

处理思路：

在键盘弹出的瞬间可以临时隐藏底部 fixed 按钮

- 手指在弹窗内部正常滚动

  如果弹窗已经滚到顶部还继续上拉 / 滚到底部继续下拉 → `e.preventDefault()`，阻止事件冒泡到底层 body，防止底层页面跟着滚。
- 输入框 blur 后手动触发滚动修正

```js
input.addEventListener('blur', () => {
  setTimeout(() => {
    window.scrollTo(0, window.scrollY)
  }, 0)
})
```

### 2. 滚动穿透

弹窗打开后，滑动弹窗内容时，底层页面也跟着滚。

常见解决方案：

```css
body.modal-open {
  overflow: hidden;
  position: fixed;
  width: 100%;
}
```

更稳妥的做法是打开弹窗时记录滚动位置，关闭时恢复：

```js
let scrollTop = 0

function lockBody() {
  scrollTop = window.scrollY
  document.body.style.position = 'fixed'
  document.body.style.top = `-${scrollTop}px`
  document.body.style.width = '100%'
}

function unlockBody() {
  document.body.style.position = ''
  document.body.style.top = ''
  document.body.style.width = ''
  window.scrollTo(0, scrollTop)
}
```

还可以用上个问题的“阻止事件冒泡到底层 body，防止底层页面跟着滚。”

### 3. 点击延迟与 touch 事件

早期移动浏览器为了判断双击缩放，会有 300ms 点击延迟。现代浏览器大多已经优化，但仍然要注意：

- 按钮点击区域不能太小。
- 避免在 `touchstart` 和 `click` 上重复绑定导致触发两次。
- 手势类交互要区分点击、滑动、长按。

### 4. 软键盘遮挡输入框

移动端点击底部输入框，软键盘弹出来，直接盖住输入框，用户看不见正在输入的内容。

常见处理：

- 输入框聚焦时滚动到可视区域。

```js
input.addEventListener('focus', () => {
  setTimeout(() => {
    input.scrollIntoView({ block: 'center' })
  }, 300)
})
```

> 这个是说 在输入框被点击时触发（focus）
>
> 而且不是一下子触发，是等了300ms，这也是等待键盘弹起
>
> 触发的内容是 滚动到视口区域

### 5. 微信内置浏览器问题

场景：H5 在**微信内置浏览器**打开（微信里点开网页），会出现一堆别的浏览器没有的专属 bug。

常见问题：

- 微信分享需要 JS-SDK 签名配置。
- iOS 和 Android 微信对返回栈、视频播放、文件上传支持不同。
- 页面缓存导致返回后状态没有刷新。
- H5 支付、授权登录需要严格匹配域名和回调地址。

面试表达：

> 微信环境我一般会先做环境判断，判断是否在微信环境，涉及分享、支付、授权这类能力时，会把 SDK 初始化、签名接口、错误回调都封装起来。页面返回状态问题会结合路由钩子（监听页面路由变化重新请求数据）、页面可见性事件（页面切回来时刷新数据）或业务缓存策略（手动控制页面数据缓存，返回时主动清除旧缓存）处理。

1.JS-SDK：调用微信自带的功能

普通网页在浏览器里打开，只能干网页自己的事；但在微信里打开，网页想调用微信本身的功能（比如分享、扫码、拍照、定位），就必须用 JS-SDK。需要在**微信公众号后台**里配置，配完才能用分享、扫码这些能力。

2.微信网页授权：获取用户身份（一键登录）

也在**微信公众号后台**配置，和 JS-SDK 是同一个后台，但属于另一项设置，不是配了 JS-SDK 就自动能用授权。

3.H5 微信支付：拉起微信付款，在**微信商户平台**单独配置。

## 四、H5 性能优化

移动端性能优化可以按加载链路拆：

```text
DNS/TCP/TLS -> HTML -> CSS/JS -> 数据接口 -> 图片资源 -> 渲染 -> 交互响应
```

### 1. 首屏加载优化

常见手段：

- 路由懒加载。
- 组件异步加载。
- 第三方库按需引入。
- 首页接口合并或并发请求。
- 骨架屏、loading、缓存上次数据。
- 静态资源走 CDN。
- 开启 gzip / brotli 压缩。

Vue 路由懒加载：

```js
const routes = [
  {
    path: '/order',
    component: () => import('@/pages/order/index.vue'),
  },
]
```

React 路由懒加载：

```jsx
const OrderPage = React.lazy(() => import('./pages/OrderPage'))
```

### 2. 图片优化

图片通常是移动端页面最大的资源。

优化方式：

- 使用 WebP / AVIF。
- 根据容器尺寸输出合适大小，避免大图缩小显示。
- 图片懒加载。
- 列表图使用缩略图。
- 首屏关键图片可以预加载。
- 非关键图片延后加载。

```html
<img src="/cover.webp" loading="lazy" alt="封面" />
```



### 3. 缓存策略

静态资源一般使用强缓存：

```text
Cache-Control: max-age=31536000, immutable
```

前提是文件名带 hash：

```text
app.8f3a2c.js
style.91bd12.css
```

HTML 一般不做长期强缓存，避免发版后用户拿到旧入口文件。

```text
Cache-Control: no-cache
```

接口数据缓存要看业务：

- 用户信息、订单状态：实时性强，谨慎缓存。
- 字典、城市列表、配置项：可以缓存。
- 首页推荐、静态内容：可以短时间缓存。

### 4. 渲染性能优化

常见问题：

- DOM 太多。
- 频繁读写布局属性导致强制同步布局。
- 大列表一次性渲染。
- 滚动事件频繁触发。
- 动画使用影响布局的属性。

优化方式：

- 长列表使用虚拟滚动。
- 滚动监听加节流。
- 动画优先使用 `transform` 和 `opacity`。
- 避免频繁修改 `top/left/width/height`。
- 合理使用 `requestAnimationFrame`。

```js
let ticking = false

window.addEventListener('scroll', () => {
  if (ticking) return

  ticking = true
  requestAnimationFrame(() => {
    // 更新吸顶、曝光、懒加载等逻辑
    ticking = false
  })
})
```

### 5. 长列表优化

如果移动端列表有几百上千条，不能一次性全部渲染。

解决方案：

- 分页加载。
- 上拉加载更多。
- 虚拟列表，只渲染可视区域附近的数据。
- 图片懒加载。
- 列表项组件减少无意义响应式依赖。

面试表达：

> 长列表我会优先做分页和懒加载，如果单页数据量仍然很大，就会用虚拟列表，只保留可视区域和缓冲区的 DOM。滚动相关逻辑会用节流或 requestAnimationFrame，避免每次 scroll 都触发重计算。

## 五、JSBridge

JSBridge 是 H5 和原生 App 通信的一层协议。

连接「网页 JS」和「手机原生 APP」的翻译桥

**微信 JS-SDK = 微信官方封装好的 JSBridge 成品**。

常见使用场景：

- 获取用户登录态。
- 调起原生分享。
- 调起支付。
- 打开原生页面。
- 获取设备信息。
- 调用相机、定位、扫码等原生能力。

### 1. 通信方式

常见方式：

```text
H5 调用 window.NativeBridge.method(params)
H5 通过 URL Scheme / iframe 通知原生
原生注入 JS 方法给 H5 调用
原生通过 evaluateJavaScript 回调 H5
```

### 2. 封装思路

```js
function callBridge(method, params = {}) {
  return new Promise((resolve, reject) => {
    if (!window.NativeBridge || !window.NativeBridge[method]) {
      reject(new Error(`Bridge method not found: ${method}`))
      return
    }

    const callbackName = `__bridge_cb_${Date.now()}_${Math.random().toString(16).slice(2)}`

    window[callbackName] = (result) => {
      resolve(result)
      delete window[callbackName]
    }

    window.NativeBridge[method]({
      ...params,
      callback: callbackName,
    })
  })
}
```

### 3. 面试怎么讲

> JSBridge 的核心是 H5 和原生约定一套协议，包括方法名、参数格式、回调方式和错误码。我在业务里会先封装统一的 bridge 调用方法，里面做环境判断、方法是否存在判断、超时处理和降级逻辑。这样页面里调用分享、登录、支付时，不需要关心底层是 iOS、Android 还是普通浏览器。
>
> 封装里主要做这 4 件事：
>
> 1. 环境判断
>
>    先检测当前页面是在哪运行的：
>
>    是在自家 APP 里吗？是 iOS 还是 Android？
>
>    还是在普通浏览器、微信里？
>
>    不同环境调用 Bridge 的写法不一样，封装里自动识别，业务代码不用关心。
>
> 2. **方法是否存在判断**
>
>    比如 APP 老版本没有「分享到朋友圈」这个功能，直接调用就会报错。
>
>    封装里会先检查一下当前 APP 支不支持这个方法，不支持就不强行调用，避免页面白屏报错。
>
> 3. **超时处理**
>
>    万一原生那边卡住了、没反应，不能让页面一直傻等。
>
>    封装里会设一个超时时间（比如 3 秒），超过时间还没收到原生的回复，就自动判定为失败，继续往下走，不会让页面卡死。
>
> 4. **降级逻辑**
>
>    如果当前不在 APP 里（比如普通浏览器打开），用不了原生能力，就给一个备用方案。
>
>    比如分享功能：
>
>    在 APP 里：直接调起原生分享面板
>
>    在普通浏览器里：弹出提示「请点击浏览器右上角菜单进行分享」
>
>    保证功能不会直接失效，体验更友好

## 六、小程序开发关注点

小程序和普通 H5 的差异：

| 对比项 | H5 | 小程序 |
| --- | --- | --- |
| 运行环境 | 浏览器 / WebView | 小程序容器 |
| DOM 操作 | 可以直接操作 DOM | 不能直接操作 DOM |
| 路由 | 前端路由控制 | 小程序页面栈 |
| 能力调用 | Web API / JSBridge | 小程序 API |
| 样式单位 | px / rem / vw | rpx / px |
| 包体限制 | 主要看加载性能 | 有主包、分包体积限制 |

小程序常见优化：

- 分包加载，减少主包体积。
- 图片放 CDN。
- 首屏接口精简。
- 避免频繁 `setData`。
- 长列表分页或虚拟列表。
- 公共组件抽象，减少重复代码。

面试表达：

> 小程序里我比较注意包体积和 setData 频率。主包只放首屏和公共能力，业务页面尽量拆到分包。列表页会分页加载，避免一次性 setData 大数组，图片资源尽量走 CDN。

## 七、框架与工程化

### 1. 请求封装

移动端请求封装通常要包含：

- baseURL。
- token 注入。
- loading 控制。
- 错误提示。
- 登录过期处理。
- 重复请求控制。
- 超时处理。

```js
import axios from 'axios'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000,
})

request.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

request.interceptors.response.use(
  (response) => response.data,
  (error) => {
    if (error.response?.status === 401) {
      // 跳转登录或刷新 token
    }
    return Promise.reject(error)
  },
)

export default request
```

### 2. 组件化思路

移动端项目常见公共组件：

- 顶部导航栏。
- 底部操作栏。
- 弹窗。
- 表单项。
- 商品卡片 / 订单卡片。
- 空状态。
- loading 状态。
- 错误重试。
- 图片上传。
- 下拉刷新 / 上拉加载。

组件设计重点：

- props 清晰。
- 事件语义明确。
- 不把业务逻辑塞进纯 UI 组件。
- loading、empty、error 状态完整。
- 样式不要污染全局。

## 八、项目经历表达模板

### 1. 完整业务项目模板

可以按这个顺序讲：

```text
项目背景 -> 我的职责 -> 技术方案 -> 遇到的问题 -> 解决方式 -> 最终结果
```

示例：

> 这个项目是一个移动端 H5 页面，主要用于用户在线预约和订单查看。我负责核心页面开发，包括首页、列表页、详情页、表单提交和订单状态展示。技术栈是 Vue3 + Vite + Pinia，适配方案用的是 vw，接口请求统一封装 axios。开发中主要处理了不同机型下的布局适配、iOS 输入框遮挡、图片加载慢和接口异常兜底。最后页面顺利上线，首屏通过路由懒加载、图片压缩和接口并发优化，加载体验比最初版本明显更稳定。

### 2. 移动端问题解决模板

示例：

> 有一次在 iOS 微信里，用户打开表单页输入内容后，底部提交按钮位置会错乱，并且键盘收起后页面无法恢复。排查后发现是 fixed 底部栏和软键盘触发视口变化导致的。我把输入态下的底部按钮做了降级处理，并在 blur 后修正滚动位置，同时让弹窗内部滚动，避免 body 滚动。处理后 iOS 和 Android 微信里的表现都稳定了。

### 3. 性能优化项目模板

示例：

> 项目初期首屏加载偏慢，我从资源体积、接口耗时和图片加载三个方向排查。资源方面做了路由懒加载和第三方库按需引入；图片方面改成压缩图和懒加载；接口方面把首屏必要接口并发请求，非首屏数据延后加载。同时加了骨架屏，减少用户等待感。优化后首屏体验明显改善。

## 九、高频面试问答

### 1. 移动端适配你怎么做？

答：

> 我会先看设计稿基准宽度和目标设备范围。如果是标准移动端 H5，我通常用 rem 或 vw 做整体适配，布局主要用 Flex/Grid。对于大屏和平板，会设置最大宽度或断点，避免页面被拉得太散。细节上会处理 1px 边框、iOS 安全区、图片比例、长文本换行和按钮点击区域。

### 2. H5 首屏慢怎么优化？

答：

> 我会先用 Network、Performance 或 Lighthouse 看瓶颈是在资源体积、接口耗时还是渲染。资源方面做压缩、分包、路由懒加载、CDN；图片做压缩、WebP、懒加载；接口做并发、缓存或延后加载；渲染上减少首屏 DOM 和非必要组件。用户体验上可以加骨架屏或缓存上次数据。

### 3. 移动端常见兼容问题有哪些？

答：

> 常见有 iOS fixed 定位异常、软键盘遮挡输入框、滚动穿透、微信浏览器返回缓存、100vh 高度不准、1px 边框、点击穿透或 touch/click 重复触发。处理方式一般是根据具体场景做样式降级、滚动锁定、事件节流、环境判断和兼容封装。

### 4. JSBridge 是什么？

答：

> JSBridge 是 H5 和原生 App 通信的桥。H5 通过约定的方法名和参数调用原生能力，比如分享、支付、登录、扫码、定位；原生也可以回调 H5。开发时我会统一封装 bridge 方法，处理环境判断、方法不存在、回调、超时和降级。

### 5. 小程序性能怎么优化？

答：

> 小程序重点关注包体积、首屏接口和 setData。一般会做分包加载，主包只放首屏和公共能力；图片走 CDN；接口减少首屏阻塞；避免频繁 setData 大对象；列表页分页加载或虚拟列表。

### 6. 你怎么和后端联调？

答：

> 我会先根据接口文档确认字段、状态码、错误码和边界情况。前端请求统一封装，页面上补齐 loading、empty、error、retry 状态。联调时会用 mock 数据先把页面流程跑通，再切真实接口。遇到问题会通过 Network、请求参数、响应数据和后端日志一起定位。

### 7. 如何保证移动端页面质量？

答：

> 我会从几个方面保证：开发阶段组件化和请求封装；样式上按设计稿还原并做多机型自测；接口上处理 loading、空状态、异常状态；上线前检查 iOS、Android、微信浏览器和主流尺寸；代码上配合 ESLint、Prettier、Git 分支和 Code Review。
