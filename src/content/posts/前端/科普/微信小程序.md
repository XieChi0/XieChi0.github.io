# 微信小程序面试复习

## 一、先建立整体感觉

微信小程序运行在微信客户端提供的容器里，可以调用微信原生能力，比如扫码、支付、定位、相册、分享等。

和 H5 相比，小程序更接近“受平台约束的轻应用”：

| 对比项 | H5 | 微信小程序 |
| --- | --- | --- |
| 运行环境 | 浏览器 / WebView | 微信客户端容器 |
| 页面结构 | HTML | WXML |
| 样式 | CSS | WXSS |
| 逻辑 | 浏览器 JS | App Service 里的 JS |
| DOM 操作 | 可以直接操作 DOM | 不能直接操作 DOM |
| 路由 | 前端路由控制 | 小程序页面栈 |
| 原生能力 | JSBridge / Web API | `wx.*` API |
| 适配单位 | px / rem / vw | rpx / px |

一句话：

> 小程序不是普通网页，也不是原生 App。它是微信提供运行环境、页面栈、组件体系和原生 API 的前端应用。

## 二、底层核心：双线程架构

小程序最重要的底层特点是：**视图层和逻辑层分离**。

| 线程 | 承载环境 | 核心职责 | 你写的代码 | 关键限制 |
| --- | --- | --- | --- | --- |
| 视图层 View Thread | WebView 渲染 | 页面结构渲染、样式展示、接收用户交互 | WXML、WXSS | 不能写业务 JS，不能直接操作 DOM |
| 逻辑层 App Service Thread | JSCore / V8 等 JS 引擎 | 执行业务 JS、管理状态、调用微信 API | app.js、页面 js、组件 js | 没有 `window` / `document` / DOM / BOM |

两个线程不直接互相操作，而是通过微信客户端 Native 层做中转。

```text
用户点击视图
  ↓
视图层捕获事件
  ↓
Native 转发事件
  ↓
逻辑层执行事件函数
  ↓
this.setData(...)
  ↓
Native 转发数据
  ↓
视图层重新渲染
```

所以小程序里不能像 H5 那样 `document.querySelector()` 后直接改 DOM。页面更新必须走 `setData`。

## 三、项目目录结构

一个标准小程序项目，根目录通常有 3 个全局文件，每个页面有 4 个同名文件。

```text
├── app.js                # 全局逻辑：小程序入口、全局生命周期、globalData
├── app.json              # 全局配置：页面路径、窗口样式、tabBar、权限等
├── app.wxss              # 全局样式
├── pages/                # 页面目录
│   ├── home/
│   │   ├── home.js       # 页面逻辑：Page()、data、事件、生命周期
│   │   ├── home.json     # 页面配置：标题、下拉刷新等
│   │   ├── home.wxml     # 页面结构
│   │   └── home.wxss     # 页面样式
│   └── detail/
│       ├── detail.js
│       ├── detail.json
│       ├── detail.wxml
│       └── detail.wxss
├── components/           # 公共组件
│   └── user-card/
│       ├── user-card.js
│       ├── user-card.json
│       ├── user-card.wxml
│       └── user-card.wxss
├── utils/                # 工具函数，如 request、formatTime
├── services/             # 接口层封装，如 user-api、order-api
└── assets/               # 图片、图标等静态资源
```

目录整理建议：

| 目录 | 放什么 | 不建议放什么 |
| --- | --- | --- |
| `pages/` | 独立页面，一个页面一个文件夹 | 公共组件、通用工具 |
| `components/` | 可复用 UI 组件 / 业务组件 | 完整页面逻辑 |
| `utils/` | 纯工具函数 | 带业务状态的接口逻辑 |
| `services/` 或 `api/` | 请求封装后的业务接口 | 页面渲染逻辑 |
| `assets/` | 本地图片、图标、字体 | 大量线上图片，优先放 CDN |

## 四、全局层：App 构造器

`app.js` 通过 `App()` 注册整个小程序实例。

```js
App({
  globalData: {
    token: '',
    userInfo: null,
  },

  onLaunch() {
    const token = wx.getStorageSync('token')
    if (token) {
      this.globalData.token = token
    }
  },

  onShow() {
    console.log('小程序进入前台')
  },

  onHide() {
    console.log('小程序进入后台')
  },
})
```

常见内容：

- `onLaunch`：小程序初始化，全局只执行一次。
- `onShow`：小程序启动或从后台回到前台时触发。
- `onHide`：小程序从前台切到后台时触发。
- `globalData`：全局共享数据，页面里可以通过 `getApp()` 读取。
- 全局方法：统一登录、初始化配置、埋点等。

注意：

> `globalData` 不是响应式的。A 页面改了 `globalData`，B 页面不会自动刷新，B 页面需要在 `onShow` 里重新读取，或者配合事件通知 / 状态管理。

## 五、全局配置：app.json

`app.json` 用来声明整个小程序的结构和外观。

```json
{
  "pages": [
    "pages/home/home",
    "pages/detail/detail"
  ],
  "window": {
    "navigationBarTitleText": "我的小程序",
    "navigationBarBackgroundColor": "#ffffff",
    "navigationBarTextStyle": "black",
    "enablePullDownRefresh": false
  },
  "tabBar": {
    "list": [
      {
        "pagePath": "pages/home/home",
        "text": "首页"
      },
      {
        "pagePath": "pages/profile/profile",
        "text": "我的"
      }
    ]
  },
  "permission": {
    "scope.userLocation": {
      "desc": "用于展示附近门店"
    }
  }
}
```

核心字段：

| 字段 | 作用 |
| --- | --- |
| `pages` | 注册所有主包页面，第一项是首页 |
| `window` | 全局导航栏、背景色、下拉刷新等窗口样式 |
| `tabBar` | 底部 / 顶部 tab 配置 |
| `permission` | 定位、相册等权限说明 |
| `subpackages` | 分包配置 |
| `preloadRule` | 分包预加载规则 |

## 六、页面层：Page 构造器

`Page()` 放在页面的 `xxx.js` 里，用来注册一个页面实例。你页面里空着的“page 构造器”目录，放在“页面层”下面是对的，因为它解释的正是页面逻辑文件怎么写。

最常见骨架：

```js
Page({
  data: {
    title: '商品详情',
    loading: false,
    goods: null,
    count: 1,
  },

  onLoad(options) {
    const { id } = options
    this.fetchDetail(id)
  },

  onShow() {
    console.log('页面显示')
  },

  onReady() {
    console.log('页面首次渲染完成')
  },

  onHide() {
    console.log('页面隐藏')
  },

  onUnload() {
    console.log('页面卸载')
  },

  async fetchDetail(id) {
    this.setData({ loading: true })
    try {
      const goods = await getGoodsDetail(id)
      this.setData({ goods })
    } finally {
      this.setData({ loading: false })
    }
  },

  handleAdd() {
    const nextCount = this.data.count + 1
    this.setData({ count: nextCount })
  },
})
```

### 1. Page 里常放什么

| 字段 / 方法 | 作用 | 类比 Vue / React |
| --- | --- | --- |
| `data` | 页面渲染需要的数据 | `data` / `state` |
| `onLoad` | 页面加载，拿路由参数、请求初始化数据 | `created` + 路由参数读取 |
| `onShow` | 页面每次显示都执行 | 页面激活 / 可见 |
| `onReady` | 页面首次渲染完成 | `mounted` 后的渲染完成 |
| `onHide` | 页面被隐藏 | 页面失活 |
| `onUnload` | 页面销毁 | `unmounted` |
| 自定义方法 | 事件处理、请求、业务逻辑 | `methods` / 组件函数 |

### 2. data 和 setData

`data` 是页面初始数据，WXML 通过 `{{ }}` 读取。

```js
Page({
  data: {
    username: '张三',
    isLogin: true,
  },

  changeName() {
    this.setData({
      username: '李四',
    })
  },
})
```

```xml
<view>{{username}}</view>
<button bindtap="changeName">修改名字</button>
```

关键点：

- 直接写 `this.data.username = '李四'` 不会触发视图更新。
- 必须用 `this.setData()` 通知视图层更新。
- `setData` 是逻辑层到视图层的通信，数据越大、次数越多，性能越差。

### 3. 事件处理函数

WXML 里用 `bindtap` 绑定页面方法。

```xml
<button bindtap="handleSubmit" data-id="{{item.id}}">
  提交
</button>
```

```js
Page({
  handleSubmit(e) {
    const { id } = e.currentTarget.dataset
    console.log('当前 id:', id)
  },
})
```

事件对象里常用：

| 字段 | 说明 |
| --- | --- |
| `e.currentTarget.dataset` | 读取 `data-*` 传参 |
| `e.detail` | 组件事件、自定义事件、表单输入值 |
| `e.target` | 触发事件的源节点 |
| `e.currentTarget` | 当前绑定事件的节点 |

`bindtap` 和 `catchtap` 的区别：

- `bindtap`：事件会继续冒泡。
- `catchtap`：事件会阻止冒泡。

### 4. 页面生命周期顺序

```text
页面创建
  ↓
onLoad(options)       页面加载，只执行一次，适合拿路由参数、初始化请求
  ↓
onShow()              页面显示，每次进入都可能执行
  ↓
onReady()             首次渲染完成，只执行一次
  ↓
onHide()              页面隐藏，比如跳到另一个页面
  ↓
onUnload()            页面卸载，比如 redirect 或返回销毁
```

面试表达：

> `onLoad` 更适合做一次性初始化，比如读取路由参数、请求详情数据。`onShow` 会在页面每次显示时触发，适合做返回刷新、重新读取全局状态、恢复页面数据。

### 5. 页面事件

除了生命周期，`Page()` 还支持一些页面级事件：

```js
Page({
  onPullDownRefresh() {
    this.refreshList().finally(() => {
      wx.stopPullDownRefresh()
    })
  },

  onReachBottom() {
    this.loadMore()
  },

  onShareAppMessage() {
    return {
      title: '分享标题',
      path: '/pages/home/home',
    }
  },
})
```

| 页面事件 | 场景 |
| --- | --- |
| `onPullDownRefresh` | 下拉刷新 |
| `onReachBottom` | 上拉触底加载更多 |
| `onPageScroll` | 页面滚动监听 |
| `onShareAppMessage` | 转发给好友 |
| `onShareTimeline` | 分享到朋友圈 |

### 6. Page 构造器常见坑

| 坑 | 原因 | 建议 |
| --- | --- | --- |
| 在回调里 `this` 丢失 | 普通函数有自己的 `this` | 用箭头函数或提前 `const that = this` |
| `onLoad` 请求回来后页面已卸载 | 异步请求晚于页面销毁 | 用状态位兜底，避免卸载后继续 setData |
| `onShow` 里重复请求过多 | 每次返回页面都会触发 | 区分初始化请求和返回刷新 |
| `data` 放太多临时变量 | 会增加传输和 diff 成本 | 不参与渲染的数据放普通实例字段 |
| 直接改 `this.data` | 不会更新视图 | 使用 `setData` |

实例字段写法：

```js
Page({
  data: {
    list: [],
  },

  pageNo: 1,
  requestTask: null,

  onUnload() {
    this.requestTask?.abort?.()
  },
})
```

`pageNo`、`requestTask` 不参与页面渲染，放在实例字段比放进 `data` 更合适。

## 七、组件层：Component 构造器

当页面里有重复 UI 模块，比如自定义导航栏、商品卡片、弹窗、表单项，就应该抽成自定义组件。组件通常放在 `components/` 目录下。

```text
components/
  user-card/
    user-card.js
    user-card.json
    user-card.wxml
    user-card.wxss
```

组件的 `json` 里需要声明：

```json
{
  "component": true
}
```

### 1. Component 基础骨架

```js
Component({
  properties: {
    name: {
      type: String,
      value: '',
    },
    age: Number,
  },

  data: {
    expanded: false,
  },

  lifetimes: {
    created() {
      console.log('组件实例创建')
    },
    attached() {
      console.log('组件挂载到页面')
    },
    ready() {
      console.log('组件布局完成')
    },
    detached() {
      console.log('组件被移除')
    },
  },

  observers: {
    'name, age'(name, age) {
      console.log('属性变化:', name, age)
    },
  },

  methods: {
    toggle() {
      this.setData({
        expanded: !this.data.expanded,
      })
    },

    handleSave() {
      this.triggerEvent('save', {
        name: this.properties.name,
      })
    },
  },
})
```

### 2. Component 里的核心字段

| 字段 | 作用 | 对应 Vue |
| --- | --- | --- |
| `properties` | 父组件传入的外部属性 | props |
| `data` | 组件内部状态 | data / state |
| `methods` | 组件内部方法、事件处理 | methods |
| `observers` | 监听 properties / data 变化 | watch |
| `lifetimes` | 组件生命周期 | created / mounted / unmounted |
| `pageLifetimes` | 组件所在页面的显示、隐藏等事件 | 页面级生命周期 |
| `options` | 组件配置，如多插槽 | 组件选项 |

### 3. 组件生命周期

小程序组件生命周期写在 `lifetimes` 里，关注的是“组件实例从创建到销毁”的过程。

```js
Component({
  lifetimes: {
    created() {
      console.log('组件实例刚创建')
    },

    attached() {
      console.log('组件进入页面节点树')
    },

    ready() {
      console.log('组件布局完成，可以安全获取节点信息')
    },

    moved() {
      console.log('组件在节点树里的位置发生变化')
    },

    detached() {
      console.log('组件离开页面节点树')
    },

    error(err) {
      console.error('组件方法抛错:', err)
    },
  },
})
```

> `created`：组件实例刚创建，此时还没有挂载到页面节点树里。
>
> 这个阶段可以初始化一些普通字段，但不要调用 `setData`，也不要获取节点信息，因为组件还没有进入页面结构。

> `attached`：组件已经进入页面节点树。
>
> 这个阶段最像 Vue 的 `mounted`，适合做初始化请求、读取 `properties`、订阅事件、初始化组件内部状态。

> `ready`：组件布局已经完成。
>
> 如果你要用 `createSelectorQuery` 获取组件节点尺寸、位置、布局信息，通常放在 `ready` 更稳。

> `moved`：组件实例被移动到节点树的另一个位置。
>
> 真实业务里用得不多，只有组件被动态移动位置时才比较有感。

> `detached`：组件从页面节点树移除。
>
> 适合清理定时器、取消事件订阅、停止异步轮询，避免页面切走后还继续执行逻辑。

> `error`：组件方法抛出错误时触发。
>
> 可以用于组件内部错误兜底，但不要把它当成全局错误监控的唯一方案。

组件生命周期大致顺序：

```text
created
  ↓
attached
  ↓
ready
  ↓
moved（位置变化时才触发）
  ↓
detached
```

组件还可以写 `pageLifetimes`，它监听的是“组件所在页面”的显示、隐藏和尺寸变化。

```js
Component({
  pageLifetimes: {
    show() {
      console.log('组件所在页面显示')
    },

    hide() {
      console.log('组件所在页面隐藏')
    },

    resize(size) {
      console.log('页面尺寸变化:', size)
    },
  },
})
```

> `lifetimes` 关注组件自己：创建、挂载、布局完成、销毁。
>
> `pageLifetimes` 关注组件所在页面：页面显示、页面隐藏、页面尺寸变化。

面试表达：

> 组件初始化逻辑一般放 `attached`，需要拿节点布局信息放 `ready`，清理定时器和事件监听放 `detached`。如果组件需要感知页面返回或切后台，可以用 `pageLifetimes.show/hide`。

### 4. 组件通信

#### 父传子：properties

父页面：

```xml
<user-card name="{{username}}" age="{{age}}" />
```

子组件：

```js
Component({
  properties: {
    name: String,
    age: Number,
  },
})
```

#### 子传父：triggerEvent

子组件：

```js
this.triggerEvent('save', {
  id: 1,
  name: this.properties.name,
})
```

父页面：

```xml
<user-card bind:save="handleSave" />
```

```js
Page({
  handleSave(e) {
    console.log(e.detail)
  },
})
```

#### 父拿子实例：selectComponent

```xml
<user-card id="userCard" />
```

```js
Page({
  refreshChild() {
    const child = this.selectComponent('#userCard')
    child.refresh()
  },
})
```

`selectComponent()` 获取的是组件实例，不是 DOM。它有点像 Vue 里的 `ref`，但会增加父子耦合，适合少量必要场景。

### 5. 组件样式隔离和穿透

小程序组件默认有样式隔离，父页面样式不会随便影响子组件。

如果父页面确实需要传样式，官方常用方案是 `externalClasses`。

```js
Component({
  externalClasses: ['custom-class'],
})
```

组件 WXML：

```xml
<view class="card custom-class">
  <slot />
</view>
```

父页面：

```xml
<user-card custom-class="profile-card" />
```

这可以理解成小程序官方支持的“样式入口”，比强行穿透更可控。

## 八、核心语法

### 1. WXML 常用语法

| HTML / Vue | 小程序 WXML | 说明 |
| --- | --- | --- |
| `<div>` | `<view>` | 块级容器 |
| `<span>` | `<text>` | 文本 |
| `<img>` | `<image>` | 图片 |
| `onclick` | `bindtap` | 点击事件 |
| `v-if` | `wx:if` | 条件渲染 |
| `v-for` | `wx:for` | 循环渲染 |
| `{{ name }}` | `{{name}}` | 数据绑定 |

### 2. 条件渲染

```xml
<view wx:if="{{loading}}">加载中...</view>
<view wx:elif="{{list.length === 0}}">暂无数据</view>
<view wx:else>有数据</view>
```

`wx:if` 是真正创建 / 销毁节点，适合切换频率不高的场景。

### 3. 列表渲染

```xml
<view wx:for="{{list}}" wx:key="id">
  <text>{{item.name}}</text>
</view>
```

建议：

- 能写稳定唯一 id，就不要用默认下标。
- 列表很大时不要一次性渲染全部数据。
- 列表项结构要尽量简单。

### 4. Vue 和小程序概念对照

| Vue | 微信小程序 |
| --- | --- |
| `props` / `defineProps()` | `properties` |
| `data()` / `ref()` | `data` |
| `methods` | `methods` |
| `watch` | `observers` |
| `emit()` | `triggerEvent()` |
| `@click` | `bindtap` / `catchtap` |
| `ref` | `selectComponent()` |
| `created` / `mounted` | `created` / `attached` / `ready` |
| `scoped + :deep()` | `externalClasses` |

## 九、事件系统

小程序事件系统主要围绕四件事：事件绑定、事件冒泡、阻止冒泡、事件传参。

### 1. bindtap 和 catchtap

```xml
<view class="parent" bindtap="handleParentTap">
  <button bindtap="handleChildTap">bindtap 按钮</button>
  <button catchtap="handleCatchTap">catchtap 按钮</button>
</view>
```

```js
Page({
  handleParentTap() {
    console.log('父容器被点击')
  },

  handleChildTap() {
    console.log('子按钮点击，事件继续冒泡')
  },

  handleCatchTap() {
    console.log('子按钮点击，事件到这里停止')
  },
})
```

> `bindtap`：绑定点击事件，但不会阻止事件冒泡。点击子元素后，子元素事件会先触发，然后继续往父元素冒泡。
>
> `catchtap`：绑定点击事件，并阻止事件继续冒泡。常用于弹窗、遮罩层、卡片内部按钮，避免点内部按钮时同时触发外层点击。

### 2. 事件冒泡

事件冒泡指的是：子节点触发事件后，事件会沿着节点树向父节点继续传递。

```xml
<view bindtap="handleCardTap">
  <button bindtap="handleBuyTap">购买</button>
</view>
```

点击按钮时，触发顺序是：

```text
handleBuyTap
  ↓
handleCardTap
```

如果不希望点击按钮时触发外层卡片点击，就把按钮改成 `catchtap`。

```xml
<view bindtap="handleCardTap">
  <button catchtap="handleBuyTap">购买</button>
</view>
```

> 面试里说清楚一句就够：`bindtap` 会冒泡，`catchtap` 会阻止冒泡。

### 3. dataset 事件传参

小程序里通常用 `data-*` 传递事件参数。

```xml
<view
  wx:for="{{list}}"
  wx:key="id"
  data-id="{{item.id}}"
  data-name="{{item.name}}"
  bindtap="handleItemTap"
>
  {{item.name}}
</view>
```

```js
Page({
  handleItemTap(e) {
    const { id, name } = e.currentTarget.dataset
    console.log(id, name)
  },
})
```

> `dataset` 适合传简单字段，比如 `id`、`type`、`index`。
>
> 不建议把整个复杂对象塞进 `data-*`，大对象可以通过 id 再去 `this.data.list` 里查。

### 4. target 和 currentTarget

| 字段 | 含义 |
| --- | --- |
| `e.target` | 真正触发事件的节点 |
| `e.currentTarget` | 当前绑定事件的节点 |

```xml
<view data-id="card-1" bindtap="handleTap">
  <text data-id="text-1">点击文字</text>
</view>
```

```js
Page({
  handleTap(e) {
    console.log(e.target.dataset.id)
    console.log(e.currentTarget.dataset.id)
  },
})
```

> 如果点击的是内部 `text`，`target` 是 `text`，`currentTarget` 是绑定事件的外层 `view`。
>
> 实战里取参数一般优先用 `e.currentTarget.dataset`，更稳定。

### 5. 常见事件

| 事件 | 场景 |
| --- | --- |
| `bindtap` | 点击 |
| `bindinput` | 输入框输入 |
| `bindchange` | 表单值变化 |
| `bindsubmit` | 表单提交 |
| `bindscroll` | 滚动 |
| `bindtouchstart` | 手指触摸开始 |
| `bindtouchmove` | 手指移动 |
| `bindtouchend` | 手指触摸结束 |

输入框示例：

```xml

```

```js
    this.setData({
      keyword: e.detail.value,
    })
  },
})
```

## 十、setData 机制与性能

`setData` 是小程序最容易被追问的 API。

```text
逻辑层
  this.setData({ name: '张三' })
        ↓
Native 序列化并转发数据
        ↓
视图层接收数据
        ↓
对比并重新渲染
```

### 1. 三个核心特点

1. **异步更新视图**：调用后不会立刻完成渲染。
2. **跨线程通信**：数据要从逻辑层传到视图层，有序列化成本。
3. **更新粒度影响性能**：传大对象、大数组、频繁传数据都会拖慢页面。

### 2. 正确写法

```js
// 错误：直接改 data 不会触发视图更新
this.data.count = this.data.count + 1

// 正确：用 setData 更新
this.setData({
  count: this.data.count + 1,
})
```

如果后续逻辑必须依赖更新后的视图，可以用回调。

```js
const nextCount = this.data.count + 1

this.setData({ count: nextCount }, () => {
  console.log('视图更新完成:', this.data.count)
})
```

### 3. 局部更新

不要每次都重传整个对象或整个数组，能更新局部就更新局部。

```js
this.setData({
  'user.name': '李四',
  [`list[${index}].checked`]: true,
})
```

分页列表可以按页存，避免每次重传越来越大的完整数组。

```js
Page({
  data: {
    listPages: [],
  },

  appendPage(newList) {
    const pageIndex = this.data.listPages.length
    this.setData({
      [`listPages[${pageIndex}]`]: newList,
    })
  },
})
```

WXML：

```xml
<block wx:for="{{listPages}}" wx:for-item="page" wx:key="*this">
  <view wx:for="{{page}}" wx:key="id">
    {{item.name}}
  </view>
</block>
```

### 4. setData 性能红线

| 坑 | 为什么 | 怎么做 |
| --- | --- | --- |
| 一次传很大的对象 / 数组 | 跨线程传输和 diff 成本高 | 只传变化字段 |
| 高频滚动里 setData | 每次滚动都通信，容易掉帧 | 节流、防抖、减少渲染字段 |
| 把不渲染的数据放进 data | data 变化会牵动视图层 | 放实例字段 |
| 后台接口字段全塞进页面 | 很多字段不参与渲染 | 前端裁剪字段 |
| 长列表一次渲染几百上千项 | 视图节点太多 | 分页、虚拟列表、回收列表 |

面试表达：

> 小程序性能优化里我会重点控制 `setData` 的频率和数据量。页面只更新视图真正需要的字段，大列表做分页或虚拟列表，不把接口原始大对象完整塞进 `data`。滚动、输入这类高频场景会节流，避免逻辑层和视图层频繁通信。

## 十一、路由跳转⭐

小程序路由是页面栈模型，不是浏览器 history。

| API | 作用 | 是否保留当前页 | 常见场景 |
| --- | --- | --- | --- |
| `wx.navigateTo` | 打开新页面 | 保留 | 列表到详情 |
| `wx.redirectTo` | 关闭当前页，打开新页 | 不保留 | 登录后跳转 |
| `wx.switchTab` | 跳到 tabBar 页面 | 按 tab 规则处理 | 首页 / 我的 |
| `wx.navigateBack` | 返回上一页或多页 | 销毁当前页 | 详情返回列表 |
| `wx.reLaunch` | 关闭所有页面，打开新页 | 不保留任何页面 | 重置到首页 |

### 1. navigateTo：保留当前页，打开新页

```js
wx.navigateTo({
  url: '/pages/detail/detail?id=123&type=goods',
})
```

> `navigateTo` 会把新页面压入页面栈，当前页面保留。
>
> 典型场景是列表页跳详情页，用户可以返回上一页。

详情页读取参数：

```js
Page({
  onLoad(options) {
    console.log(options.id)
    console.log(options.type)
  },
})
```

### 2. redirectTo：关闭当前页，打开新页

```js
wx.redirectTo({
  url: '/pages/result/result?status=success',
})
```

> `redirectTo` 会关闭当前页面，再打开目标页面。
>
> 典型场景是表单提交成功后跳结果页，不希望用户返回刚才的提交页。

### 3. switchTab：跳转 tabBar 页面

```js
wx.switchTab({
  url: '/pages/profile/profile',
})
```

> `switchTab` 只能跳 `app.json` 里配置过的 `tabBar` 页面。
>
> 跳 tabBar 页面必须用 `switchTab`，不能用 `navigateTo`。

### 4. reLaunch：关闭所有页面，重新打开

```js
wx.reLaunch({
  url: '/pages/home/home',
})
```

> `reLaunch` 会清空整个页面栈，然后打开目标页面。
>
> 典型场景是退出登录后回首页、流程结束后重置页面栈。

### 5. navigateBack：返回上一页

```js
wx.navigateBack({
  delta: 1,
})
```

> `delta` 表示返回几层页面，默认是 1。
>
> 如果详情页提交后要回到列表页，可以先操作上一页实例，再 `navigateBack`。

注意：

- `switchTab` 只能跳 `tabBar` 页面。
- `navigateTo` 不能跳 `tabBar` 页面。
- 页面栈有层数限制，太深要用 `redirectTo` 或 `reLaunch` 控制。
- 路由传参只能放在 URL query 里，复杂数据更适合用 EventChannel 或缓存。

## 十二、本地存储 setStorage

微信小程序本地存储有同步和异步两套 API。

| 类型 | API | 特点 |
| --- | --- | --- |
| 同步 | `wx.setStorageSync` / `wx.getStorageSync` | 写法简单，但会阻塞逻辑线程 |
| 异步 | `wx.setStorage` / `wx.getStorage` | 不阻塞，通过回调或 Promise 化处理 |

登录 token 常用同步存储：

```js
wx.setStorageSync('token', token)

const token = wx.getStorageSync('token')
```

列表缓存、较大数据更适合异步：

```js
wx.setStorage({
  key: 'goodsList',
  data: list,
})
```

和 H5 `localStorage` 的区别：

- API 不同。
- 小程序每个应用有独立存储空间。
- 小程序没有浏览器同源下的 `localStorage` 概念。
- 同步 API 会阻塞逻辑线程，大数据更要谨慎。

## 十三、网络请求

基础写法：

```js
wx.request({
  url: 'https://api.example.com/goods',
  method: 'GET',
  data: {
    page: 1,
  },
  header: {
    'content-type': 'application/json',
    token: wx.getStorageSync('token'),
  },
  success(res) {
    console.log(res.statusCode, res.data)
  },
  fail(err) {
    console.error(err)
  },
  complete() {
    console.log('请求结束')
  },
})
```

常见限制和注意点：

- 线上环境必须使用 HTTPS。
- 请求域名需要在微信公众平台配置合法域名。
- 开发阶段可以在开发者工具里临时关闭域名校验。
- 小程序请求不是浏览器发起的，所以不存在浏览器同源策略下的跨域问题。
- 真实项目一般会封装统一 request。

简单封装：

```js
const BASE_URL = 'https://api.example.com'

function request(options) {
  return new Promise((resolve, reject) => {
    wx.request({
      url: BASE_URL + options.url,
      method: options.method || 'GET',
      data: options.data || {},
      header: {
        'content-type': 'application/json',
        token: wx.getStorageSync('token') || '',
        ...options.header,
      },
      success(res) {
        if (res.statusCode === 401) {
          wx.redirectTo({ url: '/pages/login/login' })
          reject(res)
          return
        }

        if (res.statusCode >= 200 && res.statusCode < 300) {
          resolve(res.data)
        } else {
          wx.showToast({
            title: '请求失败',
            icon: 'none',
          })
          reject(res)
        }
      },
      fail(err) {
        wx.showToast({
          title: '网络异常',
          icon: 'none',
        })
        reject(err)
      },
    })
  })
}
```

封装时通常做这些事：

- 统一 `baseURL`。
- 自动注入 token。
- 统一处理 loading。
- 统一处理 401 登录过期。
- 统一错误提示。
- 控制重复请求、超时、重试。

## 十四、常用 API

小程序 API 基本都挂在 `wx` 对象上，常见分类包括网络、登录、媒体、位置、界面反馈、存储、路由等。

### 1. wx.request：网络请求

```js
wx.request({
  url: 'https://api.example.com/list',
  method: 'GET',
  data: {
    page: 1,
  },
  header: {
    token: wx.getStorageSync('token'),
  },
  success(res) {
    console.log(res.statusCode, res.data)
  },
  fail(err) {
    console.error(err)
  },
})
```

> `wx.request` 是小程序发接口的基础 API。
>
> 真实项目一般不会在页面里到处直接写，而是封装成 `request()`，统一处理 baseURL、token、错误提示、登录过期。

### 2. wx.login：登录凭证

```js
wx.login({
  success(res) {
    console.log(res.code)
  },
})
```

登录链路：

```text
wx.login 拿 code
  ↓
前端把 code 发给后端
  ↓
后端换 openid / session_key
  ↓
后端生成业务 token
  ↓
前端保存 token
```

> `wx.login` 拿到的 `code` 不是用户信息，也不是 token。
>
> 它只是一个临时登录凭证，必须交给后端去换用户身份。

### 3. wx.chooseImage：选择图片

```js
wx.chooseImage({
  count: 1,
  sizeType: ['compressed'],
  sourceType: ['album', 'camera'],
  success(res) {
    const tempFilePath = res.tempFilePaths[0]
    console.log(tempFilePath)
  },
})
```

> `wx.chooseImage` 用来从相册或相机选择图片。
>
> 返回的是临时文件路径，如果要长期保存或上传服务器，还需要配合 `wx.uploadFile`。

上传示例：

```js
wx.uploadFile({
  url: 'https://api.example.com/upload',
  filePath: tempFilePath,
  name: 'file',
  success(res) {
    console.log(res.data)
  },
})
```

### 4. wx.showToast / wx.showLoading

```js
wx.showToast({
  title: '保存成功',
  icon: 'success',
})

wx.showLoading({
  title: '加载中',
})

wx.hideLoading()
```

> `showToast` 适合轻提示，比如成功、失败、无权限。
>
> `showLoading` 适合接口请求、提交表单等需要等待的场景，用完记得 `hideLoading`。

### 5. wx.getStorageSync / wx.setStorageSync

```js
wx.setStorageSync('token', token)
const token = wx.getStorageSync('token')
```

> 同步存储适合小体积、立刻要用的数据，比如 token。
>
> 大数据建议用异步存储，避免阻塞逻辑线程。

### 6. wx.getLocation：获取位置

```js
wx.getLocation({
  type: 'gcj02',
  success(res) {
    console.log(res.latitude, res.longitude)
  },
})
```

> 定位属于敏感权限，要在用户触发相关功能时再申请，并准备好拒绝授权后的降级提示。

常用 API 速记：

| API | 作用 | 常见场景 |
| --- | --- | --- |
| `wx.request` | 发网络请求 | 列表、详情、提交表单 |
| `wx.login` | 获取登录 code | 登录态初始化 |
| `wx.chooseImage` | 选择图片 | 头像、评价晒图、上传凭证 |
| `wx.uploadFile` | 上传文件 | 图片上传、文件上传 |
| `wx.showToast` | 轻提示 | 成功、失败、校验提示 |
| `wx.showLoading` | 加载提示 | 接口请求、提交中 |
| `wx.setStorageSync` | 同步写缓存 | token、小配置 |
| `wx.getStorageSync` | 同步读缓存 | 恢复登录态 |
| `wx.getLocation` | 获取位置 | 附近门店、地图定位 |
| `wx.navigateTo` | 打开新页面 | 列表到详情 |

## 十五、分包加载

分包解决两个问题：

- 主包太大，影响首次启动。
- 业务模块多，所有代码都放主包不利于维护。

推荐思路：

```text
主包：启动必需、首页、tabBar 页面、公共能力
分包：低频页面、业务详情、活动页、管理页、功能模块
独立分包：可以独立启动的活动页 / 工具页
```

配置示例：

```json
{
  "pages": [
    "pages/home/home",
    "pages/profile/profile"
  ],
  "subpackages": [
    {
      "root": "packages/order",
      "name": "order",
      "pages": [
        "pages/list/list",
        "pages/detail/detail"
      ]
    }
  ],
  "preloadRule": {
    "pages/home/home": {
      "network": "all",
      "packages": ["order"]
    }
  }
}
```

对应目录：

```text
├── pages/
│   ├── home/
│   └── profile/
└── packages/
    └── order/
        └── pages/
            ├── list/
            └── detail/
```

分包原则：

- tabBar 页面必须在主包。
- 首屏必需页面放主包。
- 低频、重资源、业务独立页面放分包。
- 公共组件如果主包和多个分包都要用，通常放主包公共目录。
- 分包内私有组件放分包内部，避免主包变胖。

> 包大小限制会随平台策略调整，面试时可以说“主包和单个分包都有大小限制，整体包也有限制，实际项目以微信开发者工具上传校验和官方文档为准”。重点不是背死数字，而是说明你会用分包、预加载、资源 CDN 和按需引用控制首屏体积。

## 十六、跨页面通信

| 方式 | 适用场景 | 优点 | 缺点 |
| --- | --- | --- | --- |
| URL 参数 | 简单参数，如 id、type | 简单直接 | 长度有限，不适合敏感/复杂数据 |
| `globalData` | 全局共享状态 | 使用方便 | 不响应式，不持久 |
| 本地存储 | 跨页面、跨启动保存 | 可持久化 | 读写有成本，没有通知机制 |
| EventChannel | `navigateTo` 打开的页面之间通信 | 官方推荐，适合 A 到 B 传复杂数据 | 只适合特定页面跳转关系 |
| EventBus | 多页面广播 | 灵活解耦 | 需要自己实现和清理监听 |
| `getCurrentPages` | 操作上一页实例 | 能直接调用上一页方法 | 强耦合，页面栈变化容易出错 |

### 1. URL 传参

```js
wx.navigateTo({
  url: '/pages/detail/detail?id=123&type=goods',
})
```

```js
Page({
  onLoad(options) {
    console.log(options.id)
  },
})
```

### 2. globalData

```js
// app.js
App({
  globalData: {
    token: '',
  },
})
```

```js
// 页面里
const app = getApp()
app.globalData.token = 'xxx'
```

适合放登录态、用户信息、全局配置，但页面不会自动响应变化。

### 3. EventChannel

A 页面：

```js
wx.navigateTo({
  url: '/pages/detail/detail',
  success(res) {
    res.eventChannel.emit('acceptData', {
      id: 1,
      from: 'list',
    })
  },
})
```

B 页面：

```js
Page({
  onLoad() {
    const eventChannel = this.getOpenerEventChannel()

    eventChannel.on('acceptData', (data) => {
      console.log(data)
    })
  },
})
```

### 4. getCurrentPages 刷新上一页

典型场景：详情页提交成功后，返回列表页并刷新。

```js
const pages = getCurrentPages()
const prevPage = pages[pages.length - 2]

prevPage.setData({
  needRefresh: true,
})

wx.navigateBack()
```

列表页：

```js
Page({
  onShow() {
    if (this.data.needRefresh) {
      this.setData({ needRefresh: false })
      this.refreshList()
    }
  },
})
```

### 5. 迷你 EventBus

```js
const bus = {
  events: {},

  on(name, callback) {
    if (!this.events[name]) {
      this.events[name] = []
    }

    this.events[name].push(callback)
  },

  emit(name, data) {
    const callbacks = this.events[name] || []
    callbacks.forEach((callback) => callback(data))
  },

  off(name, callback) {
    if (!callback) {
      this.events[name] = []
      return
    }

    this.events[name] = (this.events[name] || []).filter((item) => item !== callback)
  },
}

export default bus
```

使用时要记得在页面卸载时解绑，避免重复监听。

## 十七、登录与权限

常见登录链路：

```text
wx.login()
  ↓
拿到 code
  ↓
发给后端
  ↓
后端用 code 换 openid / session_key
  ↓
后端生成业务 token
  ↓
前端保存 token
  ↓
后续请求带 token
```

前端示例：

```js
wx.login({
  success: async (res) => {
    const token = await request({
      url: '/auth/login',
      method: 'POST',
      data: {
        code: res.code,
      },
    })

    wx.setStorageSync('token', token)
  },
})
```

权限相关要点：

- 敏感能力需要用户授权，比如定位、相册、摄像头。
- 授权失败要给用户明确提示和降级方案。
- 不要一打开小程序就索要所有权限，最好在用户触发相关功能时再申请。

## 十八、性能优化清单

小程序性能优化主要围绕启动、渲染、通信、资源四条线。

### 1. 启动优化

- 控制主包体积。
- 使用分包加载。
- 首页只加载首屏必要数据。
- 图片、视频等大资源走 CDN。
- 非首屏模块延迟加载。

### 2. 渲染优化

- 减少 WXML 节点层级。
- 列表分页加载。
- 长列表使用虚拟列表 / 回收列表。
- 减少复杂选择器和过深嵌套。
- 避免在滚动中频繁更新视图。

### 3. setData 优化

- 只传变化字段。
- 不传大对象和大数组。
- 高频事件节流。
- 不参与渲染的数据不要放进 `data`。
- 用路径更新：`'list[0].checked': true`。

### 4. 请求优化

- 首屏接口并发。
- 合并非必要接口。
- 缓存低频变化数据。
- loading、empty、error、retry 状态完整。
- 登录过期统一拦截。

### 5. 资源优化

- 图片压缩。
- 使用合适尺寸的图片。
- 大图、背景图走 CDN。
- 避免把大量图片放进主包。
- 组件库按需引入。

面试表达：

> 我优化小程序一般先看启动和页面渲染。启动上控制主包体积，低频页面拆分包，图片走 CDN；渲染上减少节点层级和 setData 数据量，列表做分页或虚拟列表；接口上首屏必要数据优先，非关键数据延后，并补齐 loading、空状态和异常重试。

## 十九、高频面试问答

### 1. 小程序为什么不能直接操作 DOM？

因为小程序是双线程架构，WXML/WXSS 在视图层，JS 在逻辑层。逻辑层没有浏览器 DOM 环境，也没有 `window`、`document`，页面更新必须通过 `setData` 把数据传给视图层。

### 2. Page 和 Component 有什么区别？

`Page()` 注册页面，负责页面生命周期、路由参数、页面事件和业务流程。`Component()` 注册组件，负责可复用 UI 和局部逻辑，有 `properties`、`methods`、`lifetimes`、`observers` 等字段。页面是路由单位，组件是复用单位。

### 3. onLoad 和 onShow 有什么区别？

`onLoad` 页面创建时执行一次，适合拿路由参数和初始化请求。`onShow` 每次页面显示都会执行，适合返回刷新、重新读取全局状态、恢复页面数据。

### 4. setData 为什么影响性能？

因为 `setData` 是逻辑层到视图层的跨线程通信，数据需要序列化、传输、反序列化和渲染 diff。数据越大、调用越频繁，性能压力越大，所以要减少数据量、降低频率、做局部更新。

### 5. 小程序怎么做组件通信？

父传子用 `properties`，子传父用 `triggerEvent`，父直接调用子组件实例可以用 `selectComponent`。跨页面通信可以用 URL 参数、`globalData`、本地存储、EventChannel、EventBus 或 `getCurrentPages`，按场景选择。

### 6. 小程序请求有跨域问题吗？

没有浏览器同源策略意义上的跨域问题，因为请求不是浏览器直接发起的。但小程序有自己的平台限制：线上必须配置合法服务器域名，并且通常要求 HTTPS。

### 7. 分包怎么拆？

主包放首页、tabBar 页面、启动必需能力和公共资源。低频页面、业务详情、活动页、管理页拆分包。分包内私有组件放分包里，公共组件放主包公共目录，同时配合分包预加载提升体验。

### 8. 小程序本地存储怎么选同步还是异步？

小数据、启动时必须立即读取的数据，比如 token，可以用同步 API，代码简单。较大数据、列表缓存、非关键数据建议用异步 API，避免阻塞逻辑线程。

### 9. 小程序长列表怎么优化？

先分页加载，避免一次请求和渲染太多数据。列表项结构尽量简单，图片懒加载。数据更新时不要重传完整大数组，可以按页存储或局部更新。数据量特别大时使用虚拟列表或回收列表。

### 10. 你怎么介绍自己的小程序经验？

可以这样说：

> 我做小程序时主要关注页面结构、组件复用、接口封装和性能。页面用 `Page()` 管理生命周期和业务流程，公共 UI 抽成 `Component()`，请求层统一处理 token、错误提示和登录过期。性能上会控制主包体积，低频页面拆分包，图片走 CDN，列表分页加载，并尽量减少 `setData` 的频率和数据量。

## 二十、快速背诵版

- 小程序核心：双线程架构，视图层和逻辑层分离。
- 页面更新：不能操作 DOM，必须通过 `setData`。
- `App()`：注册全局小程序实例，放生命周期、`globalData`、全局方法。
- `Page()`：注册页面，放 `data`、页面生命周期、事件处理、业务方法。
- `Component()`：注册组件，放 `properties`、`data`、`methods`、`lifetimes`、`observers`。
- `onLoad`：页面初始化一次，拿参数、发首屏请求。
- `onShow`：每次显示执行，适合返回刷新。
- 父传子：`properties`。
- 子传父：`triggerEvent`。
- 父拿子：`selectComponent`。
- 跨页面：URL、`globalData`、storage、EventChannel、EventBus、`getCurrentPages`。
- 性能重点：主包体积、分包、图片 CDN、首屏请求、长列表、减少 `setData`。
- 请求限制：线上配置合法域名，使用 HTTPS；没有浏览器跨域问题。
