# Git Worktree 速记

## 1. Worktree 是什么

`worktree` 可以理解为：

> 同一个 Git 仓库，开出多个独立的工作目录。

它不是重新 `clone` 一个仓库，也不是简单复制项目，而是：

- Git 历史和提交对象是共享的
- 每个 worktree 都是一个真实的文件夹
- 每个 worktree 可以对应不同分支
- 每个 worktree 都可以单独修改、运行、提交

简单理解：

```txt
普通 branch：
一个文件夹里切换不同分支

worktree：
多个文件夹同时放着不同分支
```

例如：

```txt
F:/zz_blog/astro-boke       main 分支
F:/zz_blog/worktree-dev     astro-worktree-dev 分支
```

这两个目录都属于同一个 Git 仓库体系，但它们是两个不同的工作区。

---

## 2. Branch 和 Worktree 的区别

`branch` 解决的是：

> 代码版本线的问题。

`worktree` 解决的是：

> 多个分支能不能同时放在多个文件夹里并行工作的问题。

对比：

| 概念         | 含义                               |
| ------------ | ---------------------------------- |
| branch       | 一条代码版本线                     |
| worktree     | 一个真实的工作目录                 |
| commit       | 提交到当前 worktree 所在的当前分支 |
| merge        | 把一个分支的改动合到另一个分支     |
| push         | 把本地提交推送到远程仓库           |
| squash merge | 只拿改动，不保留一堆临时提交历史   |

一句话：

> branch 是“线”，worktree 是“桌子”。

普通分支像一张桌子上切换不同任务；worktree 像同时开了多张桌子，每张桌子放一个任务。

---

## 3. Cursor 里的 Worktree

在 Cursor 里，选择 `Worktree` 通常不是“复用你已经创建好的 worktree”，而是：

> Cursor 为当前 Agent 新建一个隔离的临时 worktree。

所以你可能会看到类似目录：

```txt
C:/Users/XChi/.cursor/worktrees/astro-boke/vns
C:/Users/XChi/.cursor/worktrees/astro-boke/ycm
```

这些目录是 Cursor 自动创建的 Agent 工作区。

可以这样理解：

```txt
一个 Agent ≈ 一个临时 worktree ≈ 一个临时分支
```

如果你已经手动创建了：

```txt
F:/zz_blog/worktree-dev
```

并且想让 Cursor 使用它，应该直接在 Cursor 中打开这个目录，然后选择 `Local`。

注意：

```txt
Cursor 的 Local = 当前打开的目录
Cursor 的 Worktree = 再创建一个新的隔离目录
```

所以：

```txt
打开 F:/zz_blog/worktree-dev
选择 Local
```

这时 Cursor 实际就在你的 `astro-worktree-dev` 这个 worktree 里工作。

## 4. 创建、查看和进入 Worktree

### 创建 worktree

常见写法：

```bash
# 格式：git worktree add <目录路径> <分支名>
git worktree add ../worktree-dev astro-worktree-dev
```

这会在上级目录创建 `worktree-dev` 文件夹，并把它切到 `astro-worktree-dev` 分支。

如果这个分支还不存在，可以使用 `-b` 创建新分支：

```bash
git worktree add ../worktree-dev -b astro-worktree-dev
```

### 查看所有 worktree

```bash
git worktree list
```

输出可能类似：

```txt
F:/zz_blog/astro-boke                                  26eca8f [main]
F:/zz_blog/worktree-dev                                af65611 [astro-worktree-dev]
C:/Users/XChi/.cursor/worktrees/astro-boke/vns        123abcd [cursor/vns]
```

方括号里的就是当前 worktree 对应的分支。

注意：如果你看到类似下面的内容：

```txt
F:/zz_blog/worktree-dev    dbce631  detached HEAD
```

或者：

```txt
F:/zz_blog/worktree-dev    dbce631  detached at xxx
```

说明它可能不是正常挂在某个分支上，而是处于“游离 HEAD”状态。平时为了方便提交和合并，更推荐让 worktree 对应一个明确分支，例如 `[astro-worktree-dev]`。

### 进入某个 worktree

worktree 本身就是一个真实文件夹，直接 `cd` 进去即可：

```bash
cd ../worktree-dev
```

或者打开 Cursor 的 Agent 临时 worktree：

```bash
cd "C:/Users/XChi/.cursor/worktrees/astro-boke/vns"
```

## 5. Worktree 里的代码怎么验证

worktree 本质上就是另一个项目目录，所以验证方式和普通项目一样。

### 安装依赖

如果是项目第一次进入该 worktree，通常需要安装依赖：

```bash
pnpm install
```

如果项目用的是 npm，则是：

```bash
npm install
```

### 启动开发服务

```bash
pnpm dev
```

或者：

```bash
npm run dev
```

如果主项目已经占用了端口，Vite / Astro 可能会自动换端口，比如从：

```txt
http://localhost:5173
```

换成：

```txt
http://localhost:5174
```

也可以手动指定端口：

```bash
npm run dev -- --port 5174
```

## 6. 怎么查看某个 Worktree 改了什么

不需要一定 `cd` 到目录里，也可以使用 `git -C` 指定目录执行 Git 命令。

查看某个 worktree 的修改状态：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" status -s
```

可能输出：

```txt
 M src/pages/index.astro
 M src/components/Header.astro
?? src/components/NewCard.astro
```

含义：

```txt
M  = modified，已修改
?? = untracked，新文件，还没被 Git 跟踪
```

查看具体改动：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" diff
```

只看某个文件的改动：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" diff -- src/pages/index.astro
```

也可以直接用 Cursor 打开这个 worktree 目录，这样左侧 Source Control 就能可视化查看修改文件和 diff：

```bash
cursor "C:/Users/XChi/.cursor/worktrees/astro-boke/vns"
```

## 7. Worktree 里的代码怎么提交

在哪里执行 `git commit`，就会提交到当前 worktree 所在的分支。

例如当前 worktree 是：

```txt
C:/Users/XChi/.cursor/worktrees/astro-boke/vns
```

对应分支是：

```txt
cursor/vns
```

那么执行：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" add -A
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" commit -m "temp: cursor agent changes"
```

这个提交会进入：

```txt
cursor/vns
```

而不是进入 `main`。

## 8. 推荐用法：把 Agent 分支当临时草稿

Cursor 创建很多 Agent worktree 后，确实可能出现很多临时分支。

推荐做法是：

```txt
Agent worktree 里改代码
↓
确认改得不错
↓
临时 commit
↓
回到 main
↓
squash 合并
↓
在 main 上正式 commit
↓
push main
↓
删除临时 worktree 和临时分支
```

这样最终 `main` 上只保留一个干净的正式提交。

流程示例：

```bash
# 1. 查看所有 worktree，找到 Agent 目录和分支名
git worktree list
```

假设看到：

```txt
C:/Users/XChi/.cursor/worktrees/astro-boke/vns   123abcd [cursor/vns]
```

先在 Agent worktree 里临时提交：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" add -A
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" commit -m "temp: cursor agent changes"
```

回到主项目：

```bash
cd "F:/zz_blog/astro-boke"
git checkout main
git pull
```

把 Agent 分支的改动 squash 到 main：

```bash
git merge --squash cursor/vns
```

此时改动会被拿到 main，但还没有正式提交。

在 main 上验证代码：

```bash
pnpm dev
```

确认没问题后，在 main 上正式提交：

```bash
git commit -m "feat: 修改博客页面"
git push origin main
```

如果你的远程仓库名不是 `origin`，而是 `astro-boke`，那就用：

```bash
git push astro-boke main
```

最后清理临时 worktree 和分支：

```bash
git worktree remove "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" --force
git branch -D cursor/vns
```

## 9. 另一种合并方式：Rebase 到 main 再快进合并

如果你不是 Cursor 临时 Agent，而是自己长期维护的 worktree 分支，例如：

```txt
astro-worktree-dev
```

也可以使用 rebase + fast-forward 的方式合并。

假设主分支是 `main`，worktree 分支是 `astro-worktree-dev`：

```bash
# 1. 进入 worktree 目录
cd ../worktree-dev

# 2. 确认当前分支
git checkout astro-worktree-dev

# 3. 先让 main 更新到最新
git fetch origin
git rebase origin/main
```

然后回到主仓库：

```bash
cd ../astro-boke
git checkout main
git pull

# 4. 快进合并
git merge astro-worktree-dev

# 5. 推送 main
git push origin main
```

这种方式的特点：

```txt
优点：历史比较直线、干净
缺点：rebase 过程中如果有冲突，需要解决冲突
```

如果只是 Cursor 临时 Agent 分支，通常我更推荐上一节的 `git merge --squash`，因为它更适合“只保留最终结果”。

## 10. 推送前检查

这个项目建议使用：

```txt
Node.js 22.x 或 23.x
pnpm 最新版本
```

如果本地 Node 版本不对，可以切换：

```bash
# nvm 用户
nvm install 22
nvm use 22

# fnm 用户
fnm install 22
fnm use 22
```

正式 push 前，建议按顺序执行：

```bash
# 1. 代码格式和 lint 检查
pnpm biome ci ./src

# 2. Astro / TypeScript 检查
pnpm astro check

# 3. 构建测试
pnpm astro build
```

如果 `pnpm biome ci ./src` 报格式问题，可以先运行：

```bash
pnpm biome check --write ./src
```

然后重新执行检查。

如果 `git push` 失败，并且你是用 SSH 推 GitHub，可以先测试 SSH 是否连通：

```bash
ssh -T git@github.com
```

再重新执行：

```bash
git push origin main
```

或者你的远程名是 `astro-boke` 时：

```bash
git push astro-boke main
```
如果还是失败可以用
```bash
git config --global core.sshCommand "C:/Windows/System32/OpenSSH/ssh.exe"
```

## 11. 怎么删除不用的 Worktree

先查看：

```bash
git worktree list
```

检查某个 worktree 里有没有要保留的改动：

```bash
git -C "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" status -s
```

确认不要后删除：

```bash
git worktree remove "C:/Users/XChi/.cursor/worktrees/astro-boke/vns"
```

如果提示有未提交改动，但你确认不要，可以强制删除：

```bash
git worktree remove "C:/Users/XChi/.cursor/worktrees/astro-boke/vns" --force
```

如果是自己手动创建的 worktree：

```bash
git worktree remove ../worktree-dev
```

如果你手动删除过 worktree 文件夹，Git 记录里还有残留，可以清理：

```bash
git worktree prune
```

可选：如果某个 worktree 不想被误删，可以锁定：

```bash
git worktree lock <路径>
```

解锁：

```bash
git worktree unlock <路径>
```

## 12. 常用命令速查

| 命令                                                     | 作用                            |
| -------------------------------------------------------- | ------------------------------- |
| `git worktree list`                                      | 查看所有 worktree               |
| `git worktree add ../worktree-dev astro-worktree-dev`    | 基于已有分支创建 worktree       |
| `git worktree add ../worktree-dev -b astro-worktree-dev` | 创建新分支并创建 worktree       |
| `git -C "worktree路径" status -s`                        | 查看某个 worktree 的修改状态    |
| `git -C "worktree路径" diff`                             | 查看某个 worktree 的具体改动    |
| `git -C "worktree路径" add -A`                           | 暂存某个 worktree 的改动        |
| `git -C "worktree路径" commit -m "提交说明"`             | 在某个 worktree 所在分支提交    |
| `git merge --squash 分支名`                              | 把临时分支的改动压到当前分支    |
| `git worktree remove "worktree路径"`                     | 删除某个 worktree               |
| `git worktree remove "worktree路径" --force`             | 强制删除有未提交改动的 worktree |
| `git worktree prune`                                     | 清理已失效的 worktree 引用      |
| `git worktree lock <路径>`                               | 锁定 worktree，防止被误删       |
| `git worktree unlock <路径>`                             | 解锁 worktree                   |

## 13. 最后记忆

```txt
branch = 代码分支
worktree = 分支对应的独立文件夹
commit = 提交到当前 worktree 所在分支
push = 推送到远程仓库
merge = 把一个分支合到另一个分支
squash merge = 只拿改动，不保留一堆临时提交历史
```

最重要的一句话：

> worktree 不是必须用，但它适合隔离、试错、并行开发，尤其适合 Cursor Agent 这种可能会大改代码的场景。

在 Cursor 里可以这样记：

```txt
想用已有 worktree：
直接打开那个 worktree 文件夹，选择 Local

想让 Cursor 再开一个隔离实验区：
选择 Worktree
```
