# JavaScript 对象与函数：module.exports、对象字面量、箭头函数

## 1. `module.exports = {}` 是什么

这段代码：

```js
module.exports = {
    post(path, serviceFunc) {
        // ...
    }
};
```

**是 JavaScript 语法**，用来**导出一个对象**。

### `module.exports` 是什么

在 Node.js 中，`module.exports` 是用来**导出模块**的机制。你写的内容可以被其他文件通过 `require()` 引入。

```js
// 导出
module.exports = { name: "小明", age: 18 };

// 引入（另一个文件）
const info = require("./info.js");
console.log(info.name); // "小明"
```

---

## 2. 这是一个对象吗

**是的**，这是一个**对象字面量**。

### 什么是对象字面量

用 `{}` 直接创建对象，键值对用 `:` 分隔：

```js
const person = {
    name: "小明",
    age: 18,
    sayHi() {
        console.log("你好");
    }
};
```

对应关系：

| 写出来的代码 | 等价的传统写法 |
|------------|--------------|
| `{ post(path, serviceFunc) {...} }` | `{ post: function(path, serviceFunc) {...} }` |
| `{ sayHi() {...} }` | `{ sayHi: function() {...} }` |

---

## 3. 语法演进：函数声明的进化

JavaScript 的函数声明语法经历了三次进化：

### 写法一：传统函数表达式（最老）

```js
module.exports = {
    post: function(path, serviceFunc) {
        // ...
    }
};
```

- 用 `function` 关键字
- 用 `:` 分隔键和值
- **完整但冗长**

### 写法二：方法简写语法（ES6+）

```js
module.exports = {
    post(path, serviceFunc) {
        // ...
    }
};
```

- 省略 `function` 关键字
- 省略 `:`
- 这是**方法简写**（Method Shorthand）
- 本质上仍然是函数
- **这是对象方法的正确写法！**

### 写法三：箭头函数（ES6+）

```js
module.exports = {
    post: (path, serviceFunc) => {
        // ...
    }
};
```

- 用 `=>` 箭头
- 仍然是传统格式 `key: value`

---

## 4. 你的问题：为什么没有 `:` 分隔

**因为这是方法简写语法（Method Shorthand）**，是 ES6 引入的语法糖。

### 完整的对象格式 vs 简写格式

```js
// 完整格式（有冒号）
const obj = {
    post: function(path, serviceFunc) {
        console.log("完整写法");
    }
};

// 简写格式（没有冒号）← 你代码里用的这个
const obj = {
    post(path, serviceFunc) {
        console.log("简写写法");
    }
};
```

**两种写法完全等价**，简写更简洁。

---

## 5. 箭头函数和普通函数的区别

虽然方法简写用的是 `post() {}` 而不是箭头函数，但箭头函数也是常见写法：

```js
// 普通函数
const add1 = function(a, b) {
    return a + b;
};

// 箭头函数（更简洁）
const add2 = (a, b) => {
    return a + b;
};

// 箭头函数（更更简洁，单行可以省略 return 和花括号）
const add3 = (a, b) => a + b;
```

### 箭头函数和普通函数的区别

| 特性 | 普通函数 | 箭头函数 |
|------|---------|---------|
| `this` | 有自己的 `this` | 继承外层的 `this` |
| `arguments` | 有 | 没有 |
| 能当构造函数 | 能（`new`） | 不能 |
| `yield` | 不能 | 不能 |

**在你的 Express 路由代码中，用普通函数是因为：**

- Express 需要访问 `req` 和 `res` 对象
- `this` 在路由处理中有特定用途

---

## 6. 完整代码逐行解析

```js
// 统一路由封装 - 统一响应格式
const express = require('express');
const router = express.Router();

module.exports = {                         // 导出这个对象
    post(path, serviceFunc) {              // post 是一个方法（函数）
        router.post(path, (req, res) => {  // 注册路由，req=请求，res=响应
            // 调用业务函数，传入参数和用户信息
            serviceFunc(req.body, req.user).then(result => {
                // 统一成功响应：code + data
                res.send({ code: 200, data: result });
            }).catch(error => {
                // 统一错误响应：code + msg
                res.send({ code: 500, msg: error.message });
            });
        });
        // 同时注册本地路由，支持内部调用
        localRouter.register(path, serviceFunc);
    }
};
```

**结构图解：**

```
module.exports = {
    ↑ 导出这个对象，让其他文件用 require() 引入

    post(path, serviceFunc) { ... }
    ↑ post 是对象的属性（方法）
    ↑ 简写语法，等价于 post: function(path, serviceFunc) { ... }
    ↑ path 和 serviceFunc 是函数的参数
};
```

---

## 7. 记忆口诀

> **对象字面量用 `{}`，方法简写不用冒号。**
>
> **箭头函数用 `=>`，普通函数用 `function`。**
>
> **导出用 `module.exports`，引入用 `require()`。**

---

## 8. 总结对比

| 写法 | 示例 | 特点 |
|------|------|------|
| 传统函数表达式 | `{ post: function() {} }` | 完整但冗长 |
| 方法简写 ← 你代码里用 | `{ post() {} }` | ES6+ 简洁写法 |
| 箭头函数 | `{ post: () => {} }` | 继承外层 this |

# JavaScript 相等比较：== 和 ===

## 1. 两种相等是什么

### 松散相等 `==`

使用两个等号 `==`，比较时**会先自动转换类型**，再比较。

```js
0 == false       // true
'' == false      // true
null == undefined // true
'5' == 5         // true
```

### 严格相等 `===`

使用三个等号 `===`，比较时**不转换类型**，类型和值都必须一样。

```js
0 === false      // false
'' === false     // false
null === undefined // false
'5' === 5        // false
```

---

## 2. 为什么松散相等会有奇怪结果

因为 JavaScript 在松散比较时，会按一套规则**自动把两边的值转成同一种类型**再比较。

常见转换规则：

```js
// 布尔值转成数字
true == 1        // true
false == 0       // true

// 字符串转成数字
'123' == 123     // true
'' == 0          // true

// null 和 undefined 互等
null == undefined // true
```

这导致松散相等有时候结果出乎意料。

---

## 3. 什么时候用谁

| 场景 | 推荐 |
|------|------|
| 一般值比较 | **严格相等 `===`** |
| 判断 x 是 null 或 undefined | `x == null`（惯例写法）|
| 判断 x 既不是 null 也不是 undefined | `x != null` |

```js
// 推荐：严格相等
if (userInput === '') { /* 处理空输入 */ }
if (status !== 200) { /* 处理错误 */ }

// 惯例：判断 null 或 undefined
if (value == null) { /* 相当于 value === null || value === undefined */ }
```

---

## 4. NaN 的特殊之处

无论哪种相等，NaN 都不等于任何值，包括它自己：

```js
NaN == NaN        // false
NaN === NaN       // false
```

判断一个值是否是 NaN，应该用 `Number.isNaN()`：

```js
Number.isNaN(NaN) // true
```

---

## 5. 对象比较

对象比较时，`==` 和 `===` 行为一样——都是比较引用，而不是内容：

```js
{} == {}          // false
{} === {}         // false
[] == []          // false
[] === []         // false

const obj1 = { a: 1 };
const obj2 = obj1;
obj1 === obj2     // true，同一个引用
```

---

## 6. 记忆口诀

> **松散相等爱翻译（自动类型转换），严格相等讲原则（类型值都必须对）。**
>
> **平时比较用 `===`，判断 null/undefined 用 `==`。**

---

## 7. 总结表格

| 比较 | 0 | '' | null | undefined | '5' |
|------|---|----|------|-----------|-----|
| `== false` | true | true | false | false | false |
| `=== false` | false | false | false | false | false |
| `== 0` | - | true | false | false | true |
| `=== 0` | - | false | false | false | false |

（`-` 表示与自身比较，恒为 true）

# JavaScript 异常处理：throw vs return

## 1. 两者的本质区别

| 特性 | `throw` | `return` |
|------|---------|----------|
| **性质** | 抛出异常，中断执行 | 返回值，正常退出 |
| **后续代码** | 跳到最近的 `catch` 块 | 函数直接结束 |
| **用途** | 表示出了问题 | 表示正常完成 |

---

## 2. 对比示例

### throw：出了问题，跳到 catch

```js
async function example1() {
    if (userNotFound) {
        throw new Error("用户不存在"); // 中断，跳到 catch
    }
    console.log("这里不会执行"); // 不会执行
    return "成功";
}
```

### return：正常返回空值

```js
async function example2() {
    if (userNotFound) {
        return null; // 正常返回空值
    }
    console.log("这里会执行"); // 会执行
    return "成功";
}
```

---

## 3. 在实际代码中的用法

### catch 块中用 throw 重新抛出

```js
catch (e) {
    await projectRepository.stopCalculate(projectId);
    throw e; // 重新抛出错误，让调用者知道出错了
}
```

### try 块中用 return 返回结果

```js
try {
    return await calculateFn(); // 正常返回结果
}
```

---

## 4. 什么时候用谁

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 函数正常结束，返回结果 | `return value` | 告诉调用者：搞定了，这是结果 |
| 函数正常结束，没有结果 | `return` 或 `return null` | 告诉调用者：完成了，但没结果 |
| 遇到错误，无法继续 | `throw new Error()` | 告诉调用者：出问题了，你来处理 |
| 捕获错误后想上报 | `throw e` | 重新抛出，让上层也知道出错了 |

---

## 5. 记忆口诀

> **throw 是"出错了，告诉调用者"，return 是"搞定了，把结果给你"。**

---

## 6. 一句话总结

```js
// throw：中断执行，跳到 catch
throw new Error("出问题了");

// return：正常结束，返回结果
return result;
```

# 数据库事务：保证数据一致性的机制

## 1. 什么是事务

**事务（Transaction）** 是数据库中一组操作的集合，它保证这些操作要么全部成功，要么全部失败。

简单理解：

- 不用事务：点外卖 → 商家做了 → 骑手没了 → 商家亏了
- 用事务：点外卖 → 商家开始做 → 骑手拿到 → 商家收钱（中途任何一步失败 → 全部撤销）

---

## 2. 为什么需要事务

场景举例：用户创建项目，需要做两件事：

1. 在 `projects` 表插入一条数据
2. 在 `user_projects` 表插入一条关联数据

| 不用事务（危险） | 用事务（安全） |
|----------------|---------------|
| 第1步成功 | 第1步成功 |
| 第2步失败（服务器崩了） | 第2步失败 |
| ❌ 项目创建了但没有关联用户 | ✅ 回滚第1步，数据不受影响 |

---

## 3. 代码示例（Sequelize）

```js
// 事务处理 - 保证数据一致性
async function createProjectWithUser(projectData, userId) {
    // 1. 开启事务
    const t = await sequelize.transaction();
    
    try {
        // 2. 在事务中执行多个操作
        // 创建项目
        const project = await Project.create(projectData, { transaction: t });
        
        // 创建项目与用户的关联关系
        await UserProject.create({
            userId: userId,
            projectId: project.id
        }, { transaction: t });
        
        // 3. 提交事务（两个操作都成功，才真正写入数据库）
        await t.commit();
        
        return project;
    } catch (error) {
        // 4. 出错时回滚（撤销所有操作，数据恢复原样）
        await t.rollback();
        throw error;
    }
}
```

逐行解释：

- `sequelize.transaction()`：开启一个事务，得到事务对象 `t`
- `{ transaction: t }`：把事务对象传给数据库操作，让它们在同一个事务中执行
- `t.commit()`：提交事务，所有操作真正写入数据库
- `t.rollback()`：回滚事务，撤销所有未提交的操作

---

## 4. ACID 四个特性

| 特性 | 解释 |
|------|------|
| **Atomicity（原子性）** | 多个操作打包成一个整体，要成功全成功，要失败全失败 |
| **Consistency（一致性）** | 事务前后数据都处于正确状态，不会出现"半成品" |
| **Isolation（隔离性）** | 并发执行时，互不干扰 |
| **Durability（持久性）** | 事务提交后，数据永久保存 |

---

## 5. 什么时候用事务

当一个业务操作涉及**多个数据库写操作**，且这些操作必须同时成功或同时失败时，就需要用事务。

常见场景：

- 创建订单 + 扣减库存 + 创建支付记录
- 转账：减钱 + 加钱
- 创建文章 + 创建标签关联

---

## 6. 记忆口诀

> **事务就是"打包执行"，要么全成功，要么全撤销。**

---

## 7. 一句话总结

```js
// 开启事务
const t = await sequelize.transaction();

try {
    // 多个操作放在事务中
    await op1({ transaction: t });
    await op2({ transaction: t });
    
    // 成功：提交
    await t.commit();
} catch {
    // 失败：回滚
    await t.rollback();
    throw error;
}
```
