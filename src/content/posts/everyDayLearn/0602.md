# EventBus 事件总线：组件间的对讲机

## 1. 是什么

EventBus 是一个**发布-订阅模式**的实现，用于解决**组件之间的通信问题**。

简单理解：

- **不用 EventBus**：组件 A 告诉组件 B "我被点了"，需要组件 A 知道组件 B 是谁
- **用 EventBus**：组件 A 只管广播"有人点了闸门"，组件 B 自己去听这个消息

在这个项目里，它解决的是：**地图点击了某个实体，怎么通知到对应的业务页面**。

---

## 2. 三核心方法：on → emit → off

```typescript
class EventBus {
  private events: Map<EventName, Set<EventCallback>>;

  // 订阅：告诉 EventBus "我想监听某个事件"
  on(event, callback) {
    if (!this.events.has(event)) {
      this.events.set(event, new Set());
    }
    this.events.get(event).add(callback);
  }

  // 发布：告诉 EventBus "发生了一件事"
  emit(event, data) {
    const callbacks = this.events.get(event);
    if (callbacks) {
      callbacks.forEach(cb => cb(data));
    }
  }

  // 取消订阅：不再监听某个事件
  off(event, callback) {
    const callbacks = this.events.get(event);
    if (callbacks) {
      callbacks.delete(callback);
    }
  }
}
```

逐行解释：

- `Map<EventName, Set<EventCallback>>`：用 Map 存储事件名到回调函数集合的映射
- `on`：把回调函数加入某个事件的监听队列
- `emit`：遍历并执行某个事件的所有回调函数
- `off`：从某个事件的监听队列中移除指定的回调函数

## 3. 事件名命名规范

事件名要清晰表达"什么东西发生了什么动作"：

```typescript
// 命名规范：entity:类型:动作
export const EntityEventType = {
  REACH_CLICK: 'entity:reach:click',           // 河段点击
  GATE_CLICK: 'entity:gate:click',             // 闸门点击
  DAM_CLICK: 'entity:dam:click',              // 坝点击
  PUMP_CLICK: 'entity:pump:click',            // 泵站点击
  WEIR_CLICK: 'entity:weir:click',            // 堰点击
  SUCK_CLICK: 'entity:suck:click',            // 泵吸点击
  ONE_DIMENSIONAL_STATION_CLICK: 'entity:oneDimensionalStation:click',
  BRANCH_CLICK: 'entity:branch:click',        // 分叉点点击
  CALCULATED_POINT_CLICK: 'entity:calculatedPoint:click',
};
```

格式说明：`entity:类型:动作`

- `entity`：表示这是实体相关的事件
- `类型`：具体是什么实体（gate、dam、pump 等）
- `动作`：发生了什么动作（click、hover 等）

## 4. 事件数据结构

```typescript
export type EntityEventData = {
  type: string;           // 实体类型，如 'gate'
  data: any;              // 业务数据，如 { id, name }
  entity: any;            // Cesium.Entity 实体对象
  position: { x, y };    // 屏幕点击坐标
};
```

emit 的时候把事件数据传出去，订阅者就能拿到这些信息。

## 5. 工作流程图

```
┌──────────────┐                    ┌──────────────┐
│   地图点击   │                    │  业务页面A   │
│  点击了闸门   │                    │  监听 GATE   │
└──────┬───────┘                    └──────▲───────┘
       │                                  │
       │ emit('entity:gate:click', data)  │
       └──────────────────────────────────┘
```

1. 地图检测到闸门被点击
2. 地图调用 `eventBus.emit('entity:gate:click', eventData)`
3. EventBus 找到所有监听 `GATE_CLICK` 的回调函数
4. 业务页面 A 的回调函数被触发

## 6. 具体例子：Gates.vue

### 第一步：业务页面订阅事件

```typescript
onMounted(() => {
  // onMounted：组件挂载时开始监听
  eventBus.on(EntityEventType.GATE_CLICK, handleGateClick);
});
```

### 第二步：页面卸载时取消订阅

```typescript
onUnmounted(() => {
  // onUnmounted：组件销毁时停止监听
  eventBus.off(EntityEventType.GATE_CLICK, handleGateClick);
});
```

**为什么要取消订阅？**

- 组件销毁后，如果回调函数还被 EventBus 持有，会导致内存泄漏
- 下次触发事件时，已经不存在的组件的回调函数也会被执行

### 第三步：处理点击事件

```typescript
const handleGateClick = (eventData) => {
  // eventData.data 就是 { id, name }
  // 找到表格中对应的行，选中它
  highlightTableRow(eventData.data.id);
};
```

## 7. 事件从哪里 emit

在 `LayerCommon.dispatchEventOnClick()` 中：

```typescript
// 点击地图 → 识别实体类型 → emit 对应事件
switch (entityType) {
  case 'gate':
    eventBus.emit(EntityEventType.GATE_CLICK, eventData);
    break;
  case 'weir':
    eventBus.emit(EntityEventType.WEIR_CLICK, eventData);
    break;
  // ... 其他实体类型
}
```

**总结**：EventBus 就是"地图和页面之间的对讲机"——地图按类型广播，页面按需收听。

## 8. 为什么用 EventBus 而不是直接回调

| 方式 | 缺点 |
|------|------|
| 直接回调 | 地图和页面强耦合，不好扩展 |
| **EventBus** | 解耦，地图只管 emit，业务页面自己订阅 |

直接回调的问题：

- 地图需要知道有多少个页面要监听
- 每新增一个监听者，地图代码就要改
- 难以管理多个监听者

EventBus 的优势：

- 地图只管 emit，不关心谁在听
- 业务页面自己决定要不要听
- 新增/移除监听者不需要改地图代码

## 10. 一句话总结

EventBus = 发布-订阅模式 + 事件命名规范 + 生命周期管理（on/off 配对）。

```typescript
// 订阅
eventBus.on('event:name', handler);

// 发布
eventBus.emit('event:name', data);

// 取消订阅（组件销毁时）
eventBus.off('event:name', handler);
```
