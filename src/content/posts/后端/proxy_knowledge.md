## 什么是代理

代理本质上是**中间层**，在客户端和服务器之间转发请求。

```
客户端 ──→ 代理服务器 ──→ 目标服务器
  ↑                      ↓
  └─────── 响应返回 ──────┘
```

## 正向代理 vs 反向代理

### 正向代理（Forward Proxy）

**代替客户端发送请求**

```
┌─────────────────────────────────────────────────────────────────┐
│                      正向代理                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   客户端 A ──┐                                                 │
│   客户端 B ──┼──→ 代理服务器 ──→ 目标网站 Google                │
│   客户端 C ──┘     (VPN/梯子)                                    │
│                                                                 │
│   特点：客户端知道目标是谁，但目标不知道真正是谁在访问            │
│   用途：翻墙、缓存、隐藏客户端 IP                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 反向代理（Reverse Proxy）

**代替服务器接收请求**

```
┌─────────────────────────────────────────────────────────────────┐
│                      反向代理                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                      用户浏览器                                  │
│                          │                                      │
│                          ▼                                      │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  反向代理服务器 (Caddy/Nginx)                           │   │
│   │                                                         │   │
│   │    ├── /api  ──────────→ 应用服务器 A (端口 3000)       │   │
│   │    ├── /admin ─────────→ 管理后台 B (端口 4000)         │   │
│   │    └── /static ────────→ 静态文件 C (端口 5000)         │   │
│   │                                                         │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│   特点：客户端不知道真正是哪台服务器在处理                        │
│   用途：负载均衡、安全防护、统一入口、SSL 终止                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 正向代理 vs 反向代理 对比

| 对比项 | 正向代理 | 反向代理 |
|--------|---------|---------|
| **位置** | 靠近客户端 | 靠近服务器 |
| **代理谁** | 代替客户端 | 代替服务器 |
| **客户端知道吗** | 知道（配置代理） | 不知道 |
| **服务器知道吗** | 不知道 | 知道 |
| **典型用途** | 翻墙、翻公司墙 | 负载均衡、安全防护 |
| **代表软件** | VPN、Shadowsocks | Caddy、Nginx |

### 一句话总结

```
正向代理：代替客户端"出去"
反向代理：代替服务器"进来"
```

## Vite Proxy（开发代理）

### 什么是 Vite Proxy

Vite Proxy 是**开发服务器内置的请求转发功能**，只在 `npm run dev` 时生效，用于解决开发时的**跨域问题**。

### 为什么需要 Vite Proxy

```
┌─────────────────────────────────────────────────────────────────┐
│                      跨域问题                                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   浏览器访问: localhost:5173 (前端页面)                          │
│                                                                 │
│   前端页面需要数据: http://10.53.10.96:8088/api/users            │
│                                                                 │
│   浏览器：协议不同 + 域名不同 + 端口不同 = 跨域！拒绝访问！        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Vite Proxy 如何解决

```
┌─────────────────────────────────────────────────────────────────┐
│                      Vite 代理解决跨域                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   浏览器请求: localhost:5173/dev-api/users                       │
│       │                                                        │
│       │ 浏览器以为是同源请求                                     │
│       ▼                                                        │
│   Vite 开发服务器（代理）                                        │
│       │                                                        │
│       │ 拦截请求，发送到目标服务器                               │
│       │ 去掉 /dev-api 前缀                                      │
│       ▼                                                        │
│   目标服务器: http://10.53.10.96:8088/users                     │
│                                                                 │
│   目标服务器返回数据 ──→ Vite ──→ 浏览器                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 配置示例

```typescript
// electron.vite.config.ts
server: {
  proxy: {
    // 匹配 /dev-api/wchapi 开头的请求
    '/dev-api/wchapi': {
      target: 'http://server-vm-dev-fedora.private.sh-office.wuchuanwater.com:7122',
      changeOrigin: true,
      rewrite: (p) => p.replace(/^\/dev-api/, '')
    },
    // 匹配 /dev-api 开头的请求
    '/dev-api': {
      target: 'http://10.53.10.96:8088',
      changeOrigin: true,
      rewrite: (p) => p.replace(/^\/dev-api/, '')
    },
  }
}
```

### 配置参数说明

| 参数 | 作用 |
|------|------|
| `/dev-api` | 匹配路径前缀，符合条件的请求会被代理 |
| `target` | 目标服务器地址 |
| `changeOrigin: true` | 修改请求头中的 Origin，避免目标服务器拒绝 |
| `rewrite` | 路径重写，去掉前缀 |

### 请求转换示例

```
原始请求: http://localhost:5173/dev-api/wchapi/projects
    ↓ Vite 代理
重写后:   http://10.53.10.96:8088/wchapi/projects

原始请求: http://localhost:5173/dev-api/users
    ↓ Vite 代理
重写后:   http://10.53.10.96:8088/users
```

### Vite Proxy 的特点

| 特性 | 说明 |
|------|------|
| **只在开发时生效** | `npm run build` 打包后消失 |
| **开发服务器内置** | Vite 自动启动，无需单独配置 |
| **自动拦截** | 浏览器无需配置，Vite 自动处理 |
| **解决跨域** | 让浏览器"以为"是同源请求 |

## Caddy/Nginx（生产反向代理）

### 什么是生产反向代理

生产环境中使用独立的代理服务器（如 Caddy、Nginx）作为**真正的反向代理**，对外提供统一服务入口。

### 为什么需要反向代理

```
┌─────────────────────────────────────────────────────────────────┐
│                      生产反向代理的必要性                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题1: 端口暴露                                                 │
│  ─────────                                                       │
│  直接暴露 Node.js 端口 8088？不安全！                            │
│  用户直接访问 http://服务器IP:8088                              │
│                                                                 │
│  问题2: HTTPS                                                   │
│  ─────────                                                       │
│  Node.js 不方便配置 HTTPS                                       │
│                                                                 │
│  问题3: 统一入口                                                │
│  ─────────                                                       │
│  多个服务需要统一入口                                            │
│                                                                 │
│  解决方案: 反向代理                                              │
│  ─────────                                                      │
│  Caddy/Nginx 监听 80/443 端口，对外统一服务                       │
│  内部转发到各个应用服务器                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 项目中的容器端口映射

```
┌─────────────────────────────────────────────────────────────────┐
│                      容器端口映射                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   docker-compose.yml                                            │
│   ──────────────────                                            │
│   ports:                                                        │
│     - "8000:80/tcp"         # 主机 8000 → 容器 80               │
│                                                                 │
│   容器内部                                                       │
│   ─────────                                                     │
│   Caddy 监听: 端口 80                                           │
│   Node.js 监听: 端口 81                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 容器内部通信

```
┌─────────────────────────────────────────────────────────────────┐
│                      容器内部架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户浏览器                                                     │
│       │                                                         │
│       │ http://服务器IP:8000/wc/api/projects                  │
│       ▼                                                         │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  Caddy (容器内，监听端口 80)                            │   │
│   │                                                         │   │
│   │  • 对外统一入口                                          │   │
│   │  • 自动 HTTPS                                           │   │
│   │  • 路径转发: /wc/* → localhost:81                     │   │
│   └─────────────────────────────────────────────────────────┘   │
│       │                                                         │
│       │ localhost:81                                           │
│       │ (内部通信，外部无法直接访问)                            │
│       ▼                                                         │
│   ┌─────────────────────────────────────────────────────────┐   │
│   │  Node.js 应用 (容器内，监听端口 81)                      │   │
│   │                                                         │   │
│   │  • 真正的业务逻辑                                        │   │
│   │  • 不对外暴露                                            │   │
│   │  • 只接受来自 Caddy 的请求                                │   │
│   └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 请求流程

```
用户浏览器
    │
    │ http://服务器IP:8000/wch/api/projects
    │
    ▼
主机防火墙/路由器
    │
    │ 映射到容器端口 8000 → 80
    │
    ▼
Caddy (容器端口 80)
    │
    │ 匹配 /wch 前缀
    │ 转发到 localhost:81
    │
    ▼
Node.js (容器端口 81)
    │
    │ 处理业务逻辑
    │
    ▼
返回响应给 Caddy → 用户
```

### Caddy vs Nginx 配置对比

| 对比项 | Caddy | Nginx |
|--------|-------|-------|
| **配置文件** | Caddyfile（简单） | nginx.conf（复杂） |
| **HTTPS** | 自动申请证书 | 手动配置 |
| **语法** | 人类友好 | 像编程语言 |
| **性能** | 稍低 | 更高 |
| **适用场景** | 简单部署 | 高性能生产环境 |

### Caddy 配置示例

```nginx
# Caddyfile
:80 {
    reverse_proxy /wch/* localhost:81
}
```

### Nginx 配置示例

```nginx
# nginx.conf
server {
    listen 80;
    server_name _;

    location /wch {
        proxy_pass http://localhost:81;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### 反向代理的优势

| 优势 | 说明 |
|------|------|
| **安全隐藏** | 真实服务器端口不暴露 |
| **统一入口** | 80/443 是标准端口，用户不用记复杂端口 |
| **负载均衡** | 一台代理 → 多台后端服务器 |
| **HTTPS 自动配置** | Caddy 自动申请 Let's Encrypt 证书 |
| **静态文件缓存** | 代理直接返回静态资源，减轻后端压力 |
| **请求过滤** | 代理层可过滤恶意请求 |

## 三种代理对比总结

```
┌─────────────────────────────────────────────────────────────────┐
│                      三种代理对比                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐              │
│  │  正向代理    │  │  Vite Proxy  │  │  反向代理    │              │
│  │             │  │  (开发代理)   │  │  (Caddy)    │              │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤              │
│  │ 代替客户端   │  │ 代替浏览器   │  │ 代替服务器   │              │
│  │ 出去访问     │  │ 发请求       │  │ 接收请求     │              │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤              │
│  │ 翻墙        │  │ 解决跨域     │  │ 负载均衡    │              │
│  │ 缓存        │  │ 开发调试     │  │ 安全防护    │              │
│  │ 隐藏IP      │  │             │  │ 统一入口    │              │
│  ├─────────────┤  ├─────────────┤  ├─────────────┤              │
│  │ VPN        │  │ Vite        │  │ Caddy      │              │
│  │ Shadowsocks│  │ Webpack Dev │  │ Nginx      │              │
│  └─────────────┘  └─────────────┘  └─────────────┘              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 一表总结

| 类型 | 代理谁 | 何时生效 | 解决什么问题 | 代表软件 |
|------|--------|---------|-------------|---------|
| **正向代理** | 客户端 | 始终 | 翻墙、缓存、隐藏IP | VPN、Shadowsocks |
| **Vite Proxy** | 浏览器 | 仅开发时 | 跨域问题 | Vite、Webpack Dev |
| **反向代理** | 服务器 | 生产始终 | 负载均衡、安全、统一入口 | Caddy、Nginx |

### 记忆口诀

```
正向代理：代替客户端"出去"（翻墙）
Vite 代理：代替浏览器"去拿"（解决跨域）
反向代理：代替服务器"进来"（统一入口）
```

## 项目中的代理使用

### 开发时（前端独立开发）

```
前端 npm run dev (Vite)
    │
    │ localhost:5173/dev-api/*
    │
    ▼ Vite Proxy
后端服务器 10.53.10.96:8088
```

### 生产时（Docker 容器部署）

```
用户浏览器
    │
    │ http://服务器IP:8000/wch/*
    │
    ▼
Caddy (容器端口 80)
    │
    │ /wch/* → localhost:81
    │
    ▼
Node.js (容器端口 81)
```
